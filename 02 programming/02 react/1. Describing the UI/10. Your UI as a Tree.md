Ваше приложение React обретает форму, и многие компоненты вложены друг в друга. Как React отслеживает структуру компонентов вашего приложения?

React и многие другие библиотеки пользовательского интерфейса моделируют пользовательский интерфейс как дерево. Думать о вашем приложении как о дереве полезно для понимания взаимосвязи между компонентами. Это понимание поможет вам отлаживать будущие концепции, такие как управление производительностью и состоянием.

Деревья - это модель взаимосвязи между элементами, а пользовательский интерфейс часто представлен с помощью древесных структур. Например, браузеры используют древовые структуры для моделирования HTML ([DOM](https://developer.mozilla.org/docs/Web/API/Document_Object_Model/Introduction)) и CSS ([CSSOM](https://developer.mozilla.org/docs/Web/API/CSS_Object_Model)). Мобильные платформы также используют деревья для представления своей иерархии представлений.

![[Pasted image 20240526124316.png]]
Как и браузеры и мобильные платформы, React также использует деревовые структуры для управления и моделирования взаимосвязи между компонентами в приложении React. Эти деревья являются полезными инструментами для понимания того, как данные проходят через приложение React и как оптимизировать рендеринг и размер приложения.

## Дерево Рендеринга[](https://react.dev/learn/understanding-your-ui-as-a-tree#the-render-tree "Ссылка на дерево рендеринга")
сновной особенностью компонентов является способность сочинять компоненты других компонентов. Когда мы [ввязываем компоненты](https://react.dev/learn/your-first-component#nesting-and-organizing-components), у нас есть концепция родительских и дочерних компонентов, где каждый родительский компонент сам может быть дочерним компонентом другого компонента.

Когда мы рендерим приложение React, мы можем смоделировать это отношение в дереве, известном как дерево рендеринга.
![[Pasted image 20240526124424.png]]
Корневой узел в дереве рендеринга React является [корневым](https://react.dev/learn/importing-and-exporting-components#the-root-component-file) [компонентом](https://react.dev/learn/importing-and-exporting-components#the-root-component-file) приложения. В этом случае корневым компонентом является `App`, и это первый компонент, который React отображает. Каждая стрелка в дереве указывает от родительского компонента к дочернему компоненту.
Вы заметите, что в приведенном выше дереве рендеринга нет упоминания о тегах HTML, которые отображает каждый компонент. Это связано с тем, что дерево рендеринга состоит только из [компонентов](https://react.dev/learn/your-first-component#components-ui-building-blocks) React.

React, как фреймворк пользовательского интерфейса, не зависит от платформы. На react.dev мы демонстрируем примеры, которые отображаются в Интернете, который использует HTML-разметку в качестве примитивов пользовательского интерфейса. Но приложение React может с такой же вероятностью отображаться на мобильной или настольной платформе, которая может использовать различные примитивы пользовательского интерфейса, такие как [UIView](https://developer.apple.com/documentation/uikit/uiview) или [FrameworkElement](https://learn.microsoft.com/en-us/dotnet/api/system.windows.frameworkelement?view=windowsdesktop-7.0).

Эти примитивы пользовательского интерфейса платформы не являются частью React. Деревья рендеринга React могут предоставлять информацию о нашем приложении React независимо от того, на какой платформе отображается ваше приложение.

Дерево рендеринга представляет собой один проход рендеринга React-приложения. При [условном](https://react.dev/learn/conditional-rendering) [рендеринге](https://react.dev/learn/conditional-rendering) родительский компонент может отображать различные дочерние компоненты в зависимости от переданных данных.

![[Pasted image 20240526124656.png]]
Хотя деревья рендеринга могут отличаться в зависимости от проходов рендеринга, эти деревья, как правило, полезны для определения _компонентов_ _верхнего уровня_ и _листьев_ в приложении React. Компоненты верхнего уровня являются компонентами, ближайшими к корневому компоненту, и влияют на производительность рендеринга всех компонентов под ними и часто содержат наибольшую сложность. Компоненты листьев находятся рядом с нижней частью дерева и не имеют дочерних компонентов и часто повторно рендерируются

## Дерево зависимостей модуля[](https://react.dev/learn/understanding-your-ui-as-a-tree#the-module-dependency-tree "Ссылка на дерево зависимостей модуля")
Еще одна связь в приложении React, которая может быть смоделирована с помощью дерева, - это зависимости модуля приложения. Когда мы [разбьем наши компоненты](https://react.dev/learn/importing-and-exporting-components#exporting-and-importing-a-component) и логику на отдельные файлы, мы создаем [модули JS](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Guide/Modules), где мы можем экспортировать компоненты, функции или константы.

Каждый узел в дереве зависимостей модуля является модулем, и каждая ветвь представляет собой оператор `import` в этом модуле.

Если мы возьмем предыдущее приложение Inspirations, мы сможем построить дерево зависимостей модуля, или дерево зависимостей.
![[Pasted image 20240526124800.png]]
Корневой узел дерева - это корневой модуль, также известный как файл точки входа. Часто именно модуль содержит корневой компонент.

По сравнению с деревом рендеринга одного и того же приложения, существуют похожие структуры, но некоторые заметные различия:

- Узлы, составляющие дерево, представляют собой модули, а не компоненты.
- Некомпонентные модули, такие как `inspirations.js`, также представлены в этом дереве. Дерево рендеринга инкапсулирует только компоненты.
- Copyright.js отображается в App.js, но в дереве рендеринга компонент Copyright отображается как дочерний элемент InspirationGenerator. Это связано с тем, что InspirationGenerator принимает JSX в качестве дочерних свойств, поэтому он отображает авторские права как дочерний компонент, но не импортирует модуль.
Деревья зависимостей полезны для определения того, какие модули необходимы для запуска вашего приложения React. При создании приложения React для производства обычно существует шаг сборки, который объединит все необходимые JavaScript для отправки клиенту. Инструмент, ответственный за это, называется [bundler](https://developer.mozilla.org/en-US/docs/Learn/Tools_and_testing/Understanding_client-side_tools/Overview#the_modern_tooling_ecosystem), и bundlers будут использовать дерево зависимостей, чтобы определить, какие модули должны быть включены.
По мере роста вашего приложения часто появляется и размер пакета. Большие размеры пакетов стоят дорого для клиента для загрузки и запуска. Большие размеры пакетов могут задержать время для нарисования вашего пользовательского интерфейса. Получение ощущения дерева зависимостей вашего приложения может помочь в отладке этих проблем.

### Итог
- Деревья - это распространенный способ представления отношений между сущностями. Они часто используются для моделирования пользовательского интерфейса.
- Деревья рендеринга представляют собой вложенные отношения между компонентами React в одном рендере.
- При условном рендеринге дерево рендеринга может меняться в разных рендерах. При различных значениях реквизита компоненты могут отображать различные дочерние компоненты.
- Деревья рендеринга помогают определить, что такое компоненты верхнего уровня и листьев. Компоненты верхнего уровня влияют на производительность рендеринга всех компонентов под ними, и компоненты листа часто повторно рендерируются. Их выявление полезно для понимания и отладки производительности рендеринга.
- Деревья зависимостей представляют зависимости модуля в приложении React.
- Деревья зависимостей используются инструментами сборки для объединения необходимого кода для отправки приложения.
- Деревья зависимостей полезны для отладки больших размеров пакетов, которые замедляют время рисования и предоставляют возможности для оптимизации того, какой код является пакетом.
