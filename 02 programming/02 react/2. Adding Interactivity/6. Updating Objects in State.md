Состояние может хранить любое значение JavaScript, включая объекты. Но вы не должны изменять объекты, которые вы держите в состоянии React напрямую. Вместо этого, когда вы хотите обновить объект, вам нужно создать новый (или сделать копию существующего), а затем установить состояние для использования этой копии.
## What’s a mutation? [](https://react.dev/learn/updating-objects-in-state#whats-a-mutation "Link for What’s a mutation?")
Вы можете сохранить любое значение JavaScript в состоянии.
![[Pasted image 20240526190519.png]]
До сих пор вы работали с числами, струнами и буленами. Такие значения JavaScript являются "неизменными", что означает неизменяемыми или "только для чтения". Вы можете запустить повторную рендеринг, чтобы _заменить_ значение:
![[Pasted image 20240526190632.png]]
Состояние x изменилось с 0 на 5, но само число 0 не изменилось. В JavaScript невозможно внести какие-либо изменения во встроенные примитивные значения, такие как числа, строки и логические значения.
Теперь рассмотрим объект в состоянии:
![[Pasted image 20240526190836.png]]
Технически, можно изменить содержимое _самого объекта_. **Это называется мутацией:**
![[Pasted image 20240526190903.png]]
Однако, хотя объекты в состоянии React технически изменяемы, вы должны относиться к ним **так, как если бы** они были неизменными, например, числа, булены и строки. Вместо того, чтобы мутировать их, вы всегда должны их заменять.
## Treat state as read-only [](https://react.dev/learn/updating-objects-in-state#treat-state-as-read-only "Link for Treat state as read-only")
Другими словами, вы должны **рассматривать любой объект JavaScript, который вы вводили в состояние, как только для чтения.**

Этот пример содержит объект в состоянии, представляющий текущую позицию указателя. Красная точка должна перемещаться при касании или перемещении курсора по области предварительного просмотра. Но точка остается в исходном положении:
```js
import { useState } from 'react';
export default function MovingDot() {
  const [position, setPosition] = useState({
    x: 0,
    y: 0
  });
  return (
    <div
      onPointerMove={e => {
        position.x = e.clientX;
        position.y = e.clientY;
      }}
      style={{
        position: 'relative',
        width: '100vw',
        height: '100vh',
      }}>
      <div style={{
        position: 'absolute',
        backgroundColor: 'red',
        borderRadius: '50%',
        transform: `translate(${position.x}px, ${position.y}px)`,
        left: -10,
        top: -10,
        width: 20,
        height: 20,
      }} />
    </div>
  );
}

```
Проблема в этом кусочке кода.
![[Pasted image 20240526191133.png]]
Этот код изменяет объект, присвоенный позиции из предыдущего рендеринга. Но без использования функции установки состояния React понятия не имеет, что объект изменился. Таким образом, React ничего не делает в ответ. Это все равно, что пытаться изменить заказ после того, как вы уже съели еду. Хотя изменение состояния может сработать в некоторых случаях, мы не рекомендуем его. Вы должны рассматривать значение состояния, к которому у вас есть доступ при рендеринге, как доступное только для чтения.
Чтобы фактически [запустить повторный рендер](https://react.dev/learn/state-as-a-snapshot#setting-state-triggers-renders) в этом случае, **создайте _новый_ объект и передайте его функции настройки состояния:**
![[Pasted image 20240526191234.png]]
With `setPosition`, you’re telling React:
- Replace `position` with this new object
- - И снова отобразите этот компонент
Обратите внимание, что красная точка теперь следует за вашим указателем, когда вы касаетесь или наводите на область предварительного просмотра:
#### Local mutation is fine
Такой код является проблемой, потому что он изменяет существующий объект в состоянии:
![[Pasted image 20240526191345.png]]
Но такой код **абсолютно хорош**, потому что вы мутируете новый объект, который вы _только что создали_:
![[Pasted image 20240526191407.png]]
На самом деле, это полностью эквивалентно написанию этого:
![[Pasted image 20240526191419.png]]
Мутация является проблемой только при изменении _существующих_ объектов, которые уже находятся в состоянии. Мутация объекта, который вы только что создали, в порядке, потому что _никакой другой код еще не ссылается на него._ Изменение этого не повлияет на что-то, что зависит от этого. Это называется "локальной мутацией". Вы даже можете сделать локальную мутацию [во время рендеринга.](https://react.dev/learn/keeping-components-pure#local-mutation-your-components-little-secret) Очень удобно и совершенно нормально!
## Copying objects with the spread syntax [](https://react.dev/learn/updating-objects-in-state#copying-objects-with-the-spread-syntax "Link for Copying objects with the spread syntax")
В предыдущем примере объект позиции всегда создается заново из текущей позиции курсора. Но часто вам захочется включить существующие данные в состав нового объекта, который вы создаете. Например, вы можете обновить только одно поле в форме, но сохранить предыдущие значения для всех остальных полей.

Эти поля ввода не работают, поскольку обработчики onChange изменяют состояние:
```js
import { useState } from 'react';

export default function Form() {
  const [person, setPerson] = useState({
    firstName: 'Barbara',
    lastName: 'Hepworth',
    email: 'bhepworth@sculpture.com'
  });

  function handleFirstNameChange(e) {
    person.firstName = e.target.value;
  }

  function handleLastNameChange(e) {
    person.lastName = e.target.value;
  }

  function handleEmailChange(e) {
    person.email = e.target.value;
  }

  return (
    <>
      <label>
        First name:
        <input
          value={person.firstName}
          onChange={handleFirstNameChange}
        />
      </label>
      <label>
        Last name:
        <input
          value={person.lastName}
          onChange={handleLastNameChange}
        />
      </label>
      <label>
        Email:
        <input
          value={person.email}
          onChange={handleEmailChange}
        />
      </label>
      <p>
        {person.firstName}{' '}
        {person.lastName}{' '}
        ({person.email})
      </p>
    </>
  );
}

```
Например, эта строка мутирует состояние из прошлого рендеринга:
![[Pasted image 20240526191846.png]]
Надежный способ получить поведение, которое вы ищете, - это создать новый объект и передать его в `setPerson`. Но здесь вы также хотите **скопировать в него существующие данные,** потому что изменилось только одно из полей:
![[Pasted image 20240526191908.png]]
You can use the `...` [object spread](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Operators/Spread_syntax#spread_in_object_literals) syntax so that you don’t need to copy every property separately.
![[Pasted image 20240526191921.png]]
Теперь форма работает!

Обратите внимание, что вы не объявили отдельную переменную состояния для каждого поля ввода. Для больших форм хранить все данные, сгруппированные в объекте, очень удобно, если вы правильно их обновляете!
Обратите внимание, что синтаксис ...spread является «поверхностным» — он копирует только один уровень глубины. Это делает его быстрым, но также означает, что если вы хотите обновить вложенное свойство, вам придется использовать его несколько раз.
#### Using a single event handler for multiple fields
Вы также можете использовать фигурные скобки [ и ] внутри определения объекта, чтобы указать свойство с динамическим именем. Вот тот же пример, но с одним обработчиком событий вместо трёх разных:
```js
import { useState } from 'react';

export default function Form() {
  const [person, setPerson] = useState({
    firstName: 'Barbara',
    lastName: 'Hepworth',
    email: 'bhepworth@sculpture.com'
  });

  function handleChange(e) {
    setPerson({
      ...person,
      [e.target.name]: e.target.value
    });
  }

  return (
    <>
      <label>
        First name:
        <input
          name="firstName"
          value={person.firstName}
          onChange={handleChange}
        />
      </label>
      <label>
        Last name:
        <input
          name="lastName"
          value={person.lastName}
          onChange={handleChange}
        />
      </label>
      <label>
        Email:
        <input
          name="email"
          value={person.email}
          onChange={handleChange}
        />
      </label>
      <p>
        {person.firstName}{' '}
        {person.lastName}{' '}
        ({person.email})
      </p>
    </>
  );
}

```
Here, `e.target.name` refers to the `name` property given to the `<input>` DOM element.
## Updating a nested object [](https://react.dev/learn/updating-objects-in-state#updating-a-nested-object "Link for Updating a nested object")
Рассмотрим вложенную структуру объекта, такую как:

Если вы хотите обновить `person.artwork.city`, понятно, как это сделать с мутацией:

Но в React состояние считается неизменным! Чтобы изменить город, вам сначала нужно создать новый объект изображения (предварительно заполненный данными из предыдущего), а затем создать новый объект человека, который указывает на новое изображение:

Это становится немного многословным, но это хорошо работает во многих случаях
```js
import { useState } from 'react';

export default function Form() {
  const [person, setPerson] = useState({
    name: 'Niki de Saint Phalle',
    artwork: {
      title: 'Blue Nana',
      city: 'Hamburg',
      image: 'https://i.imgur.com/Sd1AgUOm.jpg',
    }
  });

  function handleNameChange(e) {
    setPerson({
      ...person,
      name: e.target.value
    });
  }

  function handleTitleChange(e) {
    setPerson({
      ...person,
      artwork: {
        ...person.artwork,
        title: e.target.value
      }
    });
  }

  function handleCityChange(e) {
    setPerson({
      ...person,
      artwork: {
        ...person.artwork,
        city: e.target.value
      }
    });
  }

  function handleImageChange(e) {
    setPerson({
      ...person,
      artwork: {
        ...person.artwork,
        image: e.target.value
      }
    });
  }

  return (
    <>
      <label>
        Name:
        <input
          value={person.name}
          onChange={handleNameChange}
        />
      </label>
      <label>
        Title:
        <input
          value={person.artwork.title}
          onChange={handleTitleChange}
        />
      </label>
      <label>
        City:
        <input
          value={person.artwork.city}
```

#### Objects are not really nested
Объект, подобный этому, появляется "вложеным" в коде:
![[Pasted image 20240526194342.png]]
Тем не менее, "nesting" - это неточный способ думать о том, как ведут себя объекты. Когда код выполняется, нет такого понятия, как "nested" объект. Вы действительно смотрите на два разных объекта:
![[Pasted image 20240526194431.png]]
Объект obj1 не находится «внутри» obj2. Например, obj3 тоже может «указывать» на obj1:
Если бы вы изменили obj3.artwork.city, это повлияло бы как на obj2.artwork.city, так и на obj1.city. Это связано с тем, что obj3.artwork, obj2.artwork и obj1 — это один и тот же объект. Это трудно увидеть, если представить объекты как «вложенные». Вместо этого это отдельные объекты, «направляющие» друг на друга свойствами.
### Write concise update logic with Immer [](https://react.dev/learn/updating-objects-in-state#write-concise-update-logic-with-immer "Link for Write concise update logic with Immer")
Если ваш state глубоко вложен, вы можете рассмотреть возможность [его сгладивания.](https://react.dev/learn/choosing-the-state-structure#avoid-deeply-nested-state) Но если вы не хотите изменять структуру своего состояния, вы можете предпочесть ярлык для вложенных спредов. [Immer](https://github.com/immerjs/use-immer) - это популярная библиотека, которая позволяет писать с использованием удобного, но мутируемого синтаксиса и заботится о создании копий для вас. С Immer код, который вы пишете, выглядит так, как будто вы "нарушаете правила" и мутируете объект:
![[Pasted image 20240526194946.png]]
Но в отличие от обычной мутации, она не перезаписывает прошлое состояние!

Черновик, предоставленный Immer, представляет собой особый тип объекта, называемый [Proxy](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Proxy), который «записывает» все, что вы с ним делаете. Вот почему вы можете свободно мутировать его сколько угодно! Под капотом Immer определяет, какие части черновика были изменены, и создает совершенно новый объект, содержащий ваши изменения.

Чтобы попробовать Immer:

1. Запустите `npm install use-immer`, чтобы добавить Immer в качестве зависимости
2. Then replace `import { useState } from 'react'` with `import { useImmer } from 'use-immer'`

Вот приведенный выше пример, преобразованный в Immer:
```js
import { useImmer } from 'use-immer';

export default function Form() {
  const [person, updatePerson] = useImmer({
    name: 'Niki de Saint Phalle',
    artwork: {
      title: 'Blue Nana',
      city: 'Hamburg',
      image: 'https://i.imgur.com/Sd1AgUOm.jpg',
    }
  });

  function handleNameChange(e) {
    updatePerson(draft => {
      draft.name = e.target.value;
    });
  }

  function handleTitleChange(e) {
    updatePerson(draft => {
      draft.artwork.title = e.target.value;
    });
  }

  function handleCityChange(e) {
    updatePerson(draft => {
      draft.artwork.city = e.target.value;
    });
  }

  function handleImageChange(e) {
    updatePerson(draft => {
      draft.artwork.image = e.target.value;
    });
  }

  return (
    <>
      <label>
        Name:
        <input
          value={person.name}
          onChange={handleNameChange}
        />
      </label>
      <label>
        Title:
        <input
          value={person.artwork.title}
          onChange={handleTitleChange}
        />
      </label>
      <label>
        City:
        <input
          value={person.artwork.city}
          onChange={handleCityChange}
        />
      </label>
      <label>
        Image:
        <input
          value={person.artwork.image}
          onChange={handleImageChange}
        />
      </label>
      <p>
        <i>{person.artwork.title}</i>
        {' by '}
        {person.name}
        <br />
        (located in {person.artwork.city})
      </p>
      <img 
        src={person.artwork.image} 
        alt={person.artwork.title}
      />
    </>
  );
}

```
Обратите внимание, насколько лаконичнее стали обработчики событий. Вы можете сколько угодно смешивать и сочетать useState и useImmer в одном компоненте. Immer — отличный способ сохранить краткость обработчиков обновлений, особенно если в вашем состоянии есть вложенность, а копирование объектов приводит к повторяющемуся коду.
#### Why is mutating state not recommended in React?
Есть несколько причин:
- Отладка: если вы используете console.log и не меняете состояние, ваши прошлые журналы не будут затерты более поздними изменениями состояния. Таким образом, вы можете четко видеть, как менялось состояние между рендерами.
- - **Оптимизация:** Общие [стратегии оптимизации](https://react.dev/reference/react/memo) React основаны на пропуске работы, если предыдущие реквизиты или состояние совпадают со следующими. Если вы никогда не мутируете состояние, очень быстро проверить, были ли какие-либо изменения. Если `prevObj === obj`, вы можете быть уверены, что внутри него ничего не могло измениться.
- - **Новые функции:** Новые функции React, которые мы создаем, основаны на том, что состояние [рассматривается как снимок.](https://react.dev/learn/state-as-a-snapshot) Если вы мутируете прошлые версии состояния, это может помешать вам использовать новые функции.
- - **Изменения требований:** Некоторые функции приложения, такие как реализация Undo/Redo, показ истории изменений или позволяет пользователю сбросить форму до более ранних значений, легче сделать, когда ничего не мутировано. Это связано с тем, что вы можете хранить прошлые копии состояния в памяти и повторно использовать их при необходимости. Если вы начинаете с мутативного подхода, такие функции может быть трудно добавить позже.
- **Более простая реализация:** поскольку React не полагается на мутацию, ему не нужно делать ничего особенного с вашими объектами. Ему не нужно захватывать их свойства, всегда заворачивать их в прокси или делать другую работу по инициализации, как это делают многие "реактивные" решения. Именно поэтому React позволяет поставить в состояние любой объект - независимо от того, насколько он большой - без дополнительных подводных камней производительности или правильности.
На практике вы часто можете "уйти" с мутирующим состоянием в React, но мы настоятельно рекомендуем вам не делать этого, чтобы вы могли использовать новые функции React, разработанные с этим подходом. Будущие участники и, возможно, даже ваш будущий я будут вам благодарны!
