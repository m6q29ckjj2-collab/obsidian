Установка переменной состояния постанет в очередь на другой рендеринг. Но иногда вы можете захотеть выполнить несколько операций с значением, прежде чем ставить в очередь следующий рендер. Для этого полезно понять, как пакеты React обновляют состояние.
## React batches state updates [](https://react.dev/learn/queueing-a-series-of-state-updates#react-batches-state-updates "Link for React batches state updates")
Вы можете ожидать, что нажатие кнопки «+3» увеличит счетчик три раза, поскольку он трижды вызывает setNumber(number + 1):
```js
import { useState } from 'react';

export default function Counter() {
  const [number, setNumber] = useState(0);

  return (
    <>
      <h1>{number}</h1>
      <button onClick={() => {
        setNumber(number + 1);
        setNumber(number + 1);
        setNumber(number + 1);
      }}>+3</button>
    </>
  )
}

```
Однако, как вы, возможно, помните из предыдущего раздела, [значения состояния каждого рендера фиксированы](https://react.dev/learn/state-as-a-snapshot#rendering-takes-a-snapshot-in-time), поэтому значение числа внутри обработчика событий первого рендера всегда равно `0`, независимо от того, сколько раз вы вызываете `setNumber(1)`:
Но здесь есть еще один фактор. **React ждет, пока не запустится _весь_ код в обработчиках событий, прежде чем обрабатывать обновления состояния.** Вот почему повторный рендеринг происходит только _после_ всех этих вызовов `setNumber()`.
Это может напомнить вам о официанте, принимаю принимаю принимаю заказ в ресторане. Официант не бежит на кухню, когда упоминает ваше первое блюдо! Вместо этого они позволяют вам завершить ваш заказ, позволяют вам вносить в него изменения и даже принимать заказы от других людей за столом.
![[Pasted image 20240526184245.png]]
Это позволяет обновлять несколько переменных состояния, даже из нескольких компонентов, не вызывая слишком большого количества [повторных рендеров.](https://react.dev/learn/render-and-commit#re-renders-when-state-updates) Но это также означает, что пользовательский интерфейс не будет обновлен до тех пор_, пока_ ваш обработчик событий и любой код в нем не будет завершен. Такое поведение, также известное как **пакетирование,** заставляет ваше приложение React работать намного быстрее. Он также избегает работы с запутанными "полузаконченными" рендерами, когда были обновлены только некоторые из переменных.
**React не пакетирует _несколько_ преднамеренных событий, таких как клики** - каждый клик обрабатывается отдельно. Будьте уверены, что React выполняет пакетирование только тогда, когда это в целом безопасно. Это гарантирует, что, например, если первый нажмок кнопки отключит форму, второй щелчок не отправит ее снова.
## Updating the same state multiple times before the next render [](https://react.dev/learn/queueing-a-series-of-state-updates#updating-the-same-state-multiple-times-before-the-next-render "Link for Updating the same state multiple times before the next render")
Это необычный случай использования, но если вы хотите обновить одну и ту же переменную состояния несколько раз перед следующим рендером, вместо передачи _следующего значения состояния_, такого как `setNumber(number + 1)`, вы можете передать _функцию_, которая вычисляет следующее состояние на основе предыдущего в очереди, например `setNumber(n => n + 1)`). Это способ сказать React "делать что-то со значением состояния", а не просто заменять его.

Попробуйте увелить счетчик сейчас:
```js
import { useState } from 'react';

export default function Counter() {
  const [number, setNumber] = useState(0);

  return (
    <>
      <h1>{number}</h1>
      <button onClick={() => {
        setNumber(n => n + 1);
        setNumber(n => n + 1);
        setNumber(n => n + 1);
      }}>+3</button>
    </>
  )
}

```
Here, `n => n + 1` is called an **updater function.** When you pass it to a state setter:
1. React ставит эту функцию в очередь для обработки после запуска всего другого кода в обработчике событий.
2. 1. Во время следующего рендеринга React проходит через очередь и дает вам окончательное обновленное состояние.
Вот как React работает с этими строками кода при выполнении обработчика событий:
1. `setNumber(n => n + 1)`: `n => n + 1` is a function. React adds it to a queue.
2. `setNumber(n => n + 1)`: `n => n + 1` is a function. React adds it to a queue.
3. `setNumber(n => n + 1)`: `n => n + 1` is a function. React adds it to a queue.
Когда вы вызываете useState во время следующего рендеринга, React проходит через очередь. Предыдущее числовое состояние было 0, поэтому именно это React передает первой функции обновления в качестве аргумента n. Затем React принимает возвращаемое значение вашей предыдущей функции обновления и передает его следующему модулю обновления как n и так далее:
![[Pasted image 20240526184650.png]]
React сохраняет 3 как конечный результат и возвращает его из useState.

Вот почему нажатие «+3» в приведенном выше примере правильно увеличивает значение на 3.
### What happens if you update state after replacing it [](https://react.dev/learn/queueing-a-series-of-state-updates#what-happens-if-you-update-state-after-replacing-it "Link for What happens if you update state after replacing it")
А как насчет этого обработчика событий? Как вы думаете, какой номер будет на следующем рендере?
![[Pasted image 20240526184806.png]]
```js
import { useState } from 'react';

export default function Counter() {
  const [number, setNumber] = useState(0);

  return (
    <>
      <h1>{number}</h1>
      <button onClick={() => {
        setNumber(number + 5);
        setNumber(n => n + 1);
      }}>Increase the number</button>
    </>
  )
}

```
Вот что этот обработчик событий говорит React сделать:
1.  `setNumber(number + 5)`: `number` is `0`, so `setNumber(0 + 5)`. React adds _“replace with `5`”_ to its queue.
2.  `setNumber(n => n + 1)`: `n => n + 1` is an updater function. React adds _that function_ to its queue.
Во время следующего рендеринга React проходит через очередь состояния:
![[Pasted image 20240526184928.png]]
React stores `6` as the final result and returns it from `useState`.
You may have noticed that `setState(5)` actually works like `setState(n => 5)`, but `n` is unused!
### What happens if you replace state after updating it [](https://react.dev/learn/queueing-a-series-of-state-updates#what-happens-if-you-replace-state-after-updating-it "Link for What happens if you replace state after updating it")
Let’s try one more example. What do you think `number` will be in the next render?
![[Pasted image 20240526185032.png]]
```js
import { useState } from 'react';

export default function Counter() {
  const [number, setNumber] = useState(0);

  return (
    <>
      <h1>{number}</h1>
      <button onClick={() => {
        setNumber(number + 5);
        setNumber(n => n + 1);
        setNumber(42);
      }}>Increase the number</button>
    </>
  )
}

```
Here’s how React works through these lines of code while executing this event handler:
1. `setNumber(number + 5)`: `number` is `0`, so `setNumber(0 + 5)`. React adds _“replace with `5`”_ to its queue.
2. `setNumber(n => n + 1)`: `n => n + 1` is an updater function. React adds _that function_ to its queue.
3. `setNumber(42)`: React adds _“replace with `42`”_ to its queue.
During the next render, React goes through the state queue:
![[Pasted image 20240526185115.png]]
Then React stores `42` as the final result and returns it from `useState`.
Подводя итог, вот как вы можете подумать о том, что вы передаете в установитель состояния `setNumber`:
- **An updater function** (e.g. `n => n + 1`) gets added to the queue.
- - **Любое другое значение** (например, число 5) добавляет в очередь "заменить на `5`", игнорируя то, что уже поставлено в очередь.
После завершения обработки событий React запустит повторный рендер. Во время повторного рендеринга React обработает очередь. Функции обновления выполняются во время рендеринга, поэтому **функции обновления должны быть [чистыми](https://react.dev/learn/keeping-components-pure)** и _возвращать_ только результат. Не пытайтесь установить состояние изнутри или создать другие побочные эффекты. В строгом режиме React будет запускать каждую функцию обновления дважды (но отбрасывать второй результат), чтобы помочь вам найти ошибки.
### Naming conventions [](https://react.dev/learn/queueing-a-series-of-state-updates#naming-conventions "Link for Naming conventions")
Обычно можно назвать аргумент функции обновления первыми буквами соответствующей переменной состояния:
![[Pasted image 20240526185406.png]]
Если вы предпочитаете более подробный код, другое распространенное соглашение — повторять полное имя переменной состояния, например setEnabled(enabled => !enabled), или использовать префикс, например setEnabled(prevEnabled => !prevEnabled).
