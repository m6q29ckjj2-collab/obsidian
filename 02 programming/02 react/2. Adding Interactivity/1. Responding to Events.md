React позволяет добавлять _обработчики событий_ в JSX. Обработчики событий - это ваши собственные функции, которые будут запускаться в ответ на такие взаимодействия, как клик, наведение, фокусировка входных данных формы и так далее.  :

Event handler functions:

- Обычно определяются _внутри_ ваших компонентов.
- Имеют имена, которые начинаются с `handle`, за которыми следует имя события.
By convention, it is common to name event handlers as `handle` followed by the event name. You’ll often see `onClick={handleClick}`, `onMouseEnter={handleMouseEnter}`, and so on.

### Передача обработчиков событий в качестве реквизита[](https://react.dev/learn/responding-to-events#passing-event-handlers-as-props "Ссылка на проходные обработчики событий в качестве реквизита")
Часто вы хотите, чтобы родительский компонент указывал обработчик событий дочернего компонента. Рассмотрим кнопки: в зависимости от того, где вы используете компонент `Button`, вы можете выполнить другую функцию - возможно, одна воспроизводит фильм, а другая загружает изображение.

Для этого передайте реквизит, который компонент получает от своего родителя в качестве обработчика событий:

Если вы используете систему дизайна, такие компоненты, как кнопки, обычно содержат стиль, но не определяют поведение. Вместо этого такие компоненты, как PlayButton и UploadButton, будут передавать обработчики событий.

Убедитесь, что вы используете соответствующие HTML-теги для обработчиков событий. Например, для обработки кликов используйте \<button onClick={handleClick}> вместо \<div onClick={handleClick}>. Использование \<button> в реальном браузере позволяет использовать встроенные функции браузера, такие как навигация с помощью клавиатуры. Если вам не нравится стиль кнопки по умолчанию в браузере и вы хотите, чтобы она больше походила на ссылку или другой элемент пользовательского интерфейса, вы можете добиться этого с помощью CSS. Узнайте больше о написании доступной разметки.

## Распространение событий[](https://react.dev/learn/responding-to-events#event-propagation "Ссылка для распространения событий")
Оботчики событий также будут ловить события от любых дочерних компонентов, которые могут быть у вашего компонента. Мы говорим, что событие "пузыри" или "разлагается" вверх по дереву: оно начинается с того места, где произошло событие, а затем поднимается вверх по дереву.
Если вы нажмете на любую кнопку, сначала запустится ее onClick, а затем — onClick родительского \<div>. Таким образом, появятся два сообщения. Если вы щелкнете по самой панели инструментов, запустится только onClick родительского \<div>.

> Все события распространяются в React, кроме `onScroll`, который работает только с тегом JSX, к которому вы его прикрепляете.

### Stop Propagation
Обыватели событий получают **объект события** в качестве единственного аргумента. По соглашению, его обычно называют `e`, что означает "событие". Вы можете использовать этот объект для чтения информации о событии.

Этот объект события также позволяет остановить распространение. Если вы хотите предотвратить попадание события в родительские компоненты, вам необходимо вызвать `e.stopPropagation()`
Когда вы нажимаете на кнопку:

1. React вызывает обработчик `onClick`, переданный в `<button>`.
2. Этот обработчик, определенный в `Button`, делает следующее:
    - Вызовы `e.stopPropagation()`, предотвращая дальнейшее пузырькания события.
    - Вызывает функцию `onClick`, которая является реквизитом, переданным из компонента `Toolbar`.
3. Эта функция, определенная в компоненте `Toolbar`, отображает собственное оповещение кнопки.
4. Since the propagation was stopped, the parent `<div>`’s `onClick` handler does _not_ run.
В результате применения e.stopPropagation() при нажатии на кнопки теперь отображается только одно предупреждение (от \<button>), а не два из них (от \<button> и родительской панели инструментов \<div>). Нажатие кнопки — это не то же самое, что нажатие на окружающую панель инструментов, поэтому остановка распространения имеет смысл для этого пользовательского интерфейса.

#### Захват событий фазы[](https://react.dev/learn/responding-to-events#capture-phase-events "Ссылка для событий фазы захвата")
В редких случаях вам может потребоваться перехватить все события в дочерних элементах, даже если они прекратили распространение. Например, возможно, вы хотите регистрировать каждый клик в аналитике, независимо от логики распространения. Вы можете сделать это, добавив Capture в конце имени события:
![[Pasted image 20240526130652.png]]
Каждое событие распространяется в три этапа:

1. Он перемещается вниз, вызывая всех обработчиков `onClickCapture`.
2. It runs the clicked element’s `onClick` handler.
3. Он движется вверх, вызывая все обработчики `onClick`.
События захвата полезны для кода, такого как маршрутизаторы или аналитика, но вы, вероятно, не будете использовать их в коде приложения.
### Передача обработчиков в качестве альтернативы распространению[](https://react.dev/learn/responding-to-events#passing-handlers-as-alternative-to-propagation "Ссылка для обработчиков передачи в качестве альтернативы распространению")
Вы также можете добавить больше кода в этот обработчик, прежде чем вызывать родительский обработчик событий `onClick`. Эта модель является _альтернативой_ распространению. Это позволяет дочернему компоненту обрабатывать событие, а также позволяет родительскому компоненту указывать некоторое дополнительное поведение. В отличие от распространения, это не автоматически. Но преимущество этого шаблона заключается в том, что вы можете четко следовать всей цепочке кода, который выполняется в результате какого-либо события.

Если вы полагаетесь на распространение и трудно отследить, какие обработчики выполняют и почему, попробуйте вместо этого этот подход.

### Предотвращение поведения по умолчанию[](https://react.dev/learn/responding-to-events#preventing-default-behavior "Ссылка для предотвращения поведения по умолчанию")
Некоторые события браузера связаны с поведением по умолчанию. Например, событие отправки `<form>`, которое происходит при нажатии кнопки внутри него, по умолчанию перезагружает всю страницу:
You can call `e.preventDefault()` on the event object to stop this from happening:
Не путайте e.stopPropagation() и e.preventDefault(). Они оба полезны, но не связаны друг с другом:
- [`e.stopPropagation()`](https://developer.mozilla.org/docs/Web/API/Event/stopPropagation)останавливает запуск обработчиков событий, прикрепленных к вышеуказанным меткам.
- [`e.preventDefault()`](https://developer.mozilla.org/docs/Web/API/Event/preventDefault) предотвращает поведение браузера по умолчанию для нескольких событий, которые его имеют.
## Могут ли обработчики событий иметь побочные эффекты?[](https://react.dev/learn/responding-to-events#can-event-handlers-have-side-effects "Ссылка на обработчики событий могут иметь побочные эффекты?")
Абсолютно! Обработчики событий - лучшее место для побочных эффектов.

В отличие от функций рендеринга, обработчики событий не должны быть [чистыми](https://react.dev/learn/keeping-components-pure), поэтому это отличное место, чтобы что-то _изменить_ - например, изменить значение ввода в ответ на ввод или изменить список в ответ на нажатие кнопки. Однако, чтобы изменить некоторую информацию, вам сначала нужно как-то ее сохранить. В React это делается с помощью [состояния, памяти компонента.](https://react.dev/learn/state-a-components-memory) Вы узнаете все об этом на следующей странице.
