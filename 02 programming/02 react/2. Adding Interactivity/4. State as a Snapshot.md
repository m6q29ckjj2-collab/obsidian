Переменные состояния могут выглядеть как обычные переменные JavaScript, которые вы можете читать и записывать. Тем не менее, состояние ведет себя больше как снимок. Установка не меняет уже имею переменную состояния, которая у вас есть, а вместо этого вызывает повторный рендер.
## Setting state triggers renders [](https://react.dev/learn/state-as-a-snapshot#setting-state-triggers-renders "Link for Setting state triggers renders")
Вы можете подумать, что ваш пользовательский интерфейс меняется непосредственно в ответ на событие пользователя, как на клик. В React это работает немного иначе, чем эта ментальная модель. На предыдущей странице вы видели, что [состояние настройки запрашивает повторный рендер](https://react.dev/learn/render-and-commit#step-1-trigger-a-render) из React. Это означает, что для того, чтобы интерфейс отреагировал на событие, вам нужно _обновить состояние_.
n this example, when you press “send”, `setIsSent(true)` tells React to re-render the UI:
```js
import { useState } from 'react';

export default function Form() {
  const [isSent, setIsSent] = useState(false);
  const [message, setMessage] = useState('Hi!');
  if (isSent) {
    return <h1>Your message is on its way!</h1>
  }
  return (
    <form onSubmit={(e) => {
      e.preventDefault();
      setIsSent(true);
      sendMessage(message);
    }}>
      <textarea
        placeholder="Message"
        value={message}
        onChange={e => setMessage(e.target.value)}
      />
      <button type="submit">Send</button>
    </form>
  );
}

function sendMessage(message) {
  // ...
}

```
Вот что происходит, когда вы нажимаете кнопку:
1. 1. Выполняется обработчик событий `onSubmit`.
2. setIsSent(true) устанавливает для isSent значение true и ставит в очередь новый рендеринг.
3. React повторно рендерит компонент в соответствии с новым значением `isSent`.
Давайте подробнее посмотрим на отношения между состоянием и рендерингом.
## Rendering takes a snapshot in time [](https://react.dev/learn/state-as-a-snapshot#rendering-takes-a-snapshot-in-time "Link for Rendering takes a snapshot in time")
["Рендеринг"](https://react.dev/learn/render-and-commit#step-2-react-renders-your-components) означает, что React вызывает ваш компонент, который является функцией. JSX, который вы возвращаете из этой функции, похож на снимок пользовательского интерфейса во времени. Его реквизиты, обработчики событий и локальные переменные были рассчитаны **с использованием его состояния на момент рендеринга.**
В отличие от фотографии или кадра фильма, "снимок" пользовательского интерфейса, который вы возвращаете, является интерактивным. Он включает в себя логику, такую как обработчики событий, которые определяют, что происходит в ответ на входные данные. React обновляет экран, чтобы соответствовать этому снимку, и соединяет обработчики событий. В результате нажатие кнопки запустит обработчик кликов из вашего JSX.
Когда React повторно рендерирует компонент:
1. React снова вызывает вашу функцию.
2. Ваша функция возвращает новый снимок JSX.
3. Затем React обновляет экран в соответствии со снимком, который была возвращена ваша функция.

Как память компонента, состояние не похоже на обычную переменную, которая исчезает после возврата функции. Состояние на самом деле "живет" в самом React - как будто на полке! - за пределами вашей функции. Когда React вызывает ваш компонент, он дает вам снимок состояния для этого конкретного рендеринга. Ваш компонент возвращает снимок пользовательского интерфейса со свежим набором реквизитов и обработчиков событий в JSX, все они рассчитываются **с использованием значений состояния из этого рендеринга!**
![[Pasted image 20240526172412.png]]
Вот небольшой эксперимент, который покажет вам, как это работает. В этом примере вы могли бы ожидать, что нажатие кнопки «+3» увеличит счетчик в три раза, поскольку он трижды вызывает setNumber(number + 1).
```js 
import { useState } from 'react';

export default function Counter() {
  const [number, setNumber] = useState(0);

  return (
    <>
      <h1>{number}</h1>
      <button onClick={() => {
        setNumber(number + 1);
        setNumber(number + 1);
        setNumber(number + 1);
      }}>+3</button>
    </>
  )
}
```
Обратите внимание, что число увеличивается только один раз за клик!
Установка состояния меняет его только для следующего рендеринга. Во время первого рендеринга число было равно 0. Вот почему в обработчике onClick этого рендеринга значение числа по-прежнему равно 0 даже после вызова setNumber(number + 1):
Вот что обработчик кликов этой кнопки говорит React сделать:
1. `setNumber(number + 1)`: `number` is `0` so `setNumber(0 + 1)`.
	  - React готовится изменить число на 1 при следующем рендеринге.
2. `setNumber(number + 1)`: `number` is `0` so `setNumber(0 + 1)`.
	-  React готовится изменить число на 1 при следующем рендеринге.
3. `setNumber(number + 1)`: `number` is `0` so `setNumber(0 + 1)`.
	-   React готовится изменить число на 1 при следующем рендеринге.
Несмотря на то, что вы вызвали setNumber(number + 1) три раза, в этом рендере номер обработчика событий всегда равен 0, поэтому вы трижды устанавливаете состояние в 1. Вот почему после завершения обработчика событий React повторно отображает компонент с номером, равным 1, а не 3.
Вы также можете визуализировать это, мысленно заменяя переменные состояния их значениями в вашем коде. Поскольку для этого рендеринга числовая переменная состояния равна 0, ее обработчик событий выглядит следующим образом:

## State over time [](https://react.dev/learn/state-as-a-snapshot#state-over-time "Link for State over time")
Значение переменной состояния никогда не меняется в процессе рендеринга, даже если код ее обработчика событий является асинхронным. Внутри onClick этого рендеринга значение числа продолжает оставаться равным 0 даже после вызова setNumber(number + 5). Его значение было «фиксированным», когда React «сделал снимок» пользовательского интерфейса, вызвав ваш компонент.
Вот пример того, как это делает ваших обработчиков событий менее склонными к ошибкам при определении времени. Ниже приведена форма, которая отправляет сообщение с пятисекундной задержкой. Представьте себе этот сценарий:
1. Вы нажимаете кнопку "Отправить", отправляя "Привет" Алисе.
2. Прежде чем закончится пятисекундная задержка, вы измените значение поля "Ко" на "Боб".
**React сохраняет значения состояния "фиксированными" в обработчиках событий одного рендера.** Вам не нужно беспокоиться о том, изменилось ли состояние во время работы кода.
Но что, если вы хотите прочитать последнее состояние перед повторным рендером? Вы захотите использовать [функцию обновления состояния](https://react.dev/learn/queueing-a-series-of-state-updates), о которой говорится на следующей странице!
