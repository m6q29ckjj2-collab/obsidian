`useCallback` is a React Hook that lets you cache a function definition between re-renders.
![[Pasted image 20240607104852.png]]
## Reference [](https://react.dev/reference/react/useCallback#reference "Link for Reference")
### `useCallback(fn, dependencies)`
Call `useCallback` at the top level of your component to cache a function definition between re-renders:
![[Pasted image 20240607104932.png]]
- fn: значение функции, которое вы хотите кэшировать. Он может принимать любые аргументы и возвращать любые значения. React вернет (не вызовет!) вашу функцию во время первоначального рендеринга. При следующем рендеринге React снова предоставит вам ту же функцию, если зависимости не изменились с момента последнего рендеринга. В противном случае он предоставит вам функцию, которую вы передали во время текущего рендеринга, и сохранит ее на случай, если ее можно будет повторно использовать позже. React не будет вызывать вашу функцию. Функция возвращается вам, чтобы вы могли решить, когда и следует ли ее вызывать.
- зависимости: список всех реактивных значений, на которые ссылаются внутри кода fn. Реактивные значения включают реквизиты, состояние и все переменные и функции, объявленные непосредственно внутри тела вашего компонента. Если ваш линтер настроен для React, он проверит, что каждое реактивное значение правильно указано как зависимость. Список зависимостей должен иметь постоянное количество элементов и быть записан в строке, например [dep1, dep2, dep3]. React сравнит каждую зависимость с ее предыдущим значением, используя алгоритм сравнения Object.is.
#### Returns [](https://react.dev/reference/react/useCallback#returns "Link for Returns")
On the initial render, `useCallback` returns the `fn` function you have passed.
Во время последующих рендерингов он либо вернет уже сохраненную функцию fn из последнего рендеринга (если зависимости не изменились), либо вернет функцию fn, которую вы передали во время этого рендеринга.
#### Caveats [](https://react.dev/reference/react/useCallback#caveats "Link for Caveats")
- useCallback — это хук, поэтому вы можете вызывать его только на верхнем уровне вашего компонента или собственных хуков. Вы не можете вызывать его внутри циклов или условий. Если вам это нужно, извлеките новый компонент и переместите в него состояние.
- React не будет удалять кэшированную функцию, если для этого нет особой причины. Например, при разработке React выбрасывает кеш, когда вы редактируете файл вашего компонента. Как при разработке, так и при производстве React выбросит кеш, если ваш компонент приостанавливается во время первоначального монтирования. В будущем React может добавить больше функций, которые используют преимущество выбрасывания кеша — например, если в будущем React добавит встроенную поддержку виртуализированных списков, имело бы смысл выбросить кеш для элементов, которые прокручиваются за пределы. окно просмотра виртуализированной таблицы. Это должно соответствовать вашим ожиданиям, если вы полагаетесь на useCallback как на оптимизацию производительности. В противном случае более подходящим может оказаться переменная состояния или ссылка.
## Usage [](https://react.dev/reference/react/useCallback#usage "Link for Usage")
### Skipping re-rendering of components [](https://react.dev/reference/react/useCallback#skipping-re-rendering-of-components "Link for Skipping re-rendering of components")
При оптимизации производительности рендеринга иногда требуется кэшировать функции, которые вы передаете дочерним компонентам. Давайте сначала рассмотрим синтаксис того, как это сделать, а затем посмотрим, в каких случаях это полезно.

Чтобы кэшировать функцию между повторными рендерингами вашего компонента, оберните ее определение в хук useCallback:

![[Pasted image 20240607105657.png]]
You need to pass two things to `useCallback`:
1. A function definition that you want to cache between re-renders.
2. A <mark style="background: #FFB86CA6;"><font color="#974806">list of dependencies</font></mark> including every value within your component that’s used inside your function.
On the initial render, the returned function you’ll get from `useCallback` will be the function you passed.
При следующих рендерингах React сравнит зависимости с зависимостями, которые вы передали во время предыдущего рендеринга. Если ни одна из зависимостей не изменилась (по сравнению с Object.is), useCallback вернет ту же функцию, что и раньше. В противном случае useCallback вернет функцию, которую вы передали при этом рендеринге.
Другими словами, useCallback кэширует функцию между повторными рендерингами до тех пор, пока ее зависимости не изменятся.
**Давайте рассмотрим пример, чтобы увидеть, когда это полезно.**

Предположим, вы передаете функцию handleSubmit из ProductPage в компонент ShippingForm:
![[Pasted image 20240607110225.png]]
Вы заметили, что при переключении свойства темы приложение на мгновение зависает, но если вы удалите <ShippingForm /> из JSX, это произойдет быстро. Это говорит о том, что стоит попытаться оптимизировать компонент ShippingForm.
По умолчанию, когда компонент перерисовывается, React рекурсивно перерисовывает все его дочерние элементы. Вот почему при повторной визуализации ProductPage с использованием другой темы компонент ShippingForm также перерисовывается. Это подходит для компонентов, повторный рендеринг которых не требует больших вычислений. Но если вы убедились, что повторный рендеринг происходит медленно, вы можете указать ShippingForm пропустить повторный рендеринг, если его реквизиты такие же, как и при последнем рендеринге, обернув его в memo:
![[Pasted image 20240607110523.png]]
Благодаря этому изменению ShippingForm пропустит повторный рендеринг, если все его реквизиты такие же, как и при последнем рендеринге. Именно тогда кэширование функции становится важным! Допустим, вы определили handleSubmit без useCallback:
![[Pasted image 20240607110555.png]]
В JavaScript функция () {} или () => {} всегда создает другую функцию, подобно тому, как литерал объекта {} всегда создает новый объект. Обычно это не является проблемой, но это означает, что реквизиты ShippingForm никогда не будут прежними, и ваша оптимизация заметок не будет работать. Вот тут-то и пригодится useCallback:
![[Pasted image 20240607110718.png]]
Обертывая handleSubmit в useCallback, вы гарантируете, что это одна и та же функция между повторными рендерингами (пока зависимости не изменятся). Вам не обязательно заключать функцию в useCallback, если вы не делаете это по какой-то конкретной причине. В этом примере причина в том, что вы передаете его компоненту, завернутому в memo, и это позволяет ему пропустить повторный рендеринг. Есть и другие причины, по которым вам может понадобиться useCallback, которые описаны далее на этой странице.
> [!note] 
> Вам следует полагаться только на useCallback как на оптимизацию производительности. Если ваш код не работает без него, найдите основную проблему и сначала устраните ее. Затем вы можете добавить useCallback обратно.

 > [!important] 
> Вы часто будете видеть useMemo рядом с useCallback. Они оба полезны, когда вы пытаетесь оптимизировать дочерний компонент. Они позволяют вам запоминать (или, другими словами, кэшировать) то, что вы передаете:
> ![[Pasted image 20240607111003.png]]
>  Разница в том, что они позволяют вам кэшировать:
>   - useMemo кэширует результат вызова вашей функции. В этом примере он кэширует результат вызова ComputeRequirements(product), чтобы он не менялся, пока продукт не изменился. Это позволяет передать объект требований без необходимости повторной визуализации ShippingForm. При необходимости React вызовет функцию, которую вы передали во время рендеринга, для вычисления результата.
>  - useCallback кэширует саму функцию. В отличие от useMemo, он не вызывает предоставленную вами функцию. Вместо этого он кэширует предоставленную вами функцию, чтобы сам handleSubmit не менялся, пока не изменился ProductId или реферер. Это позволяет передать функцию handleSubmit без излишнего повторного рендеринга ShippingForm. Ваш код не запустится, пока пользователь не отправит форму.
>  Если вы уже знакомы с useMemo, возможно, вам будет полезно подумать об useCallback следующим образом:
>  ![[Pasted image 20240607111408.png]]
>  [Read more about the difference between `useMemo` and `useCallback`.](https://react.dev/reference/react/useMemo#memoizing-a-function)

> [!important] 
> #### Should you add useCallback everywhere? [](https://react.dev/reference/react/useCallback#should-you-add-usecallback-everywhere "Link for Should you add useCallback everywhere?")
>  Если ваше приложение похоже на этот сайт и большинство взаимодействий являются грубыми (например, замена страницы или целого раздела), мемоизация обычно не нужна. С другой стороны, если ваше приложение больше похоже на редактор рисунков и большинство взаимодействий являются детальными (например, перемещение фигур), то мемоизация может оказаться очень полезной.
>  
>  Кэширование функции с помощью useCallback полезно только в нескольких случаях:
>  - Вы передаете его как свойство компоненту, завернутому в memo. Вы хотите пропустить повторный рендеринг, если значение не изменилось. Мемоизация позволяет вашему компоненту повторно отображать только в случае изменения зависимостей.
>  - Переданная вами функция позже используется как зависимость какого-либо хука. Например, от нее зависит другая функция, обернутая в useCallback, или вы зависите от этой функции из useEffect.
>  - 
> В других случаях обертывание функции в useCallback бесполезно. В этом нет никакого существенного вреда, поэтому некоторые команды предпочитают не думать об отдельных случаях и запоминать как можно больше. Обратной стороной является то, что код становится менее читабельным. Кроме того, не всякая мемоизация эффективна: одного «всегда нового» значения достаточно, чтобы нарушить мемоизацию всего компонента.
>
> Обратите внимание, что useCallback не препятствует созданию функции. Вы всегда создаете функцию (и это нормально!), но React игнорирует ее и возвращает вам кешированную функцию, если ничего не изменилось.
>
> **На практике вы можете сделать ненужным запоминание, если будете следовать нескольким принципам:**
> 1. Когда компонент визуально обертывает другие компоненты, позвольте ему принимать JSX в качестве дочерних компонентов. Затем, если компонент-оболочка обновляет свое собственное состояние, React знает, что его дочерние элементы не нуждаются в повторной визуализации.
