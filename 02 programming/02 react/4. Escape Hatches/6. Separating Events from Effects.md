Оборатчики событий повторно запускаются только при повторном выполнении того же взаимодействия. В отличие от обработчиков событий, эффекты повторно синхронизируются, если какое-то значение, которое они читают, например, проп или переменная состояния, отличается от того, что было во время последнего рендеринга. Иногда вам также нужно сочетание обоих моделей поведения: эффект, который повторяется в ответ на некоторые ценности, но не на другие. Эта страница научит вас, как это сделать.
## Choosing between event handlers and Effects [](https://react.dev/learn/separating-events-from-effects#choosing-between-event-handlers-and-effects "Link for Choosing between event handlers and Effects")
Во-первых, давайте подведем итоги разницы между обработчиками событий и эффектами.

Представьте, что вы внедряете компонент чата. Ваши требования выглядят следующим образом:

1. Ваш компонент должен автоматически подключаться к выбранной комнате чата.
2. Когда вы нажимаете кнопку "Отправить", он должен отправить сообщение в чат.

Скажем, вы уже реализовали код для них, но не знаете, куда его поместить. Должны ли вы использовать обработчики событий или эффекты? Каждый раз, когда вам нужно ответить на этот вопрос, подумайте, [_почему_ код должен работать.](https://react.dev/learn/synchronizing-with-effects#what-are-effects-and-how-are-they-different-from-events)
### Event handlers run in response to specific interactions [](https://react.dev/learn/separating-events-from-effects#event-handlers-run-in-response-to-specific-interactions "Link for Event handlers run in response to specific interactions")
С точки зрения пользователя, отправка сообщения должна происходить, _потому что_ была нажата конкретная кнопка "Отправить". Пользователь будет довольно расстроен, если вы отправите его сообщение в любое другое время или по любой другой причине. Вот почему отправка сообщения должна быть обработчиком событий. События позволяют обрабатывать конкретные взаимодействия:
![[Pasted image 20240528200531.png]]
Используя обработчик событий, вы можете быть уверены, что sendMessage(message) запустится только в том случае, если пользователь нажмет кнопку.
### Effects run whenever synchronization is needed [](https://react.dev/learn/separating-events-from-effects#effects-run-whenever-synchronization-is-needed "Link for Effects run whenever synchronization is needed")
Напомним, что вам также нужно поддерживать связь компонента с чатом. Куда уходит этот код?

_Причина_ запуска этого кода не в каком-то конкретном взаимодействии. Неважно, почему и как пользователь перешел на экран чата. Теперь, когда они смотрят на него и могут взаимодействовать с ним, компонент должен оставаться на связи с выбранным сервером чата. Даже если компонент чата был начальным экраном вашего приложения, и пользователь вообще не выполнял никаких взаимодействий, вам _все равно_ нужно будет подключиться. Вот почему это эффект:
![[Pasted image 20240528200704.png]]
С помощью этого кода вы можете быть уверены, что всегда есть активное соединение с выбранным в данный момент сервером чата, _независимо от_ конкретных взаимодействий, выполняемых пользователем. Независимо от того, открыл ли пользователь ваше приложение, выбрал другую комнату или перешел на другой экран и обратно, ваш эффект гарантирует, что компонент _останется синхронизированным_ с выбранной в данный момент комнатой и будет [повторно подключаться, когда это необходимо.](https://react.dev/learn/lifecycle-of-reactive-effects#why-synchronization-may-need-to-happen-more-than-once)
```js
import { useState, useEffect } from 'react';
import { createConnection, sendMessage } from './chat.js';

const serverUrl = 'https://localhost:1234';

function ChatRoom({ roomId }) {
  const [message, setMessage] = useState('');

  useEffect(() => {
    const connection = createConnection(serverUrl, roomId);
    connection.connect();
    return () => connection.disconnect();
  }, [roomId]);

  function handleSendClick() {
    sendMessage(message);
  }

  return (
    <>
      <h1>Welcome to the {roomId} room!</h1>
      <input value={message} onChange={e => setMessage(e.target.value)} />
      <button onClick={handleSendClick}>Send</button>
    </>
  );
}

export default function App() {
  const [roomId, setRoomId] = useState('general');
  const [show, setShow] = useState(false);
  return (
    <>
      <label>
        Choose the chat room:{' '}
        <select
          value={roomId}
          onChange={e => setRoomId(e.target.value)}
        >
          <option value="general">general</option>
          <option value="travel">travel</option>
          <option value="music">music</option>
        </select>
      </label>
      <button onClick={() => setShow(!show)}>
        {show ? 'Close chat' : 'Open chat'}
      </button>
      {show && <hr />}
      {show && <ChatRoom roomId={roomId} />}
    </>
  );
}

```
## Reactive values and reactive logic [](https://react.dev/learn/separating-events-from-effects#reactive-values-and-reactive-logic "Link for Reactive values and reactive logic")
Интуитивно можно сказать, что обработчики событий всегда срабатывают "вручную", например, нажатием кнопки. Эффекты, с другой стороны, "автоматические": они запускаются и повторно запускаются так часто, как это необходимо для синхронизации.

Есть более точный способ подумать об этом.
Свойства, состояние и переменные, объявленные внутри тела вашего компонента, называются реактивными значениями. В этом примере serverUrl не является реактивным значением, а roomId и message — таковыми. Они участвуют в потоке данных рендеринга:
![[Pasted image 20240528201018.png]]
Подобные реактивные значения могут измениться из-за повторного рендеринга. Например, пользователь может отредактировать сообщение или выбрать другой идентификатор комнаты в раскрывающемся списке. Обработчики событий и эффекты по-разному реагируют на изменения:
- **Логика внутри обработчиков событий _не реагирует._** Он не будет запускаться снова, если пользователь не выполнит то же самое взаимодействие (например, щелчок) снова. Обработчики событий могут читать реактивные значения, не «реагируя» на их изменения.
- **Логика внутри Effects является _реактивной._** Если ваш эффект читает реактивное значение, [вы должны указать его как зависимость.](https://react.dev/learn/lifecycle-of-reactive-effects#effects-react-to-reactive-values) Затем, если повторный рендер вызывает изменение этого значения, React повторно запустит логику вашего эффекта с новым значением.

Давайте вернемся к предыдущему примеру, чтобы проиллюстрировать это различие.
### Logic inside event handlers is not reactive [](https://react.dev/learn/separating-events-from-effects#logic-inside-event-handlers-is-not-reactive "Link for Logic inside event handlers is not reactive")
Взгляните на эту строку кода. Должна ли эта логика быть реактивной или нет?
![[Pasted image 20240528201501.png]]
С точки зрения пользователя изменение сообщения не означает, что он хочет отправить сообщение. Это означает только то, что пользователь печатает. Другими словами, логика, отправляющая сообщение, не должна быть реактивной. Он не должен запускаться снова только потому, что реактивное значение изменилось. Вот почему оно принадлежит обработчику событий:
![[Pasted image 20240528201607.png]]
Обработчики событий не являются реактивными, поэтому sendMessage(message) будет запускаться только тогда, когда пользователь нажимает кнопку «Отправить».
### Logic inside Effects is reactive [](https://react.dev/learn/separating-events-from-effects#logic-inside-effects-is-reactive "Link for Logic inside Effects is reactive")
Теперь вернемся к этим строкам:
![[Pasted image 20240528201653.png]]
С точки зрения пользователя, изменение roomId означает, что он хочет подключиться к другой комнате. Другими словами, логика подключения к комнате должна быть реактивной. Вы хотите, чтобы эти строки кода «не отставали» от реактивного значения и запускались снова, если это значение отличается. Вот почему он принадлежит эффекту:
![[Pasted image 20240528201830.png]]
Эффекты являются реактивными, поэтому createConnection(serverUrl, roomId) и Connection.connect() будут выполняться для каждого отдельного значения roomId. Your Effect сохраняет соединение чата синхронизированным с выбранной в данный момент комнатой.
## Extracting non-reactive logic out of Effects
Все становится сложнее, когда вы хотите смешать реактивную логику с нереактивной логикой.

Например, представьте, что вы хотите показать уведомление, когда пользователь подключается к чату. Вы читаете текущую тему (темную или светлую) из реквизита, чтобы показать уведомление в правильном цвете:

Однако тема — это реактивное значение (оно может измениться в результате повторного рендеринга), и [every reactive value read by an Effect must be declared as its dependency](https://react.dev/learn/lifecycle-of-reactive-effects#react-verifies-that-you-specified-every-reactive-value-as-a-dependency). Теперь вам нужно указать тему как зависимость вашего эффекта:

Поиграйте с этим примером и посмотрите, сможете ли вы обнаружить проблему с помощью этого пользовательского опыта:
Когда идентификатор комнаты изменится, чат снова подключится, как и следовало ожидать. Но поскольку тема также является зависимостью, чат также переподключается каждый раз, когда вы переключаетесь между темной и светлой темой. Это не здорово!
Другими словами, вы _не_ хотите, чтобы эта строка была реактивной, даже если она находится внутри эффекта (который является реактивным):

Вам нужен способ отделить эту нереактивную логику от реактивного эффекта вокруг нее.
### Declaring an Effect Event

Используйте специальный хук под названием [`useEffectEvent`](https://react.dev/reference/react/experimental_useEffectEvent), чтобы извлечь эту нереактивную логику из вашего эффекта:

Здесь onConnected называется событием эффекта. Это часть вашей логики Effect, но она больше похожа на обработчик событий. Логика внутри него не реактивна и всегда «видит» последние значения ваших реквизитов и состояния.

Теперь вы можете вызвать событие onConnected Effect изнутри вашего эффекта:
![[Pasted image 20240528202719.png]]
Это решает проблему. Обратите внимание, что вам пришлось удалить onConnected из списка зависимостей вашего эффекта. События эффектов не являются реактивными и должны быть исключены из зависимостей.
Убедитесь, что новое поведение работает так, как вы ожидаете:
```js
import { useState, useEffect } from 'react';
import { experimental_useEffectEvent as useEffectEvent } from 'react';
import { createConnection, sendMessage } from './chat.js';
import { showNotification } from './notifications.js';

const serverUrl = 'https://localhost:1234';

function ChatRoom({ roomId, theme }) {
  const onConnected = useEffectEvent(() => {
    showNotification('Connected!', theme);
  });

  useEffect(() => {
    const connection = createConnection(serverUrl, roomId);
    connection.on('connected', () => {
      onConnected();
    });
    connection.connect();
    return () => connection.disconnect();
  }, [roomId]);

  return <h1>Welcome to the {roomId} room!</h1>
}

export default function App() {
  const [roomId, setRoomId] = useState('general');
  const [isDark, setIsDark] = useState(false);
  return (
    <>
      <label>
        Choose the chat room:{' '}
        <select
          value={roomId}
          onChange={e => setRoomId(e.target.value)}
        >
          <option value="general">general</option>
          <option value="travel">travel</option>
          <option value="music">music</option>
        </select>
      </label>
      <label>
        <input
          type="checkbox"
          checked={isDark}
          onChange={e => setIsDark(e.target.checked)}
        />
        Use dark theme
      </label>
      <hr />
      <ChatRoom
        roomId={roomId}
        theme={isDark ? 'dark' : 'light'}
      />
    </>
  );
}

```
Вы можете думать о Events как о том, что они очень похожи на обработчики событий. Основное отличие заключается в том, что обработчики событий запускаются в ответ на взаимодействие с пользователем, в то время как события эффектов запускаются вами из эффектов. События эффектов позволяют "разорвать цепочку" между реактивностью эффектов и кодом, который не должен быть реактивным.

#### Is it okay to suppress the dependency linter instead?
В существующих кодовых базах иногда можно увидеть правило lint, подавленное следующим образом:
![[Pasted image 20240528210831.png]]
После того как useEffectEvent станет стабильной частью React, мы рекомендуем никогда не подавлять линтер.

Первым недостатком подавления правила является то, что React больше не будет предупреждать вас, когда вашему эффекту необходимо «реагировать» на новую реактивную зависимость, которую вы ввели в свой код. В предыдущем примере вы добавили URL-адрес в зависимости, потому что React напомнил вам об этом. Вы больше не будете получать такие напоминания о любых будущих изменениях этого эффекта, если отключите линтер. Это приводит к ошибкам.

Вот пример запутанной ошибки, вызванной подавлением линтера. В этом примере предполагается, что функция handleMove считывает текущее значение переменной состояния canMove, чтобы решить, должна ли точка следовать за курсором. Однако canMove всегда имеет значение true внутри handleMove.
Ты понимаешь, почему?
```js
import { useState, useEffect } from 'react';

export default function App() {
  const [position, setPosition] = useState({ x: 0, y: 0 });
  const [canMove, setCanMove] = useState(true);

  function handleMove(e) {
    if (canMove) {
      setPosition({ x: e.clientX, y: e.clientY });
    }
  }

  useEffect(() => {
    window.addEventListener('pointermove', handleMove);
    return () => window.removeEventListener('pointermove', handleMove);
    // eslint-disable-next-line react-hooks/exhaustive-deps
  }, []);

  return (
    <>
      <label>
        <input type="checkbox"
          checked={canMove}
          onChange={e => setCanMove(e.target.checked)}
        />
        The dot is allowed to move
      </label>
      <hr />
      <div style={{
        position: 'absolute',
        backgroundColor: 'pink',
        borderRadius: '50%',
        opacity: 0.6,
        transform: `translate(${position.x}px, ${position.y}px)`,
        pointerEvents: 'none',
        left: -20,
        top: -20,
        width: 40,
        height: 40,
      }} />
    </>
  );
}

```
Проблема этого кода заключается в подавлении линтера зависимостей. Если вы удалите подавление, вы увидите, что этот эффект должен зависеть от функции handleMove. В этом есть смысл: handleMove объявляется внутри тела компонента, что делает его реактивным значением. Каждое реактивное значение должно быть указано как зависимость, иначе со временем оно может устареть!

Автор исходного кода «солгал» React, заявив, что Эффект не зависит ([]) от каких-либо реактивных значений. Вот почему React не выполнил повторную синхронизацию эффекта после изменения canMove (и handleMove вместе с ним). Поскольку React не выполнил повторную синхронизацию эффекта, handleMove, прикрепленный в качестве прослушивателя, представляет собой функцию handleMove, созданную во время первоначального рендеринга. Во время первоначального рендеринга canMove было true, поэтому handleMove из первоначального рендеринга всегда будет видеть это значение.
**Если вы никогда не подавите линтер, вы никогда не увидите проблем с устаревшими значениями.**
With `useEffectEvent`, there is no need to “lie” to the linter, and the code works as you would expect:
```js
import { useState, useEffect } from 'react';
import { experimental_useEffectEvent as useEffectEvent } from 'react';

export default function App() {
  const [position, setPosition] = useState({ x: 0, y: 0 });
  const [canMove, setCanMove] = useState(true);

  const onMove = useEffectEvent(e => {
    if (canMove) {
      setPosition({ x: e.clientX, y: e.clientY });
    }
  });

  useEffect(() => {
    window.addEventListener('pointermove', onMove);
    return () => window.removeEventListener('pointermove', onMove);
  }, []);

  return (
    <>
      <label>
        <input type="checkbox"
          checked={canMove}
          onChange={e => setCanMove(e.target.checked)}
        />
        The dot is allowed to move
      </label>
      <hr />
      <div style={{
        position: 'absolute',
        backgroundColor: 'pink',
        borderRadius: '50%',
        opacity: 0.6,
        transform: `translate(${position.x}px, ${position.y}px)`,
        pointerEvents: 'none',
        left: -20,
        top: -20,
        width: 40,
        height: 40,
      }} />
    </>
  );
}

```
Это не означает, что useEffectEvent всегда является правильным решением. Вам следует применять его только к тем строкам кода, которые не должны быть реактивными. В приведенной выше песочнице вы не хотели, чтобы код эффекта реагировал на canMove. Вот почему имело смысл извлечь событие эффекта.

Прочитайте "[Зависимости эффекта удаления](https://react.dev/learn/removing-effect-dependencies)" для других правильных альтернатив подавлению линтера.
### Limitations of Effect Events

События эффекта очень ограничены в том, как вы можете их использовать:

- **Называйте их только изнутри Effects.**
- **Никогда не передайте их другим компонентам или крючкам.**

События эффекта - это нереактивные "куски" вашего кода эффекта. Они должны быть рядом с Эффектом, используя их.
