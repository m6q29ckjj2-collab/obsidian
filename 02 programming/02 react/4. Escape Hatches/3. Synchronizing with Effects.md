Некоторые компоненты должны синхронизироваться с внешними системами. Например, вы можете управлять компонентом, не относяющимся к React, на основе состояния React, настроить подключение к серверу или отправить журнал аналитики, когда компонент появится на экране. _Эффекты_ позволяют запускать некоторый код после рендеринга, чтобы вы могли синхронизировать свой компонент с некоторой системой за пределами React.
## What are Effects and how are they different from events? [](https://react.dev/learn/synchronizing-with-effects#what-are-effects-and-how-are-they-different-from-events "Link for What are Effects and how are they different from events?")
Прежде чем попасть в Effects, вы должны быть знакомы с двумя типами логики внутри компонентов React:
- **Рендеринг кода** (введенный в [Описание пользовательского интерфейса](https://react.dev/learn/describing-the-ui)) находится на верхнем уровне вашего компонента. Здесь вы берете реквизит и состояние, преобразуете их и возвращаете JSX, который вы хотите увидеть на экране. [Код рендеринга должен быть чистым.](https://react.dev/learn/keeping-components-pure) Как и математическая формула, она должна только _вычислять_ результат, но не делать ничего другого.
    
- **Обработчики событий** (введенные в разделе [Добавление интерактивности](https://react.dev/learn/adding-interactivity)) - это вложенные функции внутри ваших компонентов, которые _делают_ что-то, а не просто вычисляют их. Обработчик событий может обновить поле ввода, отправить HTTP POST-запрос на покупку продукта или перевести пользователя на другой экран. Обработчики событий содержат ["побочные эффекты"](https://en.wikipedia.org/wiki/Side_effect_(computer_science)) (они меняют состояние программы), вызванные определенным действием пользователя (например, нажатием кнопки или вводом).
    

Иногда этого недостаточно. Рассмотрим компонент `ChatRoom`, который должен подключаться к серверу чата всякий раз, когда он виден на экране. Подключение к серверу не является чистым расчетом (это побочный эффект), поэтому оно не может произойти во время рендеринга. Тем не менее, нет ни одного конкретного события, такого как щелчок, который вызывает отображение `ChatRoom`.
**_Эффекты_ позволяют указать побочные эффекты, которые вызваны самим рендерингом, а не конкретным событием.** Отправка сообщения в чате является _событием_, потому что это напрямую вызвано нажатием пользователем определенной кнопки. Тем не менее, настройка подключения к серверу является _эффектом_, потому что это должно происходить независимо от того, какое взаимодействие вызвало появление компонента. Эффекты запускаются в конце [фиксации](https://react.dev/learn/render-and-commit) после обновления экрана. Это хорошее время для синхронизации компонентов React с какой-либо внешней системой (например, сетью или сторонней библиотекой).
Здесь и позже в этом тексте "Эффект" с заглавной буквы относится к определению React, специфичному выше, то есть к побочному эффекту, вызванному рендерингом. Чтобы сослаться на более широкую концепцию программирования, мы скажем "побочный эффект".
## You might not need an Effect [](https://react.dev/learn/synchronizing-with-effects#you-might-not-need-an-effect "Link for You might not need an Effect")
**Не спешите добавлять эффекты в свои компоненты.** Имейте в виду, что эффекты обычно используются для "выхода" из вашего React-кода и синхронизации с какой-либо _внешней_ системой. Сюда входят API браузера, сторонние виджеты, сеть и так далее. Если ваш Эффект корректирует только некоторое состояние на основе другого состояния, [вам может не понадобиться Эффект.](https://react.dev/learn/you-might-not-need-an-effect)
## How to write an Effect [](https://react.dev/learn/synchronizing-with-effects#how-to-write-an-effect "Link for How to write an Effect")
Чтобы написать эффект, выполните следующие три шага:

1. **Объявить эффект.** По умолчанию ваш Эффект будет запускаться после каждой [фиксации](https://react.dev/learn/render-and-commit).
2. **Укажите зависимости эффекта.** Большинство эффектов должны повторяться только _при необходимости_, а не после каждого рендеринга. Например, анимация затухания должна срабатывать только при появлении компонента. Подключение и отключение от чата должно происходить только тогда, когда компонент появляется и исчезает, или когда чат меняется. Вы узнаете, как управлять этим, указав _зависимости._
3. **При необходимости добавьте очистку.** Некоторые эффекты должны указывать, как остановить, отменить или очистить все, что они делали. Например, "подключение" требует "отключения", "подписка" нуждается в "отписке", а "отборваться" нуждается в "отмене" или "игнорировать". Вы узнаете, как это сделать, вернув _функцию очистки_.

Давайте подробно посмотрим на каждый из этих шагов.

Давайте посмотрим, как можно использовать эффект для синхронизации с внешней системой. Рассмотрим компонент React \<VideoPlayer>. Было бы неплохо контролировать, будет ли он воспроизводиться или приостановлен, передав ему свойство isPlaying:
![[Pasted image 20240528000010.png]]
Your custom `VideoPlayer` component renders the built-in browser [`<video>`](https://developer.mozilla.org/en-US/docs/Web/HTML/Element/video) tag:
![[Pasted image 20240528000025.png]]
Однако тег \<video> браузера не имеет свойства isPlaying. Единственный способ контролировать это — вручную вызвать методы play() и Pause() элемента DOM. Вам необходимо синхронизировать значение свойства isPlaying, которое сообщает, должно ли видео воспроизводиться в данный момент, с такими вызовами, как play() и Pause().

Сначала нам нужно получить ссылку на узел DOM \<video>.

У вас может возникнуть соблазн попытаться вызвать play() или паузу() во время рендеринга, но это неправильно:
```js 
import { useState, useRef, useEffect } from 'react';

function VideoPlayer({ src, isPlaying }) {
  const ref = useRef(null);

  if (isPlaying) {
    ref.current.play();  // Calling these while rendering isn't allowed.
  } else {
    ref.current.pause(); // Also, this crashes.
  }

  return <video ref={ref} src={src} loop playsInline />;
}

export default function App() {
  const [isPlaying, setIsPlaying] = useState(false);
  return (
    <>
      <button onClick={() => setIsPlaying(!isPlaying)}>
        {isPlaying ? 'Pause' : 'Play'}
      </button>
      <VideoPlayer
        isPlaying={isPlaying}
        src="https://interactive-examples.mdn.mozilla.net/media/cc0-videos/flower.mp4"
      />
    </>
  );
}

```
Причина, по которой этот код неверен, заключается в том, что он пытается что-то сделать с узлом DOM во время рендеринга. В React рендеринг [должен быть чистым расчетом](https://react.dev/learn/keeping-components-pure) JSX и не должен содержать побочных эффектов, таких как изменение DOM.
Более того, когда VideoPlayer вызывается в первый раз, его DOM еще не существует! Пока нет узла DOM, на котором можно было бы вызвать play() или Pause(), потому что React не знает, какой DOM создавать, пока вы не вернете JSX.
Решение здесь заключается в том, чтобы **обернуть** **побочный эффект с помощью `useEffect`, чтобы выместить его из расчета рендеринга:**
![[Pasted image 20240528000232.png]]
Обматывая обновление DOM в Effect, вы позволяете React сначала обновить экран. Затем ваш эффект запускается.
Когда ваш компонент VideoPlayer выполняет рендеринг (либо в первый раз, либо при повторной рендеринге), произойдет несколько вещей. Сначала React обновит экран, гарантируя, что тег \<video> находится в DOM с правильными реквизитами. Затем React запустит ваш эффект. Наконец, ваш Эффект вызовет play() или паузу() в зависимости от значения isPlaying.
Нажмите кнопку воспроизведения/паузы несколько раз и посмотрите, как видеоплеер остается синхронизированным со значением `isPlaying`:
```js
import { useState, useRef, useEffect } from 'react';

function VideoPlayer({ src, isPlaying }) {
  const ref = useRef(null);

  useEffect(() => {
    if (isPlaying) {
      ref.current.play();
    } else {
      ref.current.pause();
    }
  });

  return <video ref={ref} src={src} loop playsInline />;
}

export default function App() {
  const [isPlaying, setIsPlaying] = useState(false);
  return (
    <>
      <button onClick={() => setIsPlaying(!isPlaying)}>
        {isPlaying ? 'Pause' : 'Play'}
      </button>
      <VideoPlayer
        isPlaying={isPlaying}
        src="https://interactive-examples.mdn.mozilla.net/media/cc0-videos/flower.mp4"
      />
    </>
  );
}
```
В этом примере "внешняя система", которую вы синхронизировали с состоянием React, была медиа-API браузера. Вы можете использовать аналогичный подход для упаковки устаревшего кода, не связанного с React (например, плагины jQuery) в декларативные компоненты React.
Обратите внимание, что на практике управлять видеоплеером гораздо сложнее. Вызов play() может завершиться неудачей, пользователь может воспроизвести или приостановить воспроизведение с помощью встроенных элементов управления браузера и т. д. Этот пример очень упрощен и неполн.
По умолчанию эффекты загонаются после _каждого_ рендеринга. Вот почему такой код будет **создавать бесконечный цикл**
![[Pasted image 20240528000451.png]]
Эффекты зарабукаются в результате рендеринга. Установка состояния _триггеров_ рендеринга. Установка состояния сразу в эффекте похожа на подключение к себе электрической розетки. Эффект запускается, он устанавливает состояние, которое вызывает повторный рендер, который вызывает запуск Эффекта, он снова устанавливает состояние, это вызывает еще один повторный рендер и так далее.

Эффекты обычно должны синхронизировать ваши компоненты с _внешней_ системой. Если нет внешней системы, и вы хотите только настроить какое-то состояние на основе другого, [вам может не понадобиться эффект.](https://react.dev/learn/you-might-not-need-an-effect)
### Step 2: Specify the Effect dependencies [](https://react.dev/learn/synchronizing-with-effects#step-2-specify-the-effect-dependencies "Link for Step 2: Specify the Effect dependencies")
По умолчанию эффекты исполняются после _каждого_ рендеринга. Часто это **не то, чего вы хотите:**

- Иногда это медленно. Синхронизация с внешней системой не всегда происходит мгновенно, поэтому вы можете пропустить это, если это не необходимо. Например, вы не хотите повторно подключаться к серверу чата при каждом нажатии клавиши.
- Иногда это неправильно. Например, вы не хотите запускать анимацию затухания компонента при каждом нажатии клавиши. Анимация должна воспроизводиться только один раз, когда компонент появляется в первый раз.
Чтобы продемонстрировать проблему, вот предыдущий пример с несколькими вызовами console.log и текстовым вводом, который обновляет состояние родительского компонента. Обратите внимание, как ввод приводит к повторному запуску эффекта:
```js
import { useState, useRef, useEffect } from 'react';

function VideoPlayer({ src, isPlaying }) {
  const ref = useRef(null);

  useEffect(() => {
    if (isPlaying) {
      console.log('Calling video.play()');
      ref.current.play();
    } else {
      console.log('Calling video.pause()');
      ref.current.pause();
    }
  });

  return <video ref={ref} src={src} loop playsInline />;
}

export default function App() {
  const [isPlaying, setIsPlaying] = useState(false);
  const [text, setText] = useState('');
  return (
    <>
      <input value={text} onChange={e => setText(e.target.value)} />
      <button onClick={() => setIsPlaying(!isPlaying)}>
        {isPlaying ? 'Pause' : 'Play'}
      </button>
      <VideoPlayer
        isPlaying={isPlaying}
        src="https://interactive-examples.mdn.mozilla.net/media/cc0-videos/flower.mp4"
      />
    </>
  );
}

```
Вы можете сказать React **пропустить ненужный повторный запуск эффекта**, указав массив _зависимостей_ в качестве второго аргумента вызова `useEffect`. Начните с добавления пустого`[]` массив к приведенному выше примеру в строке 14:
![[Pasted image 20240528000723.png]]
Вы должны увидеть ошибку, в которой говорится, что `React Hook useEffect has a missing dependency: 'isPlaying'`:

function VideoPlayer({ src, isPlaying }) {

Проблема в том, что код внутри вашего эффекта зависит от свойства isPlaying, чтобы решить, что делать, но эта зависимость не была явно объявлена. Чтобы устранить эту проблему, добавьте isPlaying в массив зависимостей:
![[Pasted image 20240528000807.png]]Теперь все зависимости объявлены, поэтому ошибок нет. Указание [isPlaying] в качестве массива зависимостей сообщает React, что ему следует пропустить повторный запуск вашего эффекта, если isPlaying такой же, как и во время предыдущего рендеринга. Благодаря этому изменению ввод данных не приводит к повторному запуску эффекта, но нажатие кнопки «Воспроизведение/Пауза» приводит к:
```js
import { useState, useRef, useEffect } from 'react';

function VideoPlayer({ src, isPlaying }) {
  const ref = useRef(null);

  useEffect(() => {
    if (isPlaying) {
      console.log('Calling video.play()');
      ref.current.play();
    } else {
      console.log('Calling video.pause()');
      ref.current.pause();
    }
  }, [isPlaying]);

  return <video ref={ref} src={src} loop playsInline />;
}

export default function App() {
  const [isPlaying, setIsPlaying] = useState(false);
  const [text, setText] = useState('');
  return (
    <>
      <input value={text} onChange={e => setText(e.target.value)} />
      <button onClick={() => setIsPlaying(!isPlaying)}>
        {isPlaying ? 'Pause' : 'Play'}
      </button>
      <VideoPlayer
        isPlaying={isPlaying}
        src="https://interactive-examples.mdn.mozilla.net/media/cc0-videos/flower.mp4"
      />
    </>
  );
}

```
Массив зависимостей может содержать несколько зависимостей. React пропустит повторный запуск эффекта только в том случае, если _все_ указанные вами зависимости имеют точно такие же значения, как и во время предыдущего рендеринга. React сравнивает значения зависимостей с помощью сравнения [`Object.is`](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Object/is). Для получения подробной информации см. ссылку на [`useEffect`](https://react.dev/reference/react/useEffect#reference).
**Обратите внимание, что вы не можете "выбить" свои зависимости.** Вы получите ошибку eslint, если указанные вами зависимости не соответствуют тому, что React ожидает на основе кода внутри вашего Effect. Это поможет поймать многие ошибки в вашем коде. Если вы не хотите, чтобы какой-то код был перезапускан, [_отредактируйте сам код Effect_, чтобы не "нужна" эта зависимость.](https://react.dev/learn/lifecycle-of-reactive-effects#what-to-do-when-you-dont-want-to-re-synchronize)

#### Why was the ref omitted from the dependency array?
Этот эффект использует как ref, так и isPlaying, но только isPlaying объявлен как зависимость:
![[Pasted image 20240528001240.png]]
Это связано с тем, что объект ref имеет стабильную идентичность: React гарантирует, что вы всегда будете получать один и тот же объект из одного и того же вызова useRef при каждом рендеринге. Он никогда не меняется, поэтому сам по себе никогда не приведет к повторному запуску Эффекта. Поэтому не имеет значения, включаете вы его или нет. В том числе тоже нормально:
![[Pasted image 20240528001309.png]]
[Функции `set`](https://react.dev/reference/react/useState#setstate), возвращаемые `useState`, также имеют стабильную идентичность, поэтому вы часто увидите, что они также опущены из зависимостей. Если linter позволяет опустить зависимость без ошибок, это безопасно.
Исключение всегда стабильных зависимостей работает только тогда, когда линтер может «видеть», что объект стабилен. Например, если ref был передан из родительского компонента, вам придется указать его в массиве зависимостей. Однако это хорошо, потому что вы не можете знать, всегда ли родительский компонент передает одну и ту же ссылку или передает одну из нескольких ссылок условно. Таким образом, ваш эффект будет зависеть от того, какая ссылка будет передана.
### Step 3: Add cleanup if needed [](https://react.dev/learn/synchronizing-with-effects#step-3-add-cleanup-if-needed "Link for Step 3: Add cleanup if needed")
Рассмотрим другой пример. Вы пишете компонент ChatRoom, которому необходимо подключиться к серверу чата, когда он появится. Вам предоставляется API createConnection(), который возвращает объект с помощью методов Connect() и Disconnect(). Как сохранить подключение компонента, пока он отображается пользователю?

Подключение к чату после каждого повторного рендера будет медленным, поэтому вы добавляете массив зависимостей:

**Код внутри Effect не использует никаких реквизитов или состояния, поэтому ваш массив зависимостей`[]` (пустой). Это говорит React запускать этот код только тогда, когда компонент "монтирует", то есть появляется на экране в первый раз.**

Давайте попробуем запустить этот код:
```js
import { useEffect } from 'react';
import { createConnection } from './chat.js';

export default function ChatRoom() {
  useEffect(() => {
    const connection = createConnection();
    connection.connect();
  }, []);
  return <h1>Welcome to the chat!</h1>;
}

```
Этот эффект работает только при монтировании, поэтому вы можете ожидать, что « ✅ Connecting...» будет напечатано в консоли один раз. Однако, если вы проверите консоль, « ✅ Подключение...» будет напечатано дважды. Почему это происходит?

Представьте, что компонент ChatRoom является частью более крупного приложения с множеством разных экранов. Пользователь начинает свое путешествие на странице чата. Компонент монтируется и вызывает метод Connection.connect(). Затем представьте, что пользователь переходит на другой экран, например, на страницу настроек. Компонент ChatRoom размонтируется. Наконец, пользователь нажимает кнопку «Назад», и комната чата снова монтируется. При этом будет установлено второе соединение, но первое соединение никогда не будет разрушено! По мере того, как пользователь перемещается по приложению, соединения будут продолжать накапливаться.
Такие ошибки легко пропустить без обширного ручного тестирования. Чтобы помочь вам быстро обнаружить их, в разработке React перемонтирует каждый компонент сразу после первоначального монтирования.

Дважды, когда вы увидите журнал `"✅ Connecting..."`, вы заметите реальную проблему: ваш код не закрывает соединение при отключении компонента.

Чтобы устранить проблему, верните _функцию очистки_ из вашего эффекта:
![[Pasted image 20240528001733.png]]
React будет вызывать вашу функцию очистки каждый раз, прежде чем эффект запустится снова, и в последний раз, когда компонент размонтируется (удаляется). Давайте посмотрим, что произойдет, когда функция очистки будет реализована:
```js
import { useState, useEffect } from 'react';
import { createConnection } from './chat.js';

export default function ChatRoom() {
  useEffect(() => {
    const connection = createConnection();
    connection.connect();
    return () => connection.disconnect();
  }, []);
  return <h1>Welcome to the chat!</h1>;
}

```
Теперь вы получаете три консольных журнала в разработке:

1. `"✅ Connecting..."`
2. `"❌ Disconnected."`
3. `"✅ Connecting..."`

**Это правильное поведение в развитии.** Повторно монтируя ваш компонент, React проверяет, что переход от входа и обратно не нарушит ваш код. Отключение, а затем повторное подключение - это именно то, что должно произойти! Когда вы хорошо реализуете очистку, не должно быть видимой для пользователя разницы между запуском Эффекта один раз и его запуском, очисткой и повторным запуском. Существует дополнительная пара вызовов подключения/отключения, потому что React исследует ваш код на неу ошибок в разработке. Это нормально - не пытайтесь заставить это уйти!
В процессе производства вы увидите надпись « ✅ Connecting...», напечатанную только один раз. Перемонтирование компонентов происходит только в процессе разработки, чтобы помочь вам найти эффекты, требующие очистки. Вы можете отключить строгий режим, чтобы отказаться от поведения разработки, но мы рекомендуем оставить его включенным. Это позволяет вам найти множество ошибок, подобных приведенной выше.
## How to handle the Effect firing twice in development? [](https://react.dev/learn/synchronizing-with-effects#how-to-handle-the-effect-firing-twice-in-development "Link for How to handle the Effect firing twice in development?")
React намеренно перемонтирует ваши компоненты в разработке, чтобы найти ошибки, как в последнем примере. **Правильный вопрос заключается не в том, «как запустить эффект один раз», а в том, «как исправить мой эффект, чтобы он работал после повторного монтажа».**

Обычно ответ заключается в реализации функции очистки. Функция очистки должна остановить или отменить все, что делал эффект. Эмпирическое правило заключается в том, что пользователь не должен иметь возможность различать эффект, работающего один раз (как в производстве), и последовательность _установки → очистки → настройки_ (как вы увидите в разработке).

Большинство эффектов, которые вы напишете, впишутся в один из общих шаблонов ниже.
#### Don’t use refs to prevent Effects from firing [](https://react.dev/learn/synchronizing-with-effects#dont-use-refs-to-prevent-effects-from-firing "Link for Don’t use refs to prevent Effects from firing")
Распространенной ошибкой предотвращения повторного запуска эффектов в процессе разработки является использование ссылки, чтобы предотвратить запуск эффекта более одного раза. Например, вы можете «исправить» указанную выше ошибку с помощью useRef:
![[Pasted image 20240528002043.png]]
Благодаря этому вы увидите сообщение « ✅ Подключение...» только один раз в разработке, но это не исправит ошибку.
Когда пользователь переходит, соединение все еще не закрывается, и когда он возвращается, создается новое соединение. По мере того, как пользователь перемещается по приложению, соединения будут продолжать накапливаться, как и до "исправления".

Чтобы исправить ошибку, недостаточно просто запустить эффект один раз. Эффект должен работать после повторного монтажа, что означает, что соединение должно быть очищено, как в решении выше.

Вы не можете «отменить» уже выполненный сетевой запрос, но ваша функция очистки должна гарантировать, что выборка, которая больше не актуальна, не будет продолжать влиять на ваше приложение. Если идентификатор пользователя меняется с «Алиса» на «Боб», очистка гарантирует, что ответ «Алиса» будет проигнорирован, даже если он придет после «Боба».

Вы можете либо создать такой кэш самостоятельно, либо использовать одну из многочисленных альтернатив ручной выборке в Effects.
#### What are good alternatives to data fetching in Effects?
Запись вызовов `fetch` внутри Effects является [популярным способом извлечения данных](https://www.robinwieruch.de/react-hooks-fetch-data/), особенно в полностью клиентских приложениях. Однако это очень ручной подход, и он имеет значительные и недостатки:
- **Эффекты не работают на сервере.** Это означает, что первоначальный HTML, отрендерируемый сервером, будет включать только состояние загрузки без данных. Клиентский компьютер должен будет загрузить весь JavaScript и отобразить ваше приложение только для того, чтобы обнаружить, что теперь ему нужно загрузить данные. Это не очень эффективно.
- - **Получение непосредственно в Effects позволяет легко создавать "сетевые водопады".** Вы отображаете родительский компонент, он извлекает некоторые данные, отображает дочерние компоненты, а затем они начинают извекать свои данные. Если сеть не очень быстрая, это значительно медленнее, чем параллельное извлечение всех данных.
- **Получение непосредственно в Effects обычно означает, что вы не загружаете и не кэшируете данные.** Например, если компонент размонтируется, а затем снова монтируется, ему придется снова получить данные.
- Это не очень эргономично. При написании вызовов выборки используется довольно много шаблонного кода, который не страдает от ошибок, таких как  [race conditions](https://maxrozen.com/race-conditions-fetching-data-react-with-useeffect).
Этот список недостатков не является специфическим для React. Это относится к извлечению данных при монтирования с помощью любой библиотеки. Как и в маршрутизации, извлечение данных не является тривиальной, поэтому мы рекомендуем следующие подходы:
- **Если вы используете [фреймворк](https://react.dev/learn/start-a-new-react-project#production-grade-react-frameworks), используйте его встроенный механизм выбора данных.**Современные фреймворки React имеют интегрированные механизмы извлечения данных, которые эффективны и не страдают от вышеуказанных подводных камней.
- **В противном случае рассмотрите возможность использования или создания кэша на стороне клиента.** Популярные решения с открытым исходным кодом включают [React Query](https://tanstack.com/query/latest), [useSWR](https://swr.vercel.app/) и [React Router 6.4+.](https://beta.reactrouter.com/en/main/start/overview) Вы также можете создать свое собственное решение, и в этом случае вы будете использовать Effects под капотом, но добавьте логику для дедупликации запросов, кэширования ответов и избежания сетевых водопадов (путем предварительной загрузки данных или поднятия требований к данным на маршруты).

Вы можете продолжить извлечение данных непосредственно в Effects, если ни один из этих подходов вам не подходит.

## Putting it all together [](https://react.dev/learn/synchronizing-with-effects#putting-it-all-together "Link for Putting it all together")
Эта игровая площадка может помочь вам "почувствовать", как эффекты работают на практике.
В этом примере используется setTimeout, чтобы запланировать появление журнала консоли с входным текстом через три секунды после запуска эффекта. Функция очистки отменяет ожидающий тайм-аут. Начните с нажатия кнопки «Монтировать компонент»:
```js
import { useState, useEffect } from 'react';

function Playground() {
  const [text, setText] = useState('a');

  useEffect(() => {
    function onTimeout() {
      console.log('⏰ ' + text);
    }

    console.log('🔵 Schedule "' + text + '" log');
    const timeoutId = setTimeout(onTimeout, 3000);

    return () => {
      console.log('🟡 Cancel "' + text + '" log');
      clearTimeout(timeoutId);
    };
  }, [text]);

  return (
    <>
      <label>
        What to log:{' '}
        <input
          value={text}
          onChange={e => setText(e.target.value)}
        />
      </label>
      <h1>{text}</h1>
    </>
  );
}

export default function App() {
  const [show, setShow] = useState(false);
  return (
    <>
      <button onClick={() => setShow(!show)}>
        {show ? 'Unmount' : 'Mount'} the component
      </button>
      {show && <hr />}
      {show && <Playground />}
    </>
  );
}

```
Сначала вы увидите три журнала: «Запланировать журнал «a», «Отменить журнал «a»» и «Запланировать журнал «a» снова». Через три секунды также появится журнал с надписью a. Как вы узнали ранее, дополнительная пара расписание/отмена связана с тем, что React перемонтирует компонент один раз в разработке, чтобы убедиться, что вы правильно реализовали очистку.

Теперь отредактируйте ввод, указав abc. Если вы сделаете это достаточно быстро, вы сразу увидите журнал «Запланировать «ab», за которым сразу последуют «Отменить журнал «ab»» и «Запланировать журнал «abc». React всегда очищает эффект предыдущего рендеринга перед эффектом следующего рендеринга. Вот почему, даже если вы вводите ввод быстро, одновременно запланировано не более одного тайм-аута. Отредактируйте ввод несколько раз и посмотрите на консоль, чтобы понять, как очищаются эффекты.
Введите что-то во вход, а затем сразу же нажмите "Размонтировать компонент". Обратите внимание, как размонтирование очищает эффект последнего рендеринга. Здесь он очищает последний тайм-аут, прежде чем у него появится шанс на увол.

Наконец, отредактируйте компонент выше и закомментируйте функцию очистки, чтобы таймауты не отменялись. Попробуйте быстро набрать abcde. Что, по вашему мнению, произойдет через три секунды? Будет ли console.log(text) внутри тайм-аута печатать последний текст и создавать пять журналов abcde? Попробуйте проверить свою интуицию!

Через три секунды вы должны увидеть последовательность журналов (a, ab, abc, abcd и abcde), а не пять журналов abcde. Каждый эффект «захватывает» текстовое значение из соответствующего рендеринга.   Не имеет значения, что состояние текста изменилось: Эффект от рендера с текстом = 'ab' всегда будет видеть 'ab'. Другими словами, эффекты каждого рендеринга изолированы друг от друга. Если вам интересно, как это работает, вы можете прочитать о [closures](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Closures).

#### Размонтировать
Наконец, предположим, что пользователь уходит, а компонент ChatRoom отключается. React запускает функцию очистки последнего эффекта. Последний эффект был из третьего рендера. Очистка третьего рендеринга уничтожает соединение createConnection('travel'). Таким образом, приложение отключается от комнаты «путешествия».
#### Поведение только для развития[](https://react.dev/learn/synchronizing-with-effects#development-only-behaviors "Ссылка на поведение только для разработки")
Когда включен [строгий режим](https://react.dev/reference/react/StrictMode), React повторно устанавливает каждый компонент один раз после монтирования (состояние и DOM сохраняются). Это [поможет вам найти эффекты, которые нуждаются в очистке,](https://react.dev/learn/synchronizing-with-effects#step-3-add-cleanup-if-needed) и выявить ошибки, такие как условия гонки на ранней стадии. Кроме того, React будет повторно монтировать Effects всякий раз, когда вы сохраняете файл в разработке. Оба этих поведения являются только для развития.
