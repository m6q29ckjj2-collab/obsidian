Эффекты - это escape hatch из парадигмы React. Они позволяют вам "выйти за пределы" React и синхронизировать ваши компоненты с какой-либо внешней системой, такой как виджет без React, сеть или DOM браузера. Если нет внешней системы (например, если вы хотите обновить состояние компонента при изменении некоторых реквизитов или состояния), вам не нужен эффект. Удаление ненужных эффектов сделает ваш код проще для отслеживания, быстрее запускать и меньше подвержен ошибкам.
## How to remove unnecessary Effects [](https://react.dev/learn/you-might-not-need-an-effect#how-to-remove-unnecessary-effects "Link for How to remove unnecessary Effects")
Есть два распространенных случая, когда вам не нужны эффекты:
- **Вам не нужны эффекты для преобразования данных для рендеринга.** Например, предположем, что вы хотите отфильтровать список перед его отображением. У вас может возникнуть соблазн написать эффект, который обновляет переменную состояния при изменении списка. Однако это неэффективно. Когда вы обновляете состояние, React сначала вызовет функции вашего компонента, чтобы вычислить, что должно быть на экране. Затем React ["закоммитить"](https://react.dev/learn/render-and-commit) эти изменения в DOM, обновив экран. Затем React запустит ваши эффекты. Если ваш эффект _также_ немедленно обновляет состояние, это перезапускает весь процесс с нуля! Чтобы избежать ненужных пропусков рендеринга, преобразуйте все данные на верхнем уровне ваших компонентов. Этот код будет автоматически перезапускаться при изменении вашего реквизита или состояния.
- Вам не нужны эффекты для обработки пользовательских событий. Например, предположим, что вы хотите отправить POST-запрос /api/buy и отобразить уведомление, когда пользователь покупает продукт. В обработчике событий нажатия кнопки «Купить» вы точно знаете, что произошло. К моменту запуска эффекта вы не знаете, что сделал пользователь (например, какую кнопку нажали). Вот почему вы обычно обрабатываете пользовательские события в соответствующих обработчиках событий.
Вам нужны эффекты для [синхронизации](https://react.dev/learn/synchronizing-with-effects#what-are-effects-and-how-are-they-different-from-events) с внешними системами. Например, вы можете написать эффект, который поддерживает синхронизацию виджета jQuery с состоянием React. Вы также можете получить данные с помощью Effects: например, вы можете синхронизировать результаты поиска с текущим поисковым запросом. Имейте в виду, что современные [фреймворки](https://react.dev/learn/start-a-new-react-project#production-grade-react-frameworks) обеспечивают более эффективные встроенные механизмы получения данных, чем запись эффектов непосредственно в ваших компонентах.
Чтобы помочь вам получить правильную интуицию, давайте посмотрим на некоторые распространенные конкретные примеры!

![[Pasted image 20240528130733.png]]
Обычно React сохраняет состояние, когда тот же компонент отображается в одном и том же месте. Передавая userId в качестве ключа компоненту Profile, вы просите React рассматривать два компонента Profile с разными userId как два разных компонента, которые не должны иметь общего состояния. Компонентам, отображающим ProfilePage, не нужно передавать ему ключ: они передают userId как обычный реквизит. Тот факт, что ProfilePage передает его как ключ внутреннему компоненту Profile, является деталью реализации.

Этот компонент List получает список элементов в качестве реквизита и сохраняет выбранный элемент в переменной состояния выбора. Вы хотите сбросить выделение до нуля всякий раз, когда свойство items получает другой массив:

Это тоже не идеально. Каждый раз, когда элементы изменяются, список и его дочерние компоненты сначала будут отображаться с устаревшим значением выбора. Затем React обновит DOM и запустит эффекты. Начните с удаления эффекта. Вместо этого отрегулируйте состояние непосредственно во время рендеринга:
![[Pasted image 20240528131407.png]]
[Storing information from previous renders](https://react.dev/reference/react/useState#storing-information-from-previous-renders) из предыдущих рендеров, подобное этому, может быть трудным для понимания, но это лучше, чем обновление того же состояния в эффекте. В приведенном выше примере setSelection вызывается непосредственно во время рендеринга. React повторно отобразит список сразу после выхода с помощью оператора return. React еще не отрисовал дочерние элементы List и не обновил DOM, поэтому это позволяет дочерним элементам List пропустить рендеринг устаревшего значения выбора.

Когда вы обновляете компонент во время рендеринга, React удаляет возвращенный JSX и немедленно повторяет рендеринг. Чтобы избежать очень медленных каскадных повторов, React позволяет обновлять состояние одного и того же компонента только во время рендеринга. Если вы обновите состояние другого компонента во время рендеринга, вы увидите ошибку. Условие типа items !== prevItems необходимо, чтобы избежать циклов. Вы можете настроить состояние таким образом, но любые другие побочные эффекты (например, изменение DOM или установка тайм-аутов) должны оставаться в обработчиках событий или эффектах, чтобы компоненты [keep components pure.](https://react.dev/learn/keeping-components-pure).

**Хотя этот шаблон более эффективен, чем эффект, большинство компонентов также не должны нуждаться в нем.** Независимо от того, как вы это делаете, настройка состояния на основе реквизита или другого состояния затрудняет понимание и отладку потока данных. Всегда проверяйте, можете ли вы [сбросить все состояние с помощью ключа](https://react.dev/learn/you-might-not-need-an-effect#resetting-all-state-when-a-prop-changes) или [вычислить все во время рендеринга](https://react.dev/learn/you-might-not-need-an-effect#updating-state-based-on-props-or-state). Например, вместо хранения (и сброса) выбранного _элемента_ вы можете сохранить _идентификатор_ выбранного _элемента:_
![[Pasted image 20240528132503.png]]
Теперь вообще нет необходимости «корректировать» state. Если элемент с выбранным идентификатором присутствует в списке, он остается выбранным. Если это не так, выбор, рассчитанный во время рендеринга, будет нулевым, поскольку соответствующий элемент не найден. Это поведение отличается, но, возможно, лучше, потому что большинство изменений элементов сохраняют выбор.

### Chains of computations [](https://react.dev/learn/you-might-not-need-an-effect#chains-of-computations "Link for Chains of computations")
Иногда вы можете почувствовать соблазн связать эффекты, которые каждый из них регулирует часть состояния на основе другого состояния:
```js
function Game() {  
const [card, setCard] = useState(null);  
const [goldCardCount, setGoldCardCount] = useState(0);  
const [round, setRound] = useState(1);  
const [isGameOver, setIsGameOver] = useState(false);  

// 🔴 Avoid: Chains of Effects that adjust the state solely to trigger each other  

useEffect(() => {  
if (card !== null && card.gold) {  
		setGoldCardCount(c => c + 1);  
	}  
}, [card]);  

  

useEffect(() => {  
if (goldCardCount > 3) {  
		setRound(r => r + 1)  
		setGoldCardCount(0);  
	}  
}, [goldCardCount]);  

  

useEffect(() => {  
if (round > 5) {  
	setIsGameOver(true);  
	}  
}, [round]);  

  

useEffect(() => {  
	alert('Good game!');  
}, [isGameOver]);  

  

function handlePlaceCard(nextCard) {  
	if (isGameOver) {  
		throw Error('Game already ended.');  
		} else {  
		setCard(nextCard);  
		}  
}  
// ...
```
Есть две проблемы с этим кодом.
Одна из проблем заключается в том, что это очень неэффективно: компоненту (и его дочерним элементам) приходится выполнять повторный рендеринг между каждым вызовом набора в цепочке. В примере выше в худшем случае (setCard → render → setGoldCardCount → render → setRound → render → setIsGameOver → render) происходит три ненужных повторных рендеринга дерева ниже.

Даже если бы это не было медленно, по мере развития вашего кода вы будете сталкиваться со случаями, когда написанная вами «цепочка» не соответствует новым требованиям. Представьте, что вы добавляете способ пошагового просмотра истории игровых ходов. Вы можете сделать это, обновив каждую переменную состояния до значения из прошлого. Однако установка состояния карты на значение из прошлого снова вызовет цепочку эффектов и изменит отображаемые данные. Такой код часто бывает жестким и хрупким.

В этом случае лучше рассчитать, что можно во время рендеринга, и настроить состояние в обработчике событий:

Если вам нужно использовать следующее значение для вычислений, определите его вручную, например const nextRound = round + 1.
В некоторых случаях вы _не можете_ рассчитать следующее состояние непосредственно в обработчике событий. Тогда цепочка эффектов уместна, потому что вы синхронизируетесь с сетью.

Управление условиями гонки - это не единственная трудность с внедрением сбора данных. Вы также можете подумать о кэшировании ответов (чтобы пользователь мог нажать кнопку «Назад» и мгновенно увидеть предыдущий экран), о том, как получить данные на сервере (чтобы первоначальный HTML, отренеденный сервером, содержал изванный контент вместо спиннера), и о том, как избежать сетевых водопадов (чтобы ребенок мог получить данные, не дожидаясь каждого родителя).

**Эти проблемы применимы к любой библиотеке пользовательского интерфейса, а не только к React. Их решение не тривиально, поэтому современные [фреймворки](https://react.dev/learn/start-a-new-react-project#production-grade-react-frameworks) обеспечивают более эффективные встроенные механизмы получения данных, чем выбор данных в Effects.**

Скорее всего, вы также захотите добавить некоторую логику для обработки ошибок и отслеживать, загружается ли контент.
