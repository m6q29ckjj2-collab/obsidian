Когда вы пишете Effect, linter проверит, что вы включили все реактивные значения (например, реквизиты и состояние), которые Effect читает в список зависимостей вашего эффекта. Это гарантирует, что ваш эффект остается синхронизированным с последним реквизитом и состоянием вашего компонента. Ненужные зависимости могут привести к тому, что ваш эффект будет работать слишком часто или даже создать бесконечный цикл. Следуйте этому руководству, чтобы просмотреть и удалить ненужные зависимости от ваших эффектов.
## Dependencies should match the code [](https://react.dev/learn/removing-effect-dependencies#dependencies-should-match-the-code "Link for Dependencies should match the code")
Когда вы пишете эффект, вы сначала указываете, как [начать и остановить](https://react.dev/learn/lifecycle-of-reactive-effects#the-lifecycle-of-an-effect) все, что вы хотите, чтобы ваш эффект делал:

Затем, если вы оставите зависимости эффекта пустыми (`[]` ), линтер предложит правильные зависимости:

import { createConnection } from './chat.js';

const serverUrl = 'https://localhost:1234';

### To remove a dependency, prove that it’s not a dependency [](https://react.dev/learn/removing-effect-dependencies#to-remove-a-dependency-prove-that-its-not-a-dependency "Link for To remove a dependency, prove that it’s not a dependency")
Обратите внимание, что вы не можете "выбрать" зависимости вашего эффекта. Каждое реактивное значение, используемое кодом вашего эффекта, должно быть объявлено в вашем списке зависимостей. Список зависимостей определяется окружающим кодом:
![[Pasted image 20240528222858.png]]
[Reactive values](https://react.dev/learn/lifecycle-of-reactive-effects#all-variables-declared-in-the-component-body-are-reactive) включают реквизиты, а также все переменные и функции, объявленные непосредственно внутри вашего компонента. Поскольку roomId является реактивным значением, вы не можете удалить его из списка зависимостей. Линтер этого не допустил:
![[Pasted image 20240528222955.png]]
И линтер был бы прав! Поскольку идентификатор roomId может меняться со временем, это приведет к ошибке в вашем коде.

Чтобы удалить зависимость, «докажите» линтеру, что это не обязательно должна быть зависимость. Например, вы можете переместить roomId из вашего компонента, чтобы доказать, что он не реактивен и не изменится при повторной визуализации:
![[Pasted image 20240528223037.png]]
Теперь, когда roomId не является реактивным значением (и не может измениться при повторном рендеринге), он не обязательно должен быть зависимостью:
```js
import { useState, useEffect } from 'react';
import { createConnection } from './chat.js';

const serverUrl = 'https://localhost:1234';
const roomId = 'music';

export default function ChatRoom() {
  useEffect(() => {
    const connection = createConnection(serverUrl, roomId);
    connection.connect();
    return () => connection.disconnect();
  }, []);
  return <h1>Welcome to the {roomId} room!</h1>;
}

```
Вот почему теперь вы можете указать [пустой](https://react.dev/learn/lifecycle-of-reactive-effects#what-an-effect-with-empty-dependencies-means) [(`[]` ) список зависимостей.](https://react.dev/learn/lifecycle-of-reactive-effects#what-an-effect-with-empty-dependencies-means) Ваш эффект _действительно больше не_ зависит от какого-либо реактивного значения, поэтому ему _действительно не_ нужно повторно запускаться при изменении какого-либо реквизита или состояния компонента.
### To change the dependencies, change the code [](https://react.dev/learn/removing-effect-dependencies#to-change-the-dependencies-change-the-code "Link for To change the dependencies, change the code")
Возможно, вы заметили шаблон в своем рабочем процессе:
1. Во-первых, вы **меняете код** вашего эффекта или то, как объявляются ваши реактивные значения.
2. Затем вы следуете за линтером и настраиваете зависимости в **соответствии с измененным кодом.**
3. Если вас не устраивает список зависимостей, вы **возвращаетесь к первому шагу** (и снова меняете код).
Последняя часть важна. **Если вы хотите изменить зависимости, сначала измените окружающий код.** Вы можете думать о списке зависимостей как [о списке всех реактивных значений, используемых кодом вашего Effect.](https://react.dev/learn/lifecycle-of-reactive-effects#react-verifies-that-you-specified-every-reactive-value-as-a-dependency) Вы не _выбираете_, что включить в этот список. В списке _описывается_ ваш код. Чтобы изменить список зависимостей, измените код.

Это может показаться решением уравнения. Вы можете начать с цели (например, чтобы удалить зависимость), и вам нужно "найти" код, соответствующий этой цели. Не все находят решение уравнений забавным, и то же самое можно сказать о написании эффектов! К счастью, ниже есть список распространенных рецептов, которые вы можете попробовать.

Если у вас есть существующая кодовая база, у вас могут быть некоторые эффекты, которые подавляют линтер, например:

**Когда зависимости не соответствуют коду, существует очень высокий риск введения ошибок.** Подавляя линтер, вы "лжете" React о значениях, от которых зависит ваш эффект.

#### Why is suppressing the dependency linter so dangerous?
Подавление линтера приводит к очень неинтуитивным ошибкам, которые трудно найти и исправить. Вот один пример:
```js
import { useState, useEffect } from 'react';

export default function Timer() {
  const [count, setCount] = useState(0);
  const [increment, setIncrement] = useState(1);

  function onTick() {
	setCount(count + increment);
  }

  useEffect(() => {
    const id = setInterval(onTick, 1000);
    return () => clearInterval(id);
    // eslint-disable-next-line react-hooks/exhaustive-deps
  }, []);

  return (
    <>
      <h1>
        Counter: {count}
        <button onClick={() => setCount(0)}>Reset</button>
      </h1>
      <hr />
      <p>
        Every second, increment by:
        <button disabled={increment === 0} onClick={() => {
          setIncrement(i => i - 1);
        }}>–</button>
        <b>{increment}</b>
        <button onClick={() => {
          setIncrement(i => i + 1);
        }}>+</button>
      </p>
    </>
  );
}

```
Скажем, вы хотели запустить эффект "только mount". Вы прочитали, что [пусто (`[]` ) зависимости](https://react.dev/learn/lifecycle-of-reactive-effects#what-an-effect-with-empty-dependencies-means) делают это, поэтому вы решили игнорировать линтер и принудительно указали`[]` как зависимости.
Предполагалось, что этот счетчик будет увеличиваться каждую секунду на величину, настраиваемую с помощью двух кнопок. Однако, поскольку вы «солгали» React, что этот Эффект ни от чего не зависит, React всегда продолжает использовать функцию onTick из первоначального рендеринга. [During that render](https://react.dev/learn/state-as-a-snapshot#rendering-takes-a-snapshot-in-time) этого рендеринга счетчик был равен 0, а приращение — 1. Вот почему onTick из этого рендеринга всегда вызывает setCount(0 + 1) каждую секунду, и вы всегда видите 1. Подобные ошибки труднее исправить, когда они распространяются на несколько компоненты.
Всегда есть лучшее решение, чем игнорировать линтер! Чтобы исправить этот код, вам нужно добавить onTick в список зависимостей. (Чтобы гарантировать, что интервал устанавливается только один раз, с, [make `onTick` an Effect Event.](https://react.dev/learn/separating-events-from-effects#reading-latest-props-and-state-with-effect-events).)
**Мы рекомендуем рассматривать ошибку lint зависимости как ошибку компиляции. Если вы не подавите его, вы никогда не увидите таких ошибок.** Остальная часть этой страницы документирует альтернативы для этого и других случаев.

## Removing unnecessary dependencies [](https://react.dev/learn/removing-effect-dependencies#removing-unnecessary-dependencies "Link for Removing unnecessary dependencies")
Каждый раз, когда вы настраиваете зависимости эффекта, чтобы отразить код, смотрите на список зависимостей. Имеет ли смысл повторно запускать Эффект при изменении любой из этих зависимостей? Иногда ответ "нет":

- Возможно, вы захотите повторно выполнить _разные части_ своего эффекта в разных условиях.
- Возможно, вы захотите прочитать только _последнее значение_ некоторой зависимости, а не "реагирование" на его изменения.
- Зависимость может слишком часто меняться _непреднамеренно,_ потому что это объект или функция.

Чтобы найти правильное решение, вам нужно будет ответить на несколько вопросов о вашем эффекте. Давайте пройдемся по ним.
### Should this code move to an event handler? [](https://react.dev/learn/removing-effect-dependencies#should-this-code-move-to-an-event-handler "Link for Should this code move to an event handler?")
Первое, о чем вы должны подумать, это о том, должен ли этот код вообще быть эффектом.
Представьте себе форму. При отправке вы устанавливаете для представленной переменной состояния значение true. Вам нужно отправить POST-запрос и показать уведомление. Вы поместили эту логику в эффект, который «реагирует» на истинность отправленного сообщения:
![[Pasted image 20240528223728.png]]
Позже вы захотите стилизовать уведомление в соответствии с текущей темой, чтобы прочитать текущую тему. Поскольку тема объявлена ​​в теле компонента, это реактивное значение, поэтому вы добавляете ее как зависимость:
![[Pasted image 20240528223839.png]]
Сделав это, вы внесли ошибку. Представьте, что вы сначала отправляете форму, а затем переключаетесь между темной и светлой темами. Тема изменится, Эффект запустится повторно, и снова отобразится то же уведомление!

**Проблема здесь в том, что это не должно быть эффектом в первую очередь.** Вы хотите отправить этот POST-запрос и показать уведомление в ответ на _отправку формы,_ которая является конкретным взаимодействием. Чтобы запустить какой-то код в ответ на конкретное взаимодействие, поместите эту логику непосредственно в соответствующий обработчик событий:
![[Pasted image 20240528223916.png]]
Теперь, когда код находится в обработчике событий, он не реагирует, поэтому он будет запускаться только тогда, когда пользователь отправит форму. Узнайте больше о [выборе между обработчиками событий и эффектами](https://react.dev/learn/separating-events-from-effects#reactive-values-and-reactive-logic) и [о том, как удалить ненужные эффекты.](https://react.dev/learn/you-might-not-need-an-effect)
### Is your Effect doing several unrelated things? [](https://react.dev/learn/removing-effect-dependencies#is-your-effect-doing-several-unrelated-things "Link for Is your Effect doing several unrelated things?")
Следующий вопрос, который вы должны задать себе, заключается в том, делает ли ваш эффект несколько несвязанных вещей.
Представьте, что вы создаете форму доставки, в которой пользователю необходимо выбрать свой город и район. Вы получаете список городов с сервера в соответствии с выбранной страной, чтобы отобразить их в раскрывающемся списке:
Это хороший пример получения данных в эффекте. Вы синхронизируете состояние города с сетью в соответствии с реквизитом страны. Вы не можете сделать это в обработчике событий, потому что вам нужно получать данные, как только отображается ShippingForm, и всякий раз, когда меняется страна (независимо от того, какое взаимодействие вызывает это).

Теперь предположим, что вы добавляете второе поле выбора для городских районов, которое должно выбрать районы для текущего выбранного города. Вы можете начать с добавления второго вызова выборки для списка областей внутри того же эффекта:
![[Pasted image 20240528224127.png]]
Однако, поскольку Эффект теперь использует переменную состояния города, вам пришлось добавить город в список зависимостей. Это, в свою очередь, создало проблему: когда пользователь выбирает другой город, эффект будет повторно запущен и вызовет fetchCities(country). В результате вам придется многократно обновлять список городов без необходимости.
**Проблема с этим кодом заключается в том, что вы синхронизируете две разные несвязанные вещи:**
1. Вы хотите синхронизировать состояние городов с сетью на основе реквизита страны.
2. Вы хотите синхронизировать состояние районов с сетью на основе состояния города
Разделите логику на два эффекта, каждый из которых реагирует на реквизит, с которым ему нужно синхронизироваться:
```js
function ShippingForm({ country }) {  
const [cities, setCities] = useState(null);  
useEffect(() => {  
	let ignore = false;  
	fetch(`/api/cities?country=${country}`)  
		.then(response => response.json())  
		.then(json => {  
	if (!ignore) {  
		setCities(json);  
	}  
	});  
	return () => {  
		ignore = true;  
	};  
}, [country]); // ✅ All dependencies declared  

const [city, setCity] = useState(null);  
const [areas, setAreas] = useState(null);  
useEffect(() => {  
	if (city) {  
	let ignore = false;  
	fetch(`/api/areas?city=${city}`)  
		.then(response => response.json())  
		.then(json => {  
	if (!ignore) {  
		setAreas(json);  
	}  
	});  
	return () => {  
		ignore = true;  
	};  
	}  
}, [city]); // ✅ All dependencies declared
```
Теперь первый эффект повторяется только при изменении `country`, в то время как второй эффект повторяется при изменении `city`. Вы разделили их по цели: две разные вещи синхронизируются двумя отдельными эффектами. Два отдельных эффекта имеют два отдельных списка зависимостей, поэтому они не будут инициировать друг друга непреднамеренно.

Окончательный код длиннее оригинала, но разделение этих эффектов все еще верно. [Каждый эффект должен представлять собой независимый процесс синхронизации.](https://react.dev/learn/lifecycle-of-reactive-effects#each-effect-represents-a-separate-synchronization-process) В этом примере удаление одного эффекта не нарушает логику другого эффекта. Это означает, что они _синхронизируют разные вещи,_ и хорошо разделить их. Если вы беспокоитесь о дублирования, вы можете улучшить этот код, [извлекая повторяющуюся логику в пользовательский хук.](https://react.dev/learn/reusing-logic-with-custom-hooks#when-to-use-custom-hooks)
### Are you reading some state to calculate the next state?
Этот эффект обновляет переменную состояния сообщений вновь созданным массивом каждый раз, когда приходит новое сообщение:
![[Pasted image 20240528224938.png]]
Он использует переменную messages для создания нового массива, начиная со всех существующих сообщений, и добавляет новое сообщение в конец. Однако, поскольку сообщения — это реактивное значение, считываемое Эффектом, оно должно быть зависимостью:
![[Pasted image 20240528225153.png]]
А превращение сообщений в зависимость создает проблему.
Каждый раз, когда вы получаете сообщение, setMessages() вызывает повторную отрисовку компонента с использованием нового массива сообщений, который включает полученное сообщение. Однако, поскольку этот Эффект теперь зависит от сообщений, это также приведет к повторной синхронизации Эффекта. Таким образом, каждое новое сообщение приведет к переподключению чата. Пользователю это бы не понравилось!

Чтобы решить эту проблему, не читайте сообщения внутри Эффекта. Вместо этого передайте функцию обновления в setMessages:
![[Pasted image 20240528225445.png]]
Обратите внимание, что ваш Эффект теперь вообще не читает переменную messages. Вам нужно только передать функцию обновления, например msgs => [...msgs, полученное сообщение]. React помещает вашу функцию обновления в очередь и передает ей аргумент msgs во время следующего рендеринга. Вот почему самому Эффекту больше не нужно зависеть от сообщений. В результате этого исправления получение сообщения в чате больше не приводит к повторному подключению чата.

Чтобы решить эту проблему, вам нужно извлечь логику, которая не должна реагировать на эффект. Вы не хотите, чтобы этот эффект "реагировал" на изменения в `isMuted`. [Переместите эту нереактивную часть логики в событие эффекта:](https://react.dev/learn/separating-events-from-effects#declaring-an-effect-event)
![[Pasted image 20240528225922.png]]
События эффекта позволяют разделить эффект на реактивные части (которые должны «реагировать» на реактивные значения, такие как roomId и их изменения) и нереактивные части (которые считывают только свои последние значения, например, onMessage читает isMuted). Теперь, когда вы читаете isMuted внутри события Effect, оно не обязательно должно быть зависимостью вашего Effect. В результате чат не будет повторно подключаться, когда вы включаете и выключаете настройку «Без звука», решая исходную проблему!

#### Separating reactive and non-reactive code
В этом примере вы хотите регистрировать посещение каждый раз, когда меняется идентификатор комнаты. Вы хотите включать текущий NotificationCount в каждый журнал, но не хотите, чтобы изменение NotificationCount вызывало событие журнала.
Решение снова заключается в разделении нереактивного кода на событие Effect:

Вы хотите, чтобы ваша логика реагировала на roomId, поэтому вы читаете roomId внутри вашего эффекта. Однако вы не хотите, чтобы изменение NotificationCount регистрировало дополнительное посещение, поэтому вы читаете NotificationCount внутри Effect Event. ### Does some reactive value change unintentionally?
Иногда вам действительно хочется, чтобы ваш Эффект «реагировал» на определенное значение, но это значение меняется чаще, чем вам хотелось бы, и может не отражать каких-либо реальных изменений с точки зрения пользователя. Например, предположим, что вы создаете объект параметров в теле вашего компонента, а затем читаете этот объект изнутри вашего эффекта:
![[Pasted image 20240528230653.png]]
Этот объект объявлен в теле компонента, поэтому это [реактивное](https://react.dev/learn/lifecycle-of-reactive-effects#effects-react-to-reactive-values) [значение.](https://react.dev/learn/lifecycle-of-reactive-effects#effects-react-to-reactive-values) Когда вы читаете такое реактивное значение внутри эффекта, вы объявляете его зависимостью. Это гарантирует, что ваш эффект "реагирует" на его изменения:
![[Pasted image 20240528230809.png]]
Важно объявить это как зависимость! Это гарантирует, например, что в случае изменения идентификатора комнаты ваш Эффект повторно подключится к чату с новыми параметрами. Однако в приведенном выше коде также есть проблема. Чтобы увидеть это, попробуйте ввести ввод в песочнице ниже и посмотрите, что произойдет в консоли:
```js
import { useState, useEffect } from 'react';
import { createConnection } from './chat.js';

const serverUrl = 'https://localhost:1234';

function ChatRoom({ roomId }) {
  const [message, setMessage] = useState('');

  // Temporarily disable the linter to demonstrate the problem
  // eslint-disable-next-line react-hooks/exhaustive-deps
  const options = {
    serverUrl: serverUrl,
    roomId: roomId
  };

  useEffect(() => {
    const connection = createConnection(options);
    connection.connect();
    return () => connection.disconnect();
  }, [options]);

  return (
    <>
      <h1>Welcome to the {roomId} room!</h1>
      <input value={message} onChange={e => setMessage(e.target.value)} />
    </>
  );
}

export default function App() {
  const [roomId, setRoomId] = useState('general');
  return (
    <>
      <label>
        Choose the chat room:{' '}
        <select
          value={roomId}
          onChange={e => setRoomId(e.target.value)}
        >
          <option value="general">general</option>
          <option value="travel">travel</option>
          <option value="music">music</option>
        </select>
      </label>
      <hr />
      <ChatRoom roomId={roomId} />
    </>
  );
}

```
В песочнице выше ввод обновляет только переменную состояния `message`. С точки зрения пользователя, это не должно влиять на подключение в чате. Однако каждый раз, когда вы обновляете `message`, ваш компонент повторно рендерует. Когда ваш компонент повторно рендерится, код внутри него снова выполняется с нуля.
Новый объект параметров создается с нуля при каждом повторном рендеринге компонента ChatRoom. React видит, что объект параметров отличается от объекта параметров, созданного во время последнего рендеринга. Вот почему он повторно синхронизирует ваш Эффект (который зависит от параметров), и чат повторно подключается по мере того, как вы вводите текст.
**Эта проблема затрагивает только объекты и функции. В JavaScript каждый вновь созданный объект и функция считается отличным от всех остальных. Неважно, что содержимое внутри них может быть одинаковым!**
![[Pasted image 20240528231517.png]]
**Зависимости объектов и функций могут заставить ваш эффект повторно синхронизироваться чаще, чем вам нужно.**
Вот почему, когда это возможно, вы должны стараться избегать объектов и функций в качестве зависимостей вашего эффекта. Вместо этого попробуйте переместить их за пределы компонента, внутри Эффекта или извлечь из них примитивные значения.

Таким образом вы доказываете линтеру, что он не реактивный. Он не может измениться в результате повторного рендеринга, поэтому не обязательно должен быть зависимостью. Теперь повторный рендеринг ChatRoom не приведет к повторной синхронизации вашего Эффекта.

#### Move dynamic objects and functions inside your Effect [](https://react.dev/learn/removing-effect-dependencies#move-dynamic-objects-and-functions-inside-your-effect "Link for Move dynamic objects and functions inside your Effect")
Eсли ваш объект зависит от некоторого реактивного значения, которое может измениться в результате повторного рендеринга, например, реквизит `roomId`, вы не можете вытащить его _за пределы_ компонента. Тем не менее, вы можете переместить его создание _внутри_ кода вашего эффекта:
![[Pasted image 20240528231840.png]]
Теперь, когда параметры объявлены внутри вашего эффекта, они больше не являются зависимостью вашего эффекта. Вместо этого единственное реактивное значение, используемое вашим эффектом, — это roomId. Поскольку roomId не является объектом или функцией, вы можете быть уверены, что он не будет непреднамеренно отличаться. В JavaScript числа и строки сравниваются по их содержимому:
![[Pasted image 20240528231937.png]]
Благодаря этому исправлению чат больше не подключается, если вы редактируете входные данные:
```js
import { useState, useEffect } from 'react';
import { createConnection } from './chat.js';

const serverUrl = 'https://localhost:1234';

function ChatRoom({ roomId }) {
  const [message, setMessage] = useState('');

  useEffect(() => {
    const options = {
      serverUrl: serverUrl,
      roomId: roomId
    };
    const connection = createConnection(options);
    connection.connect();

  }, [roomId]);

  return (
    <>
      <h1>Welcome to the {roomId} room!</h1>
      <input value={message} onChange={e => setMessage(e.target.value)} />
    </>
  );
}

export default function App() {
  const [roomId, setRoomId] = useState('general');
  return (
    <>
      <label>
        Choose the chat room:{' '}
        <select
          value={roomId}
          onChange={e => setRoomId(e.target.value)}
        >
          <option value="general">general</option>
          <option value="travel">travel</option>
          <option value="music">music</option>
        </select>
      </label>
      <hr />
      <ChatRoom roomId={roomId} />
    </>
  );
}

```
Тем не менее, он повторно подключается, когда вы меняете раскрываюся список `roomId`, как и следовало ожидать.
