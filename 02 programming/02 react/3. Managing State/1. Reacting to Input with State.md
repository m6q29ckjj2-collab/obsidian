React предоставляет декларативный способ манипулирования пользовательским интерфейсом. Вместо того, чтобы напрямую манипулировать отдельными частями пользовательского интерфейса, вы описываете различные состояния, в которых может быть ваш компонент, и переключаетесь между ними в ответ на ввод пользователя. Это похоже на то, как дизайнеры думают об пользовательском интерфейсе.
## How declarative UI compares to imperative [](https://react.dev/learn/reacting-to-input-with-state#how-declarative-ui-compares-to-imperative "Link for How declarative UI compares to imperative")
Когда вы разрабатываете взаимодействия пользовательского интерфейса, вы, вероятно, думаете о том, как пользовательский интерфейс _меняется_ в ответ на действия пользователя. Рассмотрим форму, которая позволяет пользователю отправить ответ:
- Когда вы вводите что-то в форму, кнопка "Отправить" **становится включенной.**
- - Когда вы нажимаете "Отправить", как форма, так и кнопка **отключаются,** и **появляется** спиннер**.**
- - Если сетевой запрос увенчается успехом, форма **будет скрыта,** и **появится** сообщение "Спасибо".
- - Если сетевой запрос завершается сбоем, **появляется** сообщение об ошибке**,** и форма снова **включается**.
В **императивном программировании** вышеизложенное напрямую соответствует тому, как вы реализуете взаимодействие. Вы должны написать точные инструкции для манипулирования пользовательским интерфейсом в зависимости от того, что только что произошло. Вот еще один способ подумать об этом: представьте, что вы едете рядом с кем-то в машине и говорите ему по очереди, куда идти.
![[Pasted image 20240526210703.png]]
Они не знают, куда вы хотите пойти, они просто следуют вашим командам. (И если вы ошибаетесь в направлениях, вы окадетесь не в том месте!) Это называется императивом, потому что вы должны "командовать" каждым элементом, от спиннера до кнопки, сообщая компьютеру_, как_ обновить пользовательский интерфейс.
В этом примере императивного программирования пользовательского интерфейса форма строится _без_ React. Он использует только браузер [DOM](https://developer.mozilla.org/en-US/docs/Web/API/Document_Object_Model):
```js
async function handleFormSubmit(e) {
  e.preventDefault();
  disable(textarea);
  disable(button);
  show(loadingMessage);
  hide(errorMessage);
  try {
    await submitForm(textarea.value);
    show(successMessage);
    hide(form);
  } catch (err) {
    show(errorMessage);
    errorMessage.textContent = err.message;
  } finally {
    hide(loadingMessage);
    enable(textarea);
    enable(button);
  }
}

function handleTextareaChange() {
  if (textarea.value.length === 0) {
    disable(button);
  } else {
    enable(button);
  }
}

function hide(el) {
  el.style.display = 'none';
}

function show(el) {
  el.style.display = '';
}

function enable(el) {
  el.disabled = false;
}

function disable(el) {
  el.disabled = true;
}

function submitForm(answer) {
  // Pretend it's hitting the network.
  return new Promise((resolve, reject) => {
    setTimeout(() => {
      if (answer.toLowerCase() === 'istanbul') {
        resolve();
      } else {
        reject(new Error('Good guess but a wrong answer. Try again!'));
      }
    }, 1500);
  });
}

let form = document.getElementById('form');
let textarea = document.getElementById('textarea');
let button = document.getElementById('button');
let loadingMessage = document.getElementById('loading');
let errorMessage = document.getElementById('error');
let successMessage = document.getElementById('success');
form.onsubmit = handleFormSubmit;
textarea.oninput = handleTextareaChange;

```
Манипулирование пользовательским интерфейсом в обязательном порядке работает достаточно хорошо для отдельных примеров, но в более сложных системах управление им становится экспоненциально сложнее. Представьте, что вы обновляете страницу, полную различных форм, таких как эта. Добавление нового элемента пользовательского интерфейса или нового взаимодействия потребует тщательной проверки всего существующего кода, чтобы убедиться, что вы не ввели ошибку (например, забыли что-то показать или скрыть).
React был создан для решения этой проблемы.
В React вы не манипулируете пользовательский интерфейс напрямую, что означает, что вы не включаете, не отключаете, не показываете и не скрываете компоненты напрямую. Вместо этого вы **объявляете то, что хотите показать,** и React выясняет, как обновить пользовательский интерфейс. Подумайте о том, чтобы сесть в такси и сказать водителю, куда вы хотите поехать, вместо того, чтобы говорить ему, куда именно повернуть. Это работа водителя, чтобы доставить вас туда, и они могут даже знать некоторые ярлыки, которые вы не рассматривали!

## Thinking about UI declaratively [](https://react.dev/learn/reacting-to-input-with-state#thinking-about-ui-declaratively "Link for Thinking about UI declaratively")
Вы видели, как внедрить форму более важно выше. Чтобы лучше понять, как думать в React, вы пройдете через повторную реализацию этого пользовательского интерфейса в React ниже:
1. **Определите** различные визуальные состояния вашего компонента
2. **Определите**, что вызывает эти изменения состояния
3. **Представьте** состояние в памяти с помощью`useState`
4. **Удалить** любые несущественные переменные состояния
5. **Подключите** обработчики событий, чтобы установить состояния
### Шаг 1: Определите различные визуальные состояния вашего компонента
В информатике вы можете услышать о том, что ["машина](https://en.wikipedia.org/wiki/Finite-state_machine) [состояния"](https://en.wikipedia.org/wiki/Finite-state_machine) находится в одном из нескольких "состояний". Если вы работаете с дизайнером, вы, возможно, видели макеты для разных "визуальных состояний". React находится на пересечении дизайна и информатики, поэтому обе эти идеи являются источниками вдохновения.

Во-первых, вам нужно визуализировать все различные "состояния" пользовательского интерфейса, которые может увидеть пользователь:

- **Пусто**: В форме отключена кнопка "Отправить".
- **Ввод**: В форме включена кнопка "Отправить".
- **Отправка**: Форма полностью отключена. Спиннер показан.
- **Успех**: сообщение "Спасибо" отображается вместо формы.
- **Ошибка**: То же, что и состояние набора текста, но с дополнительным сообщением об ошибке.

Как и дизайнер, вы захотите "макет" или создать "маки" для различных состояний, прежде чем добавлять логику. Например, вот макет только для визуальной части формы. Этот макет контролируется реквизитом, называемым `status` со значением по умолчанию `'empty'`":
```js
export default function Form({
  status = 'empty'
}) {
  if (status === 'success') {
    return <h1>That's right!</h1>
  }
  return (
    <>
      <h2>City quiz</h2>
      <p>
        In which city is there a billboard that turns air into drinkable water?
      </p>
      <form>
        <textarea />
        <br />
        <button>
          Submit
        </button>
      </form>
    </>
  )
}

```
Вы можете назвать эту опору как угодно, имя не важно. Попробуйте изменить статус = «пустой» на статус = «успех», чтобы увидеть сообщение об успехе. Mocking позволяет быстро перерабатывать пользовательский интерфейс, прежде чем подключать какую-либо логику. Вот более детализированный прототип того же компонента, все еще «контролируемый» свойством status:
```js
export default function Form({
  // Try 'submitting', 'error', 'success':
  status = 'empty'
}) {
  if (status === 'success') {
    return <h1>That's right!</h1>
  }
  return (
    <>
      <h2>City quiz</h2>
      <p>
        In which city is there a billboard that turns air into drinkable water?
      </p>
      <form>
        <textarea disabled={
          status === 'submitting'
        } />
        <br />
        <button disabled={
          status === 'empty' ||
          status === 'submitting'
        }>
          Submit
        </button>
        {status === 'error' &&
          <p className="Error">
            Good guess but a wrong answer. Try again!
          </p>
        }
      </form>
      </>
  );
}

```
#### Displaying many visual states at once
Если компонент имеет много визуальных состояний, может быть удобно показать их все на одной странице:
```js 
import Form from './Form.js';

let statuses = [
  'empty',
  'typing',
  'submitting',
  'success',
  'error',
];

export default function App() {
  return (
    <>
      {statuses.map(status => (
        <section key={status}>
          <h4>Form ({status}):</h4>
          <Form status={status} />
        </section>
      ))}
    </>
  );
}

```

Pages like this are often called “living styleguides” or “storybooks”.

### Step 2: Determine what triggers those state changes [](https://react.dev/learn/reacting-to-input-with-state#step-2-determine-what-triggers-those-state-changes "Link for Step 2: Determine what triggers those state changes")
Вы можете инициировать обновления состояния в ответ на два вида входных данных:
- **Человеческие вводы,** такие как нажатие кнопки, ввод в поле, навигация по ссылке.
- **Компьютерные входы,** такие как прибытие сетевого ответа, завершение тайм-аута, загрузка изображения.
![[Pasted image 20240526212830.png]]
В обоих случаях **вы должны установить [переменные состояния](https://react.dev/learn/state-a-components-memory#anatomy-of-usestate) для обновления пользовательского интерфейса.** Для формы, которую вы разрабатываете, вам нужно будет изменить состояние в ответ на несколько различных входных данных:

- **Изменение ввода текста** (человека) должно переключить его из состояния _Пусто_ в состояние _ввода_ или обратно, в зависимости от того, пусто ли текстовое поле или нет.
- **Нажатие кнопки "Отправить"** (человек) должно переключить его в состояние "Отправить".
- **Успешный сетевой ответ** (компьютер) должен переключить его в состояние _Успеха_.
- **Неудачный сетевой ответ** (компьютер) должен переключить его в состояние _Error_ с соответствующим сообщением об ошибке.
Обратите внимание, что человеческие данные часто требуют [обработчиков событий](https://react.dev/learn/responding-to-events)!
Чтобы помочь визуализировать этот поток, попробуйте нарисовать каждое состояние на бумаге в виде помеченного круга, и каждое изменение между двумя состояниями в виде стрелки. Таким образом, вы можете нарисовать многие потоки и уладить ошибки задолго до реализации.
![[Pasted image 20240526213426.png]]
### Шаг 3: Представьте состояние в памяти с помощью`useState` [](https://react.dev/learn/reacting-to-input-with-state#step-3-represent-the-state-in-memory-with-usestate "Ссылка на этот заголовок")
Далее вам нужно будет представить визуальные состояния вашего компонента в памяти с помощью [`useState`.](https://react.dev/reference/react/useState) Простота - это ключ: каждая часть состояния - это "движущееся часть", и **вы хотите как можно меньше "подвижных частей".** Больше сложности приводит к большему количеству ошибок!
Начните с состояния, которое обязательно должно быть. Например, вам нужно будет сохранить ответ на ввод и ошибку (если она существует), чтобы сохранить последнюю ошибку:
![[Pasted image 20240526214657.png]]
Затем вам понадобится переменная состояния, представляющая, какое из визуальных состояний вы хотите отобразить. Обычно существует более чем один способ представить это в памяти, поэтому вам нужно будет поэкспериментировать с этим.

Если вы изо всех сил пытаетесь сразу придумать лучший способ, начните с добавления достаточного количества состояния, чтобы вы _определенно_ были уверены, что все возможные визуальные состояния охвачены:

![[Pasted image 20240526214719.png]]
Ваша первая идея, скорее всего, не будет лучшей, но это нормально - состояние рефакторинга является частью процесса!
### Шаг 4: Удалите любые несущественные переменные состояния[](https://react.dev/learn/reacting-to-input-with-state#step-4-remove-any-non-essential-state-variables "Ссылка на шаг 4: Удаление любых несущественных переменных состояния")
Вы хотите избежать дублирования в государственном контенте, поэтому вы отслеживаете только то, что необходимо. Потратив немного времени на рефакторинг структуры состояния, ваши компоненты будут легче понять, уменьшить дублирование и избежать непреднамеренных значений. Ваша цель - **предотвратить случаи, когда состояние в памяти не представляет какой-либо допустимый пользовательский интерфейс, который вы хотели бы видеть пользователь.** (Например, вы никогда не хотите показывать сообщение об ошибке и одновременно отключать ввод, иначе пользователь не сможет исправить ошибку!)
Вот несколько вопросов, которые вы можете задать о переменных вашего состояния:
- Вызывает ли это состояние парадокс? Например, isTyping и isSubmitting не могут одновременно быть истинными. Парадокс обычно означает, что государство недостаточно ограничено. Существует четыре возможных комбинации двух логических значений, но только три соответствуют допустимым состояниям. Чтобы удалить состояние «невозможно», вы можете объединить их в статус, который должен иметь одно из трех значений: «ввод», «отправка» или «успех».
- Доступна ли та же информация в другой переменной состояния? Еще один парадокс: isEmpty и isTyping не могут быть истинными одновременно. Делая их отдельными переменными состояния, вы рискуете потерять синхронизацию и вызвать ошибки. К счастью, вы можете удалить isEmpty и вместо этого проверить ответ.длина === 0.
- Вы получаете ту же информацию от обратной переменной состояния? isError не нужен, потому что вместо этого вы можете проверить ошибку !== null.
После этой очистки у вас осталось 3 (с 7!) _основные_ переменные состояния:

Вы знаете, что они необходимы, потому что вы не можете удалить ни один из них, не нарушив функциональность.
#### Eliminating “impossible” states with a reducer
Эти три переменные достаточно хорошо отражают состояние этой формы. Однако все еще существуют некоторые промежуточные состояния, которые не имеют полного смысла. Например, ненулевая ошибка не имеет смысла, если статус — «успех». Чтобы смоделировать состояние более точно, вы можете [extract it into a reducer.](https://react.dev/learn/extracting-state-logic-into-a-reducer). Редюсеры позволяют объединить несколько переменных состояния в один объект и объединить всю связанную логику!
### Шаг 5: Подключите обработчики событий для установки состояния[](https://react.dev/learn/reacting-to-input-with-state#step-5-connect-the-event-handlers-to-set-state "Ссылка на шаг 5: Подключите обработчики событий для установки состояния")
Наконец, создайте обработчики событий, которые обновляют состояние. Ниже приведена окончательная форма, со всеми обработчиками событий:
```js
import { useState } from 'react';

export default function Form() {
  const [answer, setAnswer] = useState('');
  const [error, setError] = useState(null);
  const [status, setStatus] = useState('typing');

  if (status === 'success') {
    return <h1>That's right!</h1>
  }

  async function handleSubmit(e) {
    e.preventDefault();
    setStatus('submitting');
    try {
      await submitForm(answer);
      setStatus('success');
    } catch (err) {
      setStatus('typing');
      setError(err);
    }
  }

  function handleTextareaChange(e) {
    setAnswer(e.target.value);
  }

  return (
    <>
      <h2>City quiz</h2>
      <p>
        In which city is there a billboard that turns air into drinkable water?
      </p>
      <form onSubmit={handleSubmit}>
        <textarea
          value={answer}
          onChange={handleTextareaChange}
          disabled={status === 'submitting'}
        />
        <br />
        <button disabled={
          answer.length === 0 ||
          status === 'submitting'
        }>
          Submit
        </button>
        {error !== null &&
          <p className="Error">
            {error.message}
          </p>
        }
      </form>
    </>
  );
}

function submitForm(answer) {
  // Pretend it's hitting the network.
  return new Promise((resolve, reject) => {
    setTimeout(() => {
      let shouldError = answer.toLowerCase() !== 'lima'
      if (shouldError) {
        reject(new Error('Good guess but a wrong answer. Try again!'));
      } else {
        resolve();
      }
    }, 1500);
  });
}

```
Хотя этот код длиннее, чем оригинальный пример императива, он гораздо менее хрупкий. Выражение всех взаимодействий по мере изменения состояния позволяет позже вводить новые визуальные состояния, не нарушая существующие. Он также позволяет изменять то, что должно отображаться в каждом состоянии, не изменяя логику самого взаимодействия.
