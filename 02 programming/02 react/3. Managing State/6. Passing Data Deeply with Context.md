Обычно вы передаете информацию от родительского компонента к дочернему компоненту через props. Но передача реквизита может стать многословной и неудобной, если вам придется передавать их через многие компоненты посередине, или если многим компонентам в вашем приложении нужна одна и та же информация. _Контекст_ позволяет родительскому компоненту сделать некоторую информацию доступной для любого компонента в дереве под ним - независимо от того, насколько глубоко - без явной передачи ее через реквизиты.
## The problem with passing props [](https://react.dev/learn/passing-data-deeply-with-context#the-problem-with-passing-props "Link for The problem with passing props")
[Передача реквизита](https://react.dev/learn/passing-props-to-a-component) - отличный способ явно передать данные через дерево пользовательского интерфейса к компонентам, которые его используют.

Но проходные реквизиты могут стать многословными и неудобными, когда вам нужно глубоко пройти какой-то реквизит через дерево, или если многим компонентам нужен один и тот же реквизит. Ближайший общий предок может быть далек от компонентов, которые нуждаются в данных, и [подъем](https://react.dev/learn/sharing-state-between-components) такого высокого [состояния](https://react.dev/learn/sharing-state-between-components) может привести к ситуации, называемой «бурением выступов».

![[Pasted image 20240527191452.png]]
Разве не было бы здорово, если бы был способ "телепортировать" данные в компоненты в дереве, которые в них нуждаются, без передачи реквизита? С контекстной функцией React, есть!
## Context: an alternative to passing props [](https://react.dev/learn/passing-data-deeply-with-context#context-an-alternative-to-passing-props "Link for Context: an alternative to passing props")
Контекст позволяет родительскому компоненту предоставлять данные всему дереву под ним. Контекст имеет множество применений. Вот один из примеров. Рассмотрим этот компонент заголовка, который принимает уровень в качестве своего размера:
```js
import Heading from './Heading.js';
import Section from './Section.js';

export default function Page() {
  return (
    <Section>
      <Heading level={1}>Title</Heading>
      <Heading level={2}>Heading</Heading>
      <Heading level={3}>Sub-heading</Heading>
      <Heading level={4}>Sub-sub-heading</Heading>
      <Heading level={5}>Sub-sub-sub-heading</Heading>
      <Heading level={6}>Sub-sub-sub-sub-heading</Heading>
    </Section>
  );
}

```
Допустим, вы хотите, чтобы несколько заголовков в одном разделе всегда имели одинаковый размер:
```js
import Heading from './Heading.js';
import Section from './Section.js';

export default function Page() {
  return (
    <Section>
      <Heading level={1}>Title</Heading>
      <Section>
        <Heading level={2}>Heading</Heading>
        <Heading level={2}>Heading</Heading>
        <Heading level={2}>Heading</Heading>
        <Section>
          <Heading level={3}>Sub-heading</Heading>
          <Heading level={3}>Sub-heading</Heading>
          <Heading level={3}>Sub-heading</Heading>
          <Section>
            <Heading level={4}>Sub-sub-heading</Heading>
            <Heading level={4}>Sub-sub-heading</Heading>
            <Heading level={4}>Sub-sub-heading</Heading>
          </Section>
        </Section>
      </Section>
    </Section>
  );
}

```
Currently, you pass the `level` prop to each `<Heading>` separately:
![[Pasted image 20240527200205.png]]
Было бы неплохо, если бы вместо этого вы могли передать свойство уровня компоненту \<Section> и удалить его из \<Heading>. Таким образом вы можете обеспечить, чтобы все заголовки в одном разделе имели одинаковый размер:
![[Pasted image 20240527200235.png]]
Но как компонент \<Heading> может узнать уровень ближайшего к нему \<Section>? Для этого ребенку потребуется какой-то способ «запросить» данные откуда-то сверху в дереве.
Вы не можете сделать это только с реквизитом. Именно здесь вступает в игру контекст. Вы сделаете это в три шага:
1. **Создайте** контекст. (Вы можете назвать это `LevelContext`, так как это для уровня заголовка.)
2. 1. **Use** that context from the component that needs the data. (`Heading` will use `LevelContext`.)
3. 1. **Provide** that context from the component that specifies the data. (`Section` will provide `LevelContext`.)
Контекст позволяет родителю - даже отдаленному! - предоставить некоторые данные всему дереву внутри него.
![[Pasted image 20240527200344.png]]
### Step 1: Create the context [](https://react.dev/learn/passing-data-deeply-with-context#step-1-create-the-context "Link for Step 1: Create the context")
Во-первых, вам нужно создать контекст. Вам нужно будет **экспортировать его из файла**, чтобы ваши компоненты могли его использовать:
![[Pasted image 20240527200411.png]]
Единственным аргументом createContext является значение по умолчанию. Здесь 1 относится к самому большому уровню заголовка, но вы можете передать любое значение (даже объект). Значение значения по умолчанию вы увидите на следующем шаге.
### Step 2: Use the context [](https://react.dev/learn/passing-data-deeply-with-context#step-2-use-the-context "Link for Step 2: Use the context")
Import the `useContext` Hook from React and your context:
![[Pasted image 20240527200448.png]]
Currently, the `Heading` component reads `level` from props:
![[Pasted image 20240527200513.png]]
Вместо этого удалите свойство уровня и прочитайте значение из только что импортированного контекста LevelContext:
![[Pasted image 20240527200539.png]]
`seContext` is a Hook. Just like `useState` and `useReducer`, you can only call a Hook immediately inside a React component (not inside loops or conditions). **`useContext` tells React that the `Heading` component wants to read the `LevelContext`.**
Теперь, когда компонент Heading не имеет свойства level, вам больше не нужно передавать свойство level в заголовок в JSX следующим образом:
![[Pasted image 20240527200628.png]]
Обновите JSX, чтобы вместо этого его получил `Section`:
![[Pasted image 20240527200806.png]]
Напоминаем, что это разметка, которую вы пытались получить:
```js
import Heading from './Heading.js';
import Section from './Section.js';

export default function Page() {
  return (
    <Section level={1}>
      <Heading>Title</Heading>
      <Section level={2}>
        <Heading>Heading</Heading>
        <Heading>Heading</Heading>
        <Heading>Heading</Heading>
        <Section level={3}>
          <Heading>Sub-heading</Heading>
          <Heading>Sub-heading</Heading>
          <Heading>Sub-heading</Heading>
          <Section level={4}>
            <Heading>Sub-sub-heading</Heading>
            <Heading>Sub-sub-heading</Heading>
            <Heading>Sub-sub-heading</Heading>
          </Section>
        </Section>
      </Section>
    </Section>
  );
}

```
Обратите внимание, что этот пример еще не совсем работает! Все заголовки имеют одинаковый размер, потому что**, несмотря на то, что вы _используете_ контекст, вы еще не _предоставили_ его.** React не знает, где его взять!
Если вы не предоставите контекст, React будет использовать значение по умолчанию, указанное вами на предыдущем шаге. В этом примере вы указали 1 в качестве аргумента createContext, поэтому useContext(LevelContext) возвращает 1, устанавливая всем этим заголовкам значение \<h1>. Давайте исправим эту проблему, предоставив каждому разделу свой собственный контекст.
### Step 3: Provide the context [](https://react.dev/learn/passing-data-deeply-with-context#step-3-provide-the-context "Link for Step 3: Provide the context")
Компонент `Section` в настоящее время делает свои дочерние:
![[Pasted image 20240527200943.png]]
**Оберните их поставщиком контекста**, чтобы предоставить им `LevelContext`:
![[Pasted image 20240527201018.png]]
Это говорит React: «если какой-либо компонент внутри этого \<Section> запрашивает LevelContext, дайте ему этот уровень». Компонент будет использовать значение ближайшего <LevelContext.Provider> в дереве пользовательского интерфейса над ним.
Это тот же результат, что и исходный код, но вам не нужно передавать свойство уровня каждому компоненту заголовка! Вместо этого он «вычисляет» уровень своего заголовка, запрашивая ближайший раздел выше:
1. You pass a `level` prop to the `<Section>`.
2.  `Section`обертывает своих дочерних в `<LevelContext.Provider value={level}>`.
3.  `Heading` asks the closest value of `LevelContext` above with `useContext(LevelContext)`.
## Using and providing context from the same component [](https://react.dev/learn/passing-data-deeply-with-context#using-and-providing-context-from-the-same-component "Link for Using and providing context from the same component")
В настоящее время вам все равно придется указывать уровень каждого раздела вручную:
![[Pasted image 20240527201209.png]]
Поскольку контекст позволяет вам читать информацию из компонента выше, каждый раздел может читать уровень из раздела выше и автоматически передавать уровень + 1 вниз. Вот как вы можете это сделать:
![[Pasted image 20240527201248.png]]Благодаря этому изменению вам не нужно передавать свойство уровня ни в \<Section>, ни в \<Heading>:
```js
import Heading from './Heading.js';
import Section from './Section.js';

export default function Page() {
  return (
    <Section>
      <Heading level={1}>Title</Heading>
      <Section>
        <Heading level={2}>Heading</Heading>
        <Heading level={2}>Heading</Heading>
        <Heading level={2}>Heading</Heading>
        <Section>
          <Heading level={3}>Sub-heading</Heading>
          <Heading level={3}>Sub-heading</Heading>
          <Heading level={3}>Sub-heading</Heading>
          <Section>
            <Heading level={4}>Sub-sub-heading</Heading>
            <Heading level={4}>Sub-sub-heading</Heading>
            <Heading level={4}>Sub-sub-heading</Heading>
          </Section>
        </Section>
      </Section>
    </Section>
  );
}

```
Теперь и заголовок, и раздел читают LevelContext, чтобы определить, насколько они «глубоки». И Раздел оборачивает своих дочерних элементов в LevelContext, чтобы указать, что все внутри него находится на «более глубоком» уровне.

В этом примере используются уровни заголовков, потому что они визуально показывают, как вложенные компоненты могут переопределить контекст. Но контекст полезен и для многих других сценариев использования. Вы можете передать любую информацию, необходимую всему поддереву: текущую цветовую тему, зарегистрированного в данный момент пользователя и так далее.

## Context passes through intermediate components [](https://react.dev/learn/passing-data-deeply-with-context#context-passes-through-intermediate-components "Link for Context passes through intermediate components")
Bы можете вставить столько компонентов, сколько хотите, между компонентом, который обеспечивает контекст, и тем, который его использует. Это включает в себя как встроенные компоненты, такие как `<div>`, так и компоненты, которые вы можете создать самостоятельно.
В этом примере один и тот же компонент Post (с пунктирной рамкой) отображается на двух разных уровнях вложенности. Обратите внимание, что \<Heading> внутри него автоматически получает свой уровень из ближайшего \<Section>:
```js
import Heading from './Heading.js';
import Section from './Section.js';

export default function ProfilePage() {
  return (
    <Section>
      <Heading>My Profile</Heading>
      <Post
        title="Hello traveller!"
        body="Read about my adventures."
      />
      <AllPosts />
    </Section>
  );
}

function AllPosts() {
  return (
    <Section>
      <Heading>Posts</Heading>
      <RecentPosts />
    </Section>
  );
}

function RecentPosts() {
  return (
    <Section>
      <Heading>Recent Posts</Heading>
      <Post
        title="Flavors of Lisbon"
        body="...those pastéis de nata!"
      />
      <Post
        title="Buenos Aires in the rhythm of tango"
        body="I loved it!"
      />
    </Section>
  );
}

function Post({ title, body }) {
  return (
    <Section isFancy={true}>
      <Heading>
        {title}
      </Heading>
      <p><i>{body}</i></p>
    </Section>
  );
}

```
Вы не сделали ничего особенного, чтобы это сработало. Раздел определяет контекст дерева внутри него, поэтому вы можете вставить \<Heading> где угодно, и он будет иметь правильный размер. Попробуйте это в песочнице выше!
**Контекст позволяет писать компоненты, которые «адаптируются к своему окружению» и отображаются по-разному в зависимости от того_, где_ (или, другими словами, _в каком контексте_) они отображаются.**

Принцип работы контекста может напомнить вам о наследовании свойств CSS. В CSS вы можете указать цвет: синий для \<div>, и любой узел DOM внутри него, независимо от его глубины, унаследует этот цвет, если только какой-либо другой узел DOM в середине не переопределит его цветом: зеленый. Аналогично, в React единственный способ переопределить некоторый контекст, поступающий сверху, — это обернуть дочерние элементы в поставщика контекста с другим значением.
В CSS различные свойства, такие как цвет и фоновый цвет, не переопределяют друг друга. Вы можете установить красный цвет для всех \<div>, не затрагивая цвет фона. Аналогично, разные контексты React не переопределяют друг друга. Каждый контекст, который вы создаете с помощью createContext(), полностью отделен от других и связывает воедино компоненты, использующие и предоставляющие этот конкретный контекст. Один компонент может без проблем использовать или предоставлять множество различных контекстов.
## Before you use context [](https://react.dev/learn/passing-data-deeply-with-context#before-you-use-context "Link for Before you use context")
Контекст очень заманчивый в использовании! Тем не менее, это также означает, что его слишком легко злоупотреблять. **То, что вам нужно пройти несколько уровней, не означает, что вы должны поместить эту информацию в контекст.**
Вот несколько альтернатив, которые вы должны рассмотреть, прежде чем использовать контекст:
1. **Начните с [передачи реквизита.](https://react.dev/learn/passing-props-to-a-component)** Если ваши компоненты не тривиальны, не редкость передавать дюжину реквизита через дюжину компонентов. Это может показаться лохом, но это очень ясно, какие компоненты используют какие данные! Человек, который, который ваш код, будет рад, что вы сделали поток данных явным с реквизитом.
2. Извлеките компоненты и передайте им JSX как дочерние элементы. Если вы передаете некоторые данные через множество слоев промежуточных компонентов, которые не используют эти данные (а передают их только ниже), это часто означает, что вы забыли по пути извлечь некоторые компоненты. Например, возможно, вы передаете реквизиты данных, такие как публикации, визуальным компонентам, которые не используют их напрямую, например <Layout Posts={posts} />. Вместо этого заставьте Layout использовать дочерние элементы в качестве реквизита и визуализировать <Layout><Posts messages={posts} /></Layout>. Это уменьшает количество слоев между компонентом, определяющим данные, и тем, которому они нужны.
Если ни один из этих подходов не подходит для вас, рассмотрите контекст.
## Use cases for context [](https://react.dev/learn/passing-data-deeply-with-context#use-cases-for-context "Link for Use cases for context")
- **Тема:** Если ваше приложение позволяет пользователю изменить свой внешний вид (например, темный режим), вы можете поместить поставщика контекста в верхней части вашего приложения и использовать этот контекст в компонентах, которые должны настроить свой внешний вид.
- - **Текущая учетная запись:** многим компонентам может потребоваться знать пользователя, который в данный момент вошел в систему. Помещая его в контекст, его удобно читать в любом месте дерева. Некоторые приложения также позволяют управлять несколькими учетными записями одновременно (например, оставлять комментарий в качестве другого пользователя). В этих случаях может быть удобно обернуть часть пользовательского интерфейса во вложенного провайдера с другой стоимостью текущего счета.
- - **Маршрутизация:** Большинство маршрутных решений используют контекст внутри для удержания текущего маршрута. Вот как каждая ссылка "знает", активна она или нет. Если вы создаете свой собственный маршрутизатор, вы, возможно, тоже захотите это сделать.
- - **Управление состоянием:** По мере роста вашего приложения вы можете получить много состояния ближе к верхней части вашего приложения. Многие удаленные компоненты ниже могут захотеть изменить его. Обычно [используется редуктор вместе с контекстом](https://react.dev/learn/scaling-up-with-reducer-and-context) для управления сложным состоянием и передает его удаленным компонентам без особых хлопот.
Контекст не ограничивается статическими значениями. Если вы передадите другое значение при следующем рендере, React обновит все компоненты, прочитав его ниже! Вот почему контекст часто используется в сочетании со состоянием.
В целом, если какая-то информация нужна удаленным компонентам в разных частях дерева, это хороший признак того, что контекст вам поможет.
