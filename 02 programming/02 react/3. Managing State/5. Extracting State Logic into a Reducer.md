Компоненты с множеством обновлений состояния, распределенные по многим обработчикам событий, могут стать ошеломляющими. В этих случаях вы можете консолидировать всю логику обновления состояния за пределами вашего компонента в одной функции, называемой _редуктором._
## Consolidate state logic with a reducer [](https://react.dev/learn/extracting-state-logic-into-a-reducer#consolidate-state-logic-with-a-reducer "Link for Consolidate state logic with a reducer")
По мере усложнения компонентов становится все труднее сразу увидеть все различные способы обновления состояния компонента. Например, приведенный ниже компонент TaskApp хранит массив задач в состоянии и использует три разных обработчика событий для добавления, удаления и редактирования задач:
```js
import { useState } from 'react';
import AddTask from './AddTask.js';
import TaskList from './TaskList.js';

export default function TaskApp() {
  const [tasks, setTasks] = useState(initialTasks);

  function handleAddTask(text) {
    setTasks([
      ...tasks,
      {
        id: nextId++,
        text: text,
        done: false,
      },
    ]);
  }

  function handleChangeTask(task) {
    setTasks(
      tasks.map((t) => {
        if (t.id === task.id) {
          return task;
        } else {
          return t;
        }
      })
    );
  }

  function handleDeleteTask(taskId) {
    setTasks(tasks.filter((t) => t.id !== taskId));
  }

  return (
    <>
      <h1>Prague itinerary</h1>
      <AddTask onAddTask={handleAddTask} />
      <TaskList
        tasks={tasks}
        onChangeTask={handleChangeTask}
        onDeleteTask={handleDeleteTask}
      />
    </>
  );
}

let nextId = 3;
const initialTasks = [
  {id: 0, text: 'Visit Kafka Museum', done: true},
  {id: 1, text: 'Watch a puppet show', done: false},
  {id: 2, text: 'Lennon Wall pic', done: false},
];

```
Каждый из его обработчиков событий вызывает `setTasks`, чтобы обновить состояние. По мере роста этого компонента растет и количество логики состояния, разбросанной по нему. Чтобы уменьшить эту сложность и сохранить всю вашу логику в одном легкодоступном месте, вы можете переместить эту логику состояния в одну функцию за пределами вашего компонента, **называемую «reducer».**
Reducers are a different way to handle state. You can migrate from `useState` to `useReducer` in three steps:
1. **Переход** от состояния установки к действиям диспетчеризации.
2. **Напишите** функцию редуктора.
3. **Используйте** редуктор из вашего компонента.
### Step 1: Move from setting state to dispatching actions [](https://react.dev/learn/extracting-state-logic-into-a-reducer#step-1-move-from-setting-state-to-dispatching-actions "Link for Step 1: Move from setting state to dispatching actions")
Ваши обработчики событий в настоящее время указывают_, что делать_, устанавливая состояние:
![[Pasted image 20240527160023.png]]
Удалите всю логику установки состояния. У вас осталось три обработчика событий:
- `handleAddTask(text)`вызывается, когда пользователь нажимает "Добавить".
- - `handleChangeTask(task)`вызывается, когда пользователь переключает задачу или нажимает "Сохранить".
- - `handleDeleteTask(taskId)`вызывается, когда пользователь нажимает "Удалить".
Управление состоянием с помощью редукторов немного отличается от непосредственной установки состояния. Вместо того, чтобы говорить React "что делать", устанавливая состояние, вы указываете "что только что сделал пользователь", отправляя "действия" от ваших обработчиков событий. (Логика обновления состояния будет жить в другом месте!) Таким образом, вместо того, чтобы "устанавливать `tasks`" с помощью обработчика событий, вы отправляете действие "добавлено/изменено/удалено задачу". Это более описательное намерение пользователя.

Объект, который вы передаете в `dispatch`, называется "действием":

Это обычный объект JavaScript. Вы сами решаете, что в него поместить, но в целом оно должно содержать минимальную информацию о том, что произошло. (Вы добавите саму функцию диспетчеризации на более позднем этапе.)

Объект действия может иметь любую форму.
По соглашению, ему обычно присваивают строковый тип, описывающий произошедшее, и передают любую дополнительную информацию в другие поля. Тип специфичен для компонента, поэтому в этом примере подойдет либо «added», либо «added_task». Выберите имя, которое говорит о том, что произошло!

### Step 2: Write a reducer function [](https://react.dev/learn/extracting-state-logic-into-a-reducer#step-2-write-a-reducer-function "Link for Step 2: Write a reducer function")
Функция редуктора - это то место, где вы поместите свою логику состояния. Он принимает два аргумента, текущее состояние и объект действия, и возвращает следующее состояние:
![[Pasted image 20240527160338.png]]
React установит состояние на то, что вы вернете из редуктора.

Чтобы переместить логику установки состояния из обработчиков событий в функцию редуктора в этом примере, вы будете:

1. Объявите текущее состояние (`tasks`) в качестве первого аргумента.
2. Declare the `action` object as the second argument.
3. Возвращает _следующее_ состояние из редуктора (в которое React установит состояние).

Вот вся логика настройки состояния, перенесенная на функцию редуктора:
![[Pasted image 20240527160419.png]]
Поскольку функция редуктора принимает состояние (`tasks`) в качестве аргумента, вы можете **объявить ее за пределами вашего компонента.** Это снижает уровень отступа и может облегчить чтение вашего кода.
В приведенном выше коде используются операторы if/else, но это соглашение об использовании [операторов switch](https://developer.mozilla.org/docs/Web/JavaScript/Reference/Statements/switch) внутри редукторов. Результат тот же, но может быть проще читать операторы switch с первого взгляда.
![[Pasted image 20240527160517.png]]
Мы рекомендуем заключить каждый блок case в фигурные скобки { и }, чтобы переменные, объявленные внутри разных регистров, не конфликтовали друг с другом. Кроме того, дело обычно должно заканчиваться возвратом. Если вы забудете вернуться, код «провалится» на следующий случай, что может привести к ошибкам!
сли вам еще не нравятся операторы switch, использование if/else вполне нормально.
#### Why are reducers called this way?
Хотя редукторы могут "уменьшить" количество кода внутри вашего компонента, на самом деле они названы в честь операции [`reduce()`](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Array/Reduce), которую вы можете выполнить в массивах.
Операция reduce() позволяет вам взять массив и «накопить» одно значение из многих:
![[Pasted image 20240527160630.png]]
Функция, которую вы передаете для `reduce`, известна как "reducer". Он берет _результат_ до _сих пор_ и _текущий элемент,_ а затем возвращает _следующий результат._ Редукторы реакции являются примером той же идеи: они берут _состояние до сих пор_ и _действие_ и возвращают _следующее состояние._ Таким образом, они накапливают действия с течением времени в состоянии.
Вы даже можете использовать метод reduce() с начальным состоянием и массивом действий для расчета конечного состояния, передав ему функцию редуктора:
Вероятно, вам не придется делать это самостоятельно, но это похоже на то, что делает React!

Finally, you need to hook up the `tasksReducer` to your component. Import the `useReducer` Hook from React:

Затем вы можете заменить `useState`:
![[Pasted image 20240527161003.png]]
с `useReducer`, как так:
![[Pasted image 20240527161014.png]]
Хук useReducer похож на useState — вы должны передать ему начальное состояние, и он возвращает значение с состоянием и способ установки состояния (в данном случае, функцию отправки). Но это немного другое.
The `useReducer` Hook takes two arguments:

1. Функция редуктора
2. Начальное состояние
И он возвращает:
1. state value
2. dispatch отправки (для "отправки" действий пользователя в редуктор)

Теперь он полностью подключен! Здесь редуктор объявляется в нижней части файла компонента:
```js
import { useReducer } from 'react';
import AddTask from './AddTask.js';
import TaskList from './TaskList.js';

export default function TaskApp() {
  const [tasks, dispatch] = useReducer(tasksReducer, initialTasks);

  function handleAddTask(text) {
    dispatch({
      type: 'added',
      id: nextId++,
      text: text,
    });
  }

  function handleChangeTask(task) {
    dispatch({
      type: 'changed',
      task: task,
    });
  }

  function handleDeleteTask(taskId) {
    dispatch({
      type: 'deleted',
      id: taskId,
    });
  }

  return (
    <>
      <h1>Prague itinerary</h1>
      <AddTask onAddTask={handleAddTask} />
      <TaskList
        tasks={tasks}
        onChangeTask={handleChangeTask}
        onDeleteTask={handleDeleteTask}
      />
    </>
  );
}

function tasksReducer(tasks, action) {
  switch (action.type) {
    case 'added': {
      return [
        ...tasks,
        {
          id: action.id,
          text: action.text,
          done: false,
        },
      ];
    }
    case 'changed': {
      return tasks.map((t) => {
        if (t.id === action.task.id) {
          return action.task;
        } else {
          return t;
        }
      });
    }
    case 'deleted': {
      return tasks.filter((t) => t.id !== action.id);
    }
    default: {
      throw Error('Unknown action: ' + action.type);
    }
  }
}

let nextId = 3;
const initialTasks = [
  {id: 0, text: 'Visit Kafka Museum', done: true},
  {id: 1, text: 'Watch a puppet show', done: false},
  {id: 2, text: 'Lennon Wall pic', done: false},
];

```

Логика компонентов может быть легче читать, когда вы разделяете такие проблемы. Теперь обработчики событий определяют только _то, что произошло_ путем отправки действий, а функция редуктора определяет_, как состояние обновляется_ в ответ на них.
## Comparing `useState` and `useReducer` [](https://react.dev/learn/extracting-state-logic-into-a-reducer#comparing-usestate-and-usereducer "Ссылка на этот заголовок")
Pедукторы не лишены недостатков! Вот несколько способов их сравнить:
- Размер кода. Как правило, при использовании useState вам приходится писать меньше кода заранее. При использовании useReducer вам необходимо написать как функцию редуктора, так и действия диспетчеризации. Однако useReducer может помочь сократить объем кода, если многие обработчики событий изменяют состояние одинаковым образом.
- Читабельность: useState очень легко читается, когда обновления состояния просты. Когда они становятся более сложными, они могут раздуть код вашего компонента и затруднить его сканирование. В этом случае useReducer позволяет четко отделить логику обновления от того, что происходит с обработчиками событий.
- Отладка: если у вас есть ошибка с useState, может быть сложно определить, где состояние было установлено неправильно и почему. С помощью useReducer вы можете добавить журнал консоли в свой редуктор, чтобы видеть каждое обновление состояния и то, почему это произошло (из-за какого действия). Если каждое действие правильное, вы будете знать, что ошибка кроется в самой логике редюсера. Однако вам придется выполнить больше кода, чем при использовании useState.
- - **Тестирование:** редуктор - это чистая функция, которая не зависит от вашего компонента. Это означает, что вы можете экспортировать и тестировать его отдельно. Хотя, как правило, лучше всего тестировать компоненты в более реалистичной среде, для сложной логики обновления состояния может быть полезно утверждать, что ваш редуктор возвращает определенное состояние для определенного начального состояния и действия.
- Личные предпочтения: некоторым нравятся редукторы, другим нет. Это нормально. Это вопрос предпочтений. Вы всегда можете конвертировать между useState и useReducer туда и обратно: они эквивалентны!
Мы рекомендуем использовать редуктор, если вы часто сталкиваетесь с ошибками из-за некорректного обновления состояния какого-либо компонента и хотите придать его коду больше структуры. Вам не обязательно использовать редьюсеры для всего: не стесняйтесь смешивать и сочетать! Вы даже можете использовать State и useReducer в одном компоненте.
## Writing reducers well [](https://react.dev/learn/extracting-state-logic-into-a-reducer#writing-reducers-well "Link for Writing reducers well")
Имейте в виду эти два совета при написании редукторов:
- **Редукторы должны быть чистыми.** Подобно [функциям обновления состояния](https://react.dev/learn/queueing-a-series-of-state-updates), редукторы работают во время рендеринга! (Действия поставлены в очередь до следующего рендеринга.) Это означает, что редукторы [должны быть чистыми](https://react.dev/learn/keeping-components-pure) - одни и те же входы всегда приводят к одному и тому же выходу. Они не должны отправлять запросы, планировать тайм-ауты или выполнять какие-либо побочные эффекты (операции, которые влияют на вещи за пределами компонента). Они должны обновлять [объекты](https://react.dev/learn/updating-objects-in-state) и [массивы](https://react.dev/learn/updating-arrays-in-state) без мутаций.
- **Каждое действие описывает одно взаимодействие с пользователем, даже если это приводит к многочисленным изменениям в данных.** Например, если пользователь нажимает "Сброс" в форме с пятью полями, управляемыми редуктором, имеет смысл отправить одно действие `reset_form`, а не пять отдельных действий `set_field`. Если вы регистрируете каждое действие в редукторе, этот журнал должен быть достаточно четким, чтобы вы реконструировали, какие взаимодействия или ответы происходили в каком порядке. Это помогает с отладкой!
## Writing concise reducers with Immer [](https://react.dev/learn/extracting-state-logic-into-a-reducer#writing-concise-reducers-with-immer "Link for Writing concise reducers with Immer")
Как и при обновлении объектов и массивов в обычном состоянии, вы можете использовать библиотеку Immer, чтобы сделать редукторы более краткими. Здесь useImmerReducer позволяет вам изменять состояние с помощью push или arr[i] = присваивания:
```js
import { useImmerReducer } from 'use-immer';
import AddTask from './AddTask.js';
import TaskList from './TaskList.js';

function tasksReducer(draft, action) {
  switch (action.type) {
    case 'added': {
      draft.push({
        id: action.id,
        text: action.text,
        done: false,
      });
      break;
    }
    case 'changed': {
      const index = draft.findIndex((t) => t.id === action.task.id);
      draft[index] = action.task;
      break;
    }
    case 'deleted': {
      return draft.filter((t) => t.id !== action.id);
    }
    default: {
      throw Error('Unknown action: ' + action.type);
    }
  }
}

export default function TaskApp() {
  const [tasks, dispatch] = useImmerReducer(tasksReducer, initialTasks);

  function handleAddTask(text) {
    dispatch({
      type: 'added',
      id: nextId++,
      text: text,
    });
  }

  function handleChangeTask(task) {
    dispatch({
      type: 'changed',
      task: task,
    });
  }

  function handleDeleteTask(taskId) {
    dispatch({
      type: 'deleted',
      id: taskId,
    });
  }

  return (
    <>
      <h1>Prague itinerary</h1>
      <AddTask onAddTask={handleAddTask} />
      <TaskList
        tasks={tasks}
        onChangeTask={handleChangeTask}
        onDeleteTask={handleDeleteTask}
      />
    </>
  );
}

let nextId = 3;
const initialTasks = [
  {id: 0, text: 'Visit Kafka Museum', done: true},
  {id: 1, text: 'Watch a puppet show', done: false},
  {id: 2, text: 'Lennon Wall pic', done: false},
];

```
Редюсеры должны быть чистыми, поэтому они не должны изменять состояние. Но Immer предоставляет вам специальный черновой объект, который можно безопасно изменять. Под капотом Immer создаст копию вашего состояния с изменениями, которые вы внесли в черновик. Вот почему редукторы, управляемые useImmerReducer, могут изменять свой первый аргумент и им не нужно возвращать состояние.
