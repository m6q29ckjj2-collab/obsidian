There are four ways to navigate between routes in Next.js:
- Using the [`<Link>` Component](https://nextjs.org/docs/app/building-your-application/routing/linking-and-navigating#link-component)
- Using the [`useRouter` hook](https://nextjs.org/docs/app/building-your-application/routing/linking-and-navigating#userouter-hook) ([Client Components](https://nextjs.org/docs/app/building-your-application/rendering/client-components))
- Using the [`redirect` function](https://nextjs.org/docs/app/building-your-application/routing/linking-and-navigating#redirect-function) ([Server Components](https://nextjs.org/docs/app/building-your-application/rendering/server-components))
- Using the native [History API](https://nextjs.org/docs/app/building-your-application/routing/linking-and-navigating#using-the-native-history-api)
На этой странице будет описано, как использовать каждую из этих опций, а также более подробно описано, как работает навигация.

\<Link> — это встроенный компонент, который расширяет HTML-тег \<a>, обеспечивая предварительную выборку и навигацию между маршрутами на стороне клиента. Это основной и рекомендуемый способ навигации между маршрутами в Next.js.
You can use it by importing it from `next/link`, and passing a `href` prop to the component:
![[Pasted image 20240523233338.png]]
There are other optional props you can pass to `<Link>`. See the [API reference](https://nextjs.org/docs/app/api-reference/components/link) for more.

## [How Routing and Navigation Works](https://nextjs.org/docs/app/building-your-application/routing/linking-and-navigating#how-routing-and-navigation-works)
App Router использует гибридный подход для маршрутизации и навигации. На сервере код вашего приложения автоматически разделяется по сегментам маршрута. А на клиенте Next.js предварительно выбирает и кэширует сегменты маршрута. Это означает, что когда пользователь переходит к новому маршруту, браузер не перезагружает страницу, а перерисовываются только те сегменты маршрута, которые изменились, что улучшает удобство навигации и производительность.

### [1. Code Splitting](https://nextjs.org/docs/app/building-your-application/routing/linking-and-navigating#1-code-splitting)
Разделение кода позволяет разделить код приложения на более мелкие пакеты, которые будут загружаться и выполняться браузером. Это уменьшает объем передаваемых данных и время выполнения каждого запроса, что приводит к повышению производительности.

Серверные компоненты позволяют автоматически разбивать код вашего приложения по сегментам маршрута. Это означает, что в навигацию загружается только код, необходимый для текущего маршрута.

### [2. Prefetching](https://nextjs.org/docs/app/building-your-application/routing/linking-and-navigating#2-prefetching)
Предварительная выборка — это способ предварительной загрузки маршрута в фоновом режиме до того, как пользователь его посетит.

В Next.js существует два способа предварительной выборки маршрутов:
- Компонент \<Link>: маршруты автоматически выбираются заранее, когда они становятся видимыми в области просмотра пользователя. Предварительная загрузка происходит при первой загрузке страницы или когда она появляется в поле зрения при прокрутке.
- - **`router.prefetch()`**: The `useRouter` hook can be used to prefetch routes programmatically.
Поведение предварительной выборки <Link> по умолчанию (т. е. когда свойство предварительной выборки не указано или установлено в значение null) различается в зависимости от того, как вы используете loading.js. Только общий макет, расположенный вниз по визуализированному «дереву» компонентов до первого файла loading.js, предварительно извлекается и кэшируется на 30 секунд. Это снижает стоимость получения всего динамического маршрута и означает, что вы можете отображать мгновенное состояние загрузки для лучшей визуальной обратной связи с пользователями.
Вы можете отключить предварительную выборку, установив для свойства prefetch значение false. Альтернативно, вы можете выполнить предварительную выборку всех данных страницы за пределами границ загрузки, установив для свойства prefetch значение true.
>Good to know
>Prefetching is not enabled in development, only in production

### [3. Caching](https://nextjs.org/docs/app/building-your-application/routing/linking-and-navigating#3-caching)
Next.js имеет кэш на стороне клиента в памяти, называемый кэшем маршрутизатора. Когда пользователи перемещаются по приложению, полезная нагрузка компонента React Server, состоящая из заранее выбранных сегментов маршрутов и посещенных маршрутов, сохраняется в кеше.

Это означает, что при навигации кеш используется максимально повторно, вместо того, чтобы делать новый запрос к серверу — повышение производительности за счет уменьшения количества запросов и передаваемых данных.

Learn more about how the [Router Cache](https://nextjs.org/docs/app/building-your-application/caching#router-cache) works and how to configure it.

### [4. Partial Rendering](https://nextjs.org/docs/app/building-your-application/routing/linking-and-navigating#4-partial-rendering)
Частичный рендеринг означает, что на клиенте перерисовываются только те сегменты маршрута, которые изменяются при навигации, а все общие сегменты сохраняются.

Например, при навигации между двумя родственными маршрутами, /dashboard/settings и /dashboard/analytics, страницы настроек и аналитики будут отображаться, а общий макет информационной панели будет сохранен.
![[Pasted image 20240524113203.png]]
Без частичного рендеринга каждая навигация приведет к повторному рендерингу всей страницы на клиенте. Рендеринг только того сегмента, который изменяется, сокращает объем передаваемых данных и время выполнения, что приводит к повышению производительности.

### [5. Soft Navigation](https://nextjs.org/docs/app/building-your-application/routing/linking-and-navigating#5-soft-navigation)
Браузеры выполняют «жесткую навигацию» при переходе между страницами. Маршрутизатор приложений Next.js обеспечивает «мягкую навигацию» между страницами, гарантируя повторную визуализацию только измененных сегментов маршрута (частичная визуализация). Это позволяет сохранять состояние React клиента во время навигации.

### [6. Back and Forward Navigation](https://nextjs.org/docs/app/building-your-application/routing/linking-and-navigating#6-back-and-forward-navigation)
По умолчанию Next.js сохраняет положение прокрутки для навигации вперед и назад и повторно использует сегменты маршрута в кэше маршрутизатора.

### [7. Routing between `pages/` and `app/`](https://nextjs.org/docs/app/building-your-application/routing/linking-and-navigating#7-routing-between-pages-and-app)
При постепенном переходе со pages на app/ маршрутизатор Next.js автоматически выполняет сложную навигацию между ними. Для обнаружения переходов от pages/ к app/ существует фильтр клиентского маршрутизатора, который использует вероятностную проверку маршрутов приложений, что иногда может приводить к ложным срабатываниям. По умолчанию такие случаи должны быть очень редкими, поскольку мы настраиваем вероятность ложного срабатывания на уровне 0,01%. Эту вероятность можно настроить с помощью параметра Experimental.clientRouterFilterAllowedRate в файле next.config.js. Важно отметить, что снижение уровня ложных срабатываний приведет к увеличению размера создаваемого фильтра в клиентском пакете.
В качестве альтернативы, если вы предпочитаете полностью отключить эту обработку и управлять маршрутизацией между страницами/ и приложением/ вручную, вы можете установить для экспериментального.clientRouterFilter значение false в next.config.js. Если эта функция отключена, любые динамические маршруты на страницах, которые перекрываются с маршрутами приложения, по умолчанию не будут правильно перемещаться.
