Промежуточное программное обеспечение позволяет запускать код до завершения запроса. Затем на основе входящего запроса вы можете изменить ответ, переписав, перенаправив, изменив заголовки запроса или ответа или ответив напрямую.

Промежуточное программное обеспечение запускается до сопоставления кэшированного содержимого и маршрутов. Дополнительные сведения см. в разделе «Сопоставление путей».
## [Use Cases](https://nextjs.org/docs/app/building-your-application/routing/middleware#use-cases)
Интеграция промежуточного программного обеспечения в ваше приложение может привести к значительному улучшению производительности, безопасности и удобства работы пользователей. Некоторые распространенные сценарии, в которых промежуточное программное обеспечение особенно эффективно, включают:
- Аутентификация и авторизация: обеспечьте идентификацию пользователя и проверьте файлы cookie сеанса, прежде чем предоставлять доступ к определенным страницам или маршрутам API.
- Перенаправления на стороне сервера: перенаправление пользователей на уровне сервера на основе определенных условий (например, языкового стандарта, роли пользователя).
- Переписывание путей: поддержка A/B-тестирования, развертывания функций или устаревших путей путем динамического перезаписи путей к маршрутам или страницам API на основе свойств запроса.
- Обнаружение ботов: защитите свои ресурсы, обнаруживая и блокируя трафик ботов.
- Ведение журнала и аналитика: собирайте и анализируйте данные запроса для получения ценной информации перед обработкой страницей или API.
- Пометка функций: динамически включайте или отключайте функции для плавного развертывания или тестирования функций.
Не менее важно распознавать ситуации, когда промежуточное программное обеспечение может оказаться неоптимальным. Вот несколько сценариев, о которых следует помнить:
- Сложная выборка и манипулирование данными: промежуточное программное обеспечение не предназначено для прямого извлечения или манипулирования данными, вместо этого это следует делать с помощью обработчиков маршрутов или утилит на стороне сервера.
- Тяжелые вычислительные задачи. Промежуточное программное обеспечение должно быть легким и быстро реагировать, иначе оно может вызвать задержки при загрузке страницы. Тяжелые вычислительные задачи или длительные процессы следует выполнять с помощью специальных обработчиков маршрутов.
- Расширенное управление сеансами. Хотя промежуточное программное обеспечение может управлять основными задачами сеанса, расширенное управление сеансами должно осуществляться с помощью специальных служб аутентификации или в обработчиках маршрутов.
- Прямые операции с базой данных: выполнение прямых операций с базой данных в промежуточном программном обеспечении не рекомендуется. Взаимодействие с базой данных должно осуществляться с помощью обработчиков маршрутов или серверных утилит.
## [Convention](https://nextjs.org/docs/app/building-your-application/routing/middleware#convention)
Используйте файл middleware.ts (или .js) в корне вашего проекта, чтобы определить промежуточное программное обеспечение. Например, на том же уровне, что и страницы или приложение, или внутри src, если применимо.
> [!note]
> Хотя для каждого проекта поддерживается только один файл middleware.ts, вы все равно можете организовать логику промежуточного программного обеспечения модульно. Выделите функциональные возможности промежуточного программного обеспечения в отдельные файлы .ts или .js и импортируйте их в основной файл middleware.ts. Это обеспечивает более чистое управление промежуточным программным обеспечением для конкретных маршрутов, объединенным в middleware.ts для централизованного управления. Применяя единый файл промежуточного программного обеспечения, он упрощает настройку, предотвращает потенциальные конфликты и оптимизирует производительность, избегая использования нескольких уровней промежуточного программного обеспечения.

## [Matching Paths](https://nextjs.org/docs/app/building-your-application/routing/middleware#matching-paths)
Промежуточное программное обеспечение будет вызываться для каждого маршрута в вашем проекте. Учитывая это, крайне важно использовать средства сопоставления для точного определения или исключения определенных маршрутов. Ниже приводится порядок выполнения:
1. `headers` from `next.config.js`
2.  `redirects` from `next.config.js`
3.  Middleware (`rewrites`, `redirects`, etc.)
4. `beforeFiles` (`rewrites`) from `next.config.js`
5. Filesystem routes (`public/`, `_next/static/`, `pages/`, `app/`, etc.)
6. `afterFiles` (`rewrites`) from `next.config.js`
7. Dynamic Routes (`/blog/[slug]`)
8. `fallback` (`rewrites`) from `next.config.js`

Существует два способа определить, по каким путям будет работать промежуточное ПО:
1.  [Custom matcher config](https://nextjs.org/docs/app/building-your-application/routing/middleware#matcher)
2. [Conditional statements](https://nextjs.org/docs/app/building-your-application/routing/middleware#conditional-statements)

> [!note] **Good to know**
> Значения сопоставителя должны быть константами, чтобы их можно было статически анализировать во время сборки. Динамические значения, такие как переменные, будут игнорироваться.

Configured matchers:
1. 1. MUST start with `/`
2. Can include named parameters: `/about/:path` matches `/about/a` and `/about/b`but not `/about/a/c`
3. Can have modifiers on named parameters (starting with `:`): `/about/:path*` matches `/about/a/b/c` because `*` is _zero or more_. `?` is _zero or one_ and `+` _one or more_
4. Can use regular expression enclosed in parenthesis: `/about/(.*)` is the same as `/about/:path*`
Read more details on [path-to-regexp](https://github.com/pillarjs/path-to-regexp#path-to-regexp-1) documentation.
> [!note] **Good to know**:
> В целях обратной совместимости Next.js всегда рассматривает /public как /public/index. Следовательно, будет соответствовать сопоставитель /public/:path.

## [NextResponse](https://nextjs.org/docs/app/building-your-application/routing/middleware#nextresponse)
The `NextResponse` API allows you to:
- `redirect` входящий запрос на другой URL
- `rewrite` ответ, отображая заданный URL-адрес
- Set request headers for API Routes, `getServerSideProps`, and `rewrite` destinations
- Set response cookies
- Set response headers

Чтобы получить ответ от Middleware, вы можете:
1. `rewrite` to a route ([Page](https://nextjs.org/docs/app/building-your-application/routing/pages-and-layouts) or [Route Handler](https://nextjs.org/docs/app/building-your-application/routing/route-handlers)) that produces a response
2. 1. return a `NextResponse` directly. See [Producing a Response](https://nextjs.org/docs/app/building-your-application/routing/middleware#producing-a-response)

## [Using Cookies](https://nextjs.org/docs/app/building-your-application/routing/middleware#using-cookies)
Файлы cookie представляют собой обычные заголовки. По запросу они сохраняются в заголовке Cookie. В ответе они находятся в заголовке Set-Cookie. Next.js предоставляет удобный способ доступа к этим файлам cookie и управления ими через расширение файлов cookie в NextRequest и NextResponse.
1. Для входящих запросов файлы cookie имеют следующие методы: get, getAll, set и delete cookie. Вы можете проверить наличие файла cookie с помощью команды has или удалить все файлы cookie с помощью команды Clear.
2. Для исходящих ответов файлы cookie имеют следующие методы get, getAll, set и delete.

### [`waitUntil` and `NextFetchEvent`](https://nextjs.org/docs/app/building-your-application/routing/middleware#waituntil-and-nextfetchevent)
The `NextFetchEvent` object extends the native [`FetchEvent`](https://developer.mozilla.org/docs/Web/API/FetchEvent) object, and includes the [`waitUntil()`](https://developer.mozilla.org/docs/Web/API/ExtendableEvent/waitUntil) metho
Метод waitUntil() принимает обещание в качестве аргумента и продлевает срок службы промежуточного программного обеспечения до тех пор, пока обещание не будет выполнено. Это полезно для выполнения работы в фоновом режиме.

## [Advanced Middleware Flags](https://nextjs.org/docs/app/building-your-application/routing/middleware#advanced-middleware-flags)
В версии 13.1 Next.js для промежуточного программного обеспечения были введены два дополнительных флага: SkipMiddlewareUrlNormalize и SkipTrailingSlashRedirect для обработки расширенных вариантов использования.
SkipTrailingSlashRedirect отключает перенаправления Next.js для добавления или удаления конечных косых черт. Это позволяет настраиваемой обработке внутри промежуточного программного обеспечения сохранять косую черту для некоторых путей, но не для других, что может упростить инкрементную миграцию.
![[Pasted image 20240524211827.png]]
![[Pasted image 20240524211838.png]]
SkipMiddlewareUrlNormalize позволяет отключить нормализацию URL-адресов в Next.js, чтобы сделать обработку прямых посещений и переходов клиентов одинаковой. В некоторых сложных случаях этот параметр обеспечивает полный контроль с использованием исходного URL-адреса.

## [Runtime](https://nextjs.org/docs/app/building-your-application/routing/middleware#runtime)
Middleware currently only supports the [Edge runtime](https://nextjs.org/docs/app/building-your-application/rendering/edge-and-nodejs-runtimes). The Node.js runtime can not be used.
