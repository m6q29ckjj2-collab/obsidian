Специальный файл loading.js поможет вам создать осмысленный пользовательский интерфейс загрузки с помощью React Suspense. Благодаря этому соглашению вы можете отображать мгновенное состояние загрузки с сервера во время загрузки содержимого сегмента маршрута. Новый контент автоматически заменяется после завершения рендеринга.
![[Pasted image 20240524115708.png]]
## [Instant Loading States](https://nextjs.org/docs/app/building-your-application/routing/loading-ui-and-streaming#instant-loading-states)
Состояние мгновенной загрузки — это резервный пользовательский интерфейс, который отображается сразу после навигации. Вы можете предварительно визуализировать индикаторы загрузки, такие как скелеты и счетчики, или небольшую, но значимую часть будущих экранов, например обложку, заголовок и т. д. Это помогает пользователям понять, как приложение реагирует, и обеспечивает лучший пользовательский опыт.
Create a loading state by adding a `loading.js` file inside a folder.
![[Pasted image 20240524115802.png]]
В той же папке файл loading.js будет вложен в файл Layout.js. Он автоматически обернет файл page.js и все дочерние элементы ниже границей \<Suspense>.
![[Pasted image 20240524115842.png]]
> Good to know
> - Навигация осуществляется мгновенно, даже при маршрутизации, ориентированной на сервер.
> - Навигация является прерываемой, то есть для изменения маршрута не нужно ждать полной загрузки содержимого маршрута, прежде чем перейти к другому маршруту.
> - Общие макеты остаются интерактивными, пока загружаются новые сегменты маршрута.

 >Recommendation 
 >Используйте соглашение loading.js для сегментов маршрута (макетов и страниц), поскольку Next.js оптимизирует эту функциональность.

## [Streaming with Suspense](https://nextjs.org/docs/app/building-your-application/routing/loading-ui-and-streaming#streaming-with-suspense)
Помимо loading.js, вы также можете вручную создавать границы приостановки для своих собственных компонентов пользовательского интерфейса. App Router поддерживает потоковую передачу с помощью Suspense для сред выполнения Node.js и Edge.
> Good to know
> Некоторые браузеры буферизуют потоковый ответ. Вы можете не увидеть потоковый ответ, пока его размер не превысит 1024 байта. Обычно это влияет только на приложения «Hello World», но не на реальные приложения.

### [What is Streaming?](https://nextjs.org/docs/app/building-your-application/routing/loading-ui-and-streaming#what-is-streaming)
Чтобы узнать, как работает потоковая передача в React и Next.js, полезно понять серверный рендеринг (SSR) и его ограничения.
При использовании SSR необходимо выполнить ряд шагов, прежде чем пользователь сможет видеть страницу и взаимодействовать с ней:
1. Сначала все данные для данной страницы загружаются на сервер.
2. Затем сервер отображает HTML-код страницы.
3. HTML, CSS и JavaScript страницы отправляются клиенту.
4. Неинтерактивный пользовательский интерфейс отображается с использованием сгенерированного HTML и CSS.
5. Наконец, React [hydrates](https://react.dev/reference/react-dom/client/hydrateRoot#hydrating-server-rendered-html)  пользовательский интерфейс, делая его интерактивным.
![[Pasted image 20240524131641.png]]
Эти шаги являются последовательными и блокирующими, что означает, что сервер может отображать HTML-код страницы только после получения всех данных. А на клиенте React может hydrate пользовательский интерфейс только после загрузки кода для всех компонентов на странице.

SSR с React и Next.js помогает улучшить воспринимаемую производительность загрузки, показывая пользователю неинтерактивную страницу как можно скорее.
![[Pasted image 20240524131901.png]]
Однако это все равно может быть медленным, поскольку необходимо завершить получение всех данных на сервере, прежде чем страница сможет быть показана пользователю.
*Streaming* позволяет разбить HTML-код страницы на более мелкие фрагменты и постепенно отправлять эти фрагменты с сервера клиенту.
![[Pasted image 20240524132034.png]]
Это позволяет отображать части страницы быстрее, не дожидаясь загрузки всех данных, прежде чем можно будет отобразить какой-либо пользовательский интерфейс.
Потоковая передача хорошо работает с компонентной моделью React, поскольку каждый компонент можно рассматривать как фрагмент. Компоненты, которые имеют более высокий приоритет (например, информация о продукте) или которые не полагаются на данные, могут быть отправлены первыми (например, макет), и React может начать гидратацию раньше. Компоненты с более низким приоритетом (например, обзоры, сопутствующие товары) могут быть отправлены в том же запросе к серверу после получения их данных.
![[Pasted image 20240524132215.png]]
Потоковая передача особенно полезна, когда вы хотите предотвратить блокировку отрисовки страницы длинными запросами данных, поскольку она может сократить время до [Time To First Byte (TTFB)](https://web.dev/ttfb/) и [First Contentful Paint (FCP)](https://web.dev/first-contentful-paint/). Это также помогает улучшить  [Time to Interactive (TTI)](https://developer.chrome.com/en/docs/lighthouse/performance/interactive/), особенно на медленных устройствах.

### [Example](https://nextjs.org/docs/app/building-your-application/routing/loading-ui-and-streaming#example)
\<Suspense> работает путем оборачивания компонента, выполняющего асинхронное действие (например, выборку данных), отображения резервного пользовательского интерфейса (например, скелета, счетчика) во время его выполнения, а затем замены компонента после завершения действия.
![[Pasted image 20240524132457.png]]
Используя Suspense, вы получаете следующие преимущества:
- Потоковое рендеринг на сервере. Постепенный рендеринг HTML от сервера к клиенту.
- Выборочная гидратация. React определяет приоритетность того, какие компоненты сделать интерактивными в первую очередь, в зависимости от взаимодействия с пользователем.
Дополнительные примеры и варианты использования Suspense см. в документации React.

### [SEO](https://nextjs.org/docs/app/building-your-application/routing/loading-ui-and-streaming#seo)
Next.js будет ждать завершения извлечения данных внутри generateMetadata перед потоковой передачей пользовательского интерфейса клиенту. Это гарантирует, что первая часть потокового ответа будет содержать теги \<head>.

Поскольку потоковая передача отображается на сервере, она не влияет на SEO. Вы можете использовать инструмент проверки расширенных результатов от Google [Rich Results Test](https://search.google.com/test/rich-results), чтобы увидеть, как ваша страница отображается для веб-сканеров Google, и просмотреть сериализованный HTML ([source](https://web.dev/rendering-on-the-web/#seo-considerations))

### [Status Codes](https://nextjs.org/docs/app/building-your-application/routing/loading-ui-and-streaming#status-codes)
При потоковой передаче будет возвращен код состояния 200, сигнализирующий об успешном выполнении запроса.
Сервер по-прежнему может сообщать клиенту об ошибках или проблемах в самом потоковом контенте, например, при использовании перенаправления или notFound. Поскольку заголовки ответа уже отправлены клиенту, код состояния ответа не может быть обновлен. Это не влияет на SEO.
