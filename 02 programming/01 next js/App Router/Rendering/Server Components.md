Серверные компоненты React позволяют вам писать пользовательский интерфейс, который можно визуализировать и при необходимости кэшировать на сервере. В Next.js работа рендеринга далее разделяется по сегментам маршрута, чтобы обеспечить потоковую и частичную рендеринг, и существует три различные стратегии рендеринга на сервере:

- [Static Rendering](https://nextjs.org/docs/app/building-your-application/rendering/server-components#static-rendering-default)
- [Dynamic Rendering](https://nextjs.org/docs/app/building-your-application/rendering/server-components#dynamic-rendering)
- [Streaming](https://nextjs.org/docs/app/building-your-application/rendering/server-components#streaming)

На этой странице рассказывается о том, как работают серверные компоненты, когда вы можете их использовать, а также о различных стратегиях рендеринга сервера.
## [Benefits of Server Rendering](https://nextjs.org/docs/app/building-your-application/rendering/server-components#benefits-of-server-rendering)
There are a couple of benefits to doing the rendering work on the server, including:

- Получение данных: серверные компоненты позволяют перенести получение данных на сервер, ближе к источнику данных. Это может повысить производительность за счет сокращения времени, необходимого для получения данных, необходимых для рендеринга, и количества запросов, которые должен сделать клиент.

- Безопасность: серверные компоненты позволяют хранить на сервере конфиденциальные данные и логику, например токены и ключи API, без риска раскрытия их клиенту.

- Кэширование: при рендеринге на сервере результат можно кэшировать и повторно использовать при последующих запросах и среди пользователей. Это может повысить производительность и снизить затраты за счет уменьшения объема рендеринга и выборки данных, выполняемых при каждом запросе.

- Производительность. Серверные компоненты предоставляют дополнительные инструменты для оптимизации производительности по сравнению с базовым уровнем. Например, если вы начинаете с приложения, полностью состоящего из клиентских компонентов, перемещение неинтерактивных частей вашего пользовательского интерфейса в серверные компоненты может уменьшить объем необходимого клиентского JavaScript. Это полезно для пользователей с медленным Интернетом или менее мощными устройствами, поскольку в браузере меньше клиентского JavaScript для загрузки, анализа и выполнения.

- Первоначальная загрузка страницы и первая отрисовка контента (FCP). На сервере мы можем сгенерировать HTML, чтобы пользователи могли немедленно просмотреть страницу, не дожидаясь, пока клиент загрузит, проанализирует и выполнит JavaScript, необходимый для рендеринга страницы.
- Поисковая оптимизация и возможность совместного использования в социальных сетях. Отрисованный HTML-код может использоваться ботами поисковых систем для индексации ваших страниц, а ботами социальных сетей — для создания предварительного просмотра социальных карточек для ваших страниц.
- Потоковая передача: серверные компоненты позволяют разделить работу по рендерингу на части и передавать их клиенту по мере их готовности. Это позволяет пользователю видеть части страницы раньше, не дожидаясь, пока вся страница будет отображена на сервере.

## [Using Server Components in Next.js](https://nextjs.org/docs/app/building-your-application/rendering/server-components#using-server-components-in-nextjs)
По умолчанию Next.js использует серверные компоненты. Это позволяет автоматически реализовывать серверный рендеринг без дополнительной настройки, и при необходимости вы можете выбрать использование клиентских компонентов, см. «Клиентские компоненты».

## [How are Server Components rendered?](https://nextjs.org/docs/app/building-your-application/rendering/server-components#how-are-server-components-rendered)
На сервере Next.js использует API React для управления рендерингом. Работа рендеринга разбивается на куски: по отдельным сегментам маршрута и границам ожидания.

Каждый фрагмент рендерится в два этапа:
- React преобразует серверные компоненты в специальный формат данных, называемый полезной нагрузкой серверного компонента React (RSC Payload).
- Next.js использует инструкции JavaScript для полезной нагрузки RSC и клиентского компонента для рендеринга HTML на сервере.
Then, on the client:
- HTML используется для немедленного отображения быстрого неинтерактивного предварительного просмотра маршрута — это только для начальной загрузки страницы.
- Полезная нагрузка React Server Components используется для согласования деревьев клиентских и серверных компонентов и обновления DOM.
- Инструкции JavaScript используются для гидратации клиентских компонентов и делают приложение интерактивным.
> [!info] 
> **What is the React Server Component Payload (RSC)?**
> Полезная нагрузка RSC — это компактное двоичное представление визуализированного дерева компонентов сервера React. Он используется React на клиенте для обновления DOM браузера. Полезная нагрузка RSC содержит:
> - Визуализированный результат серверных компонентов
> - Заполнители для отображения клиентских компонентов и ссылки на их файлы JavaScript.
> - Любые реквизиты, передаваемые из серверного компонента в клиентский компонент. 

## [Server Rendering Strategies](https://nextjs.org/docs/app/building-your-application/rendering/server-components#server-rendering-strategies)
Существует три подмножества серверного рендеринга: статический, динамический и потоковый.
### [Static Rendering (Default)](https://nextjs.org/docs/app/building-your-application/rendering/server-components#static-rendering-default)
При статическом рендеринге маршруты визуализируются во время сборки или в фоновом режиме после повторной проверки данных. Результат кэшируется и может быть отправлен в [Content Delivery Network (CDN)](https://developer.mozilla.org/docs/Glossary/CDN). Такая оптимизация позволяет разделить результат работы рендеринга между пользователями и запросами сервера.

Статическая отрисовка полезна, когда маршрут содержит данные, которые не персонализированы для пользователя и могут быть известны во время сборки, например статическая запись в блоге или страница продукта.

### [Dynamic Rendering](https://nextjs.org/docs/app/building-your-application/rendering/server-components#dynamic-rendering)
При динамическом рендеринге маршруты отображаются для каждого пользователя во время запроса.

Динамическая отрисовка полезна, когда маршрут содержит данные, персонализированные для пользователя, или информацию, которая может быть известна только во время запроса, например файлы cookie или параметры поиска URL-адреса.

> [!info] 
> **Dynamic Routes with Cached Data**
>  На большинстве веб-сайтов маршруты не являются полностью статическими или полностью динамическими — это спектр. Например, у вас может быть страница электронной коммерции, которая использует кэшированные данные о продуктах, которые периодически проверяются, но также содержит некэшированные персонализированные данные о клиентах.
>  В Next.js вы можете динамически отображать маршруты, содержащие как кэшированные, так и некэшированные данные. Это связано с тем, что полезная нагрузка и данные RSC кэшируются отдельно. Это позволяет вам выбрать динамический рендеринг, не беспокоясь о влиянии на производительность получения всех данных во время запроса.
>  Learn more about the [full-route cache](https://nextjs.org/docs/app/building-your-application/caching#full-route-cache) and [Data Cache](https://nextjs.org/docs/app/building-your-application/caching#data-cache).

#### [Switching to Dynamic Rendering](https://nextjs.org/docs/app/building-your-application/rendering/server-components#switching-to-dynamic-rendering)
Если во время рендеринга будет обнаружена динамическая функция или запрос некэшированных данных, Next.js переключится на динамический рендеринг всего маршрута. В этой таблице показано, как динамические функции и кэширование данных влияют на то, отображается ли маршрут статически или динамически:

В таблице выше, чтобы маршрут был полностью статическим, все данные должны быть кэшированы. Однако вы можете использовать динамически отображаемый маршрут, который использует как кэшированные, так и некэшированные выборки данных.

Вам как разработчику не нужно выбирать между статическим и динамическим рендерингом, поскольку Next.js автоматически выберет лучшую стратегию рендеринга для каждого маршрута на основе используемых функций и API. Вместо этого вы сами выбираете, когда кэшировать или повторно проверять [cache or revalidate specific data](https://nextjs.org/docs/app/building-your-application/data-fetching/fetching-caching-and-revalidating), а также можете выбрать [stream](https://nextjs.org/docs/app/building-your-application/rendering/server-components#streaming) передачу частей вашего пользовательского интерфейса.
#### [Dynamic Functions](https://nextjs.org/docs/app/building-your-application/rendering/server-components#dynamic-functions)
Динамические функции полагаются на информацию, которая может быть известна только во время запроса, например файлы cookie пользователя, текущие заголовки запросов или параметры поиска URL-адреса. В Next.js такими динамическими функциями являются:

- **[`cookies()`](https://nextjs.org/docs/app/api-reference/functions/cookies) and [`headers()`](https://nextjs.org/docs/app/api-reference/functions/headers)**: Использование их в серверном компоненте приведет к динамическому рендерингу всего маршрута во время запроса.
- **[`searchParams`](https://nextjs.org/docs/app/api-reference/file-conventions/page#searchparams-optional)**: Использование свойства searchParams на странице приведет к динамическому рендерингу страницы во время запроса.
Использование любой из этих функций приведет к динамическому рендерингу всего маршрута во время запроса.

### [Streaming](https://nextjs.org/docs/app/building-your-application/rendering/server-components#streaming)
![[Pasted image 20240711155610.png]]
Потоковая передача позволяет постепенно отображать пользовательский интерфейс с сервера. Работа разбивается на части и передается клиенту по мере готовности. Это позволяет пользователю сразу видеть части страницы, прежде чем весь контент завершится.

![[Pasted image 20240711155649.png]]
Потоковая передача встроена в маршрутизатор приложений Next.js по умолчанию. Это помогает улучшить как производительность начальной загрузки страницы, так и пользовательский интерфейс, который зависит от более медленной выборки данных, которая может блокировать отрисовку всего маршрута. Например, отзывы на странице товара.

Вы можете начать потоковую передачу сегментов маршрута, используя loading.js и компоненты пользовательского интерфейса с помощью  [React Suspense](https://nextjs.org/docs/app/building-your-application/routing/loading-ui-and-streaming) Дополнительную информацию см. в разделе «[Loading UI and Streaming](https://nextjs.org/docs/app/building-your-application/routing/loading-ui-and-streaming)».
