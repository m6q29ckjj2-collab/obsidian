При создании приложений React вам необходимо учитывать, какие части вашего приложения должны отображаться на сервере или на клиенте. На этой странице описаны некоторые рекомендуемые шаблоны композиции при использовании серверных и клиентских компонентов.

## [Server Component Patterns](https://nextjs.org/docs/app/building-your-application/rendering/composition-patterns#server-component-patterns)
Прежде чем выбрать рендеринг на стороне клиента, вы можете выполнить некоторую работу на сервере, например получить данные или получить доступ к вашей базе данных или серверным службам.

Вот некоторые распространенные шаблоны при работе с серверными компонентами:
### [Sharing data between components](https://nextjs.org/docs/app/building-your-application/rendering/composition-patterns#sharing-data-between-components)
При получении данных на сервере могут возникнуть случаи, когда вам потребуется разделить данные между разными компонентами. Например, у вас может быть макет и страница, которые зависят от одних и тех же данных.

Вместо использования контекста React (который недоступен на сервере) или передачи данных в качестве реквизита вы можете использовать функцию fetch или кэша React для извлечения одних и тех же данных в компонентах, которым они нужны, не беспокоясь о дублировании запросов для одних и тех же данных. . Это связано с тем, что React расширяет возможности выборки для автоматического запоминания запросов данных, а функцию кэширования можно использовать, когда выборка недоступна.

Узнайте больше о  [memoization](https://nextjs.org/docs/app/building-your-application/caching#request-memoization) в React.
### [Keeping Server-only Code out of the Client Environment](https://nextjs.org/docs/app/building-your-application/rendering/composition-patterns#keeping-server-only-code-out-of-the-client-environment)
Поскольку модули JavaScript могут использоваться совместно модулями серверных и клиентских компонентов, код, который когда-либо предназначался для запуска только на сервере, может проникнуть в клиент.

Например, возьмем следующую функцию выборки данных:
![[Pasted image 20240711191719.png]]
На первый взгляд кажется, что getData работает как на сервере, так и на клиенте. Однако эта функция содержит API_KEY, написанный с намерением, чтобы она выполнялась только на сервере.

Поскольку переменная среды API_KEY не имеет префикса NEXT_PUBLIC, это частная переменная, доступ к которой возможен только на сервере. Чтобы предотвратить утечку переменных среды клиенту, Next.js заменяет частные переменные среды пустой строкой.

В результате, хотя метод getData() можно импортировать и выполнить на клиенте, он не будет работать должным образом. И хотя если сделать переменную общедоступной, функция будет работать на клиенте, но вы, возможно, не захотите раскрывать клиенту конфиденциальную информацию.

Чтобы предотвратить такого рода непреднамеренное использование клиентом серверного кода, мы можем использовать пакет только для сервера, чтобы выдавать другим разработчикам ошибку во время сборки, если они когда-либо случайно импортируют один из этих модулей в клиентский компонент.

Соответствующий пакет client-only можно использовать для обозначения модулей, содержащих код только для клиента — например, код, который обращается к объекту окна.

### [Using Third-party Packages and Providers](https://nextjs.org/docs/app/building-your-application/rendering/composition-patterns#using-third-party-packages-and-providers)
Поскольку серверные компоненты — это новая функция React, сторонние пакеты и поставщики в экосистеме только начинают добавлять директиву «use client» к компонентам, которые используют функции только для клиента, такие как useState, useEffect и createContext.

Сегодня многие компоненты из пакетов npm, использующие только клиентские функции, еще не имеют этой директивы. Эти сторонние компоненты будут работать должным образом в клиентских компонентах, поскольку у них есть директива «использовать клиент», но они не будут работать в серверных компонентах.

Например, предположим, что вы установили гипотетический пакет acme-carousel, который имеет компонент <Carousel />. Этот компонент использует useState, но у него еще нет директивы use client.

## [Client Components](https://nextjs.org/docs/app/building-your-application/rendering/composition-patterns#client-components)
### [Moving Client Components Down the Tree](https://nextjs.org/docs/app/building-your-application/rendering/composition-patterns#moving-client-components-down-the-tree)
Чтобы уменьшить размер пакета клиентского JavaScript, мы рекомендуем переместить клиентские компоненты вниз по дереву компонентов.

Например, у вас может быть макет со статическими элементами (например, логотипом, ссылками и т. д.) и интерактивной панелью поиска, использующей состояние.

Вместо того, чтобы делать весь макет клиентским компонентом, переместите интерактивную логику в клиентский компонент (например, <SearchBar />) и сохраните свой макет как серверный компонент. Это означает, что вам не нужно отправлять клиенту весь Javascript-компонент макета.
![[Pasted image 20240711193129.png]]
### [Passing props from Server to Client Components (Serialization)](https://nextjs.org/docs/app/building-your-application/rendering/composition-patterns#passing-props-from-server-to-client-components-serialization)
Если вы извлекаете данные из серверного компонента, вы можете передать данные в качестве реквизита клиентским компонентам. Реквизиты, передаваемые с сервера клиентским компонентам, должны быть  [serializable](https://react.dev/reference/react/use-server#serializable-parameters-and-return-values) с помощью React.

Если ваши клиентские компоненты зависят от данных, которые не подлежат сериализации, вы [fetch data on the client with a third party library](https://nextjs.org/docs/app/building-your-application/data-fetching/fetching-caching-and-revalidating#fetching-data-on-the-client-with-third-party-libraries)и или на сервере через  [Route Handler](https://nextjs.org/docs/app/building-your-application/routing/route-handlers).
## [Interleaving Server and Client Components](https://nextjs.org/docs/app/building-your-application/rendering/composition-patterns#interleaving-server-and-client-components)
При чередовании клиентских и серверных компонентов может быть полезно визуализировать ваш пользовательский интерфейс в виде дерева компонентов. Начиная с корневого макета, который представляет собой серверный компонент, вы можете затем отображать определенные поддеревья компонентов на клиенте, добавляя директиву «use client».

В эти клиентские поддеревья вы по-прежнему можете вкладывать серверные компоненты или вызывать действия сервера, однако следует помнить о некоторых вещах:

- В течение жизненного цикла запрос-ответ ваш код перемещается с сервера на клиент. Если вам нужно получить доступ к данным или ресурсам на сервере, находясь на клиенте, вы будете делать новый запрос к серверу, а не переключаться туда и обратно.
- Когда к серверу поступает новый запрос, сначала отрисовываются все серверные компоненты, включая те, которые вложены в клиентские компоненты. Визуализированный результат (полезная нагрузка RSC) будет содержать ссылки на расположение клиентских компонентов. Затем на клиенте React использует полезную нагрузку RSC для согласования серверных и клиентских компонентов в одном дереве.
- Поскольку клиентские компоненты отображаются после серверных компонентов, вы не можете импортировать серверный компонент в модуль клиентского компонента (поскольку для этого потребуется новый запрос обратно на сервер). Вместо этого вы можете передать серверный компонент в качестве реквизита клиентскому компоненту. См. разделы[unsupported pattern](https://nextjs.org/docs/app/building-your-application/rendering/composition-patterns#unsupported-pattern-importing-server-components-into-client-components) and [supported pattern](https://nextjs.org/docs/app/building-your-application/rendering/composition-patterns#supported-pattern-passing-server-components-to-client-components-as-props) ниже.
