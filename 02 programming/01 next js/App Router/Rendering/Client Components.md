Клиентские компоненты позволяют вам писать интерактивный пользовательский интерфейс, который  [prerendered on the server](https://github.com/reactwg/server-components/discussions/4)  и может использовать клиентский JavaScript для запуска в браузере.

На этой странице будет описано, как работают клиентские компоненты, как они отображаются и когда их можно использовать.

## [Benefits of Client Rendering](https://nextjs.org/docs/app/building-your-application/rendering/client-components#benefits-of-client-rendering)
Выполнение рендеринга на клиенте имеет несколько преимуществ, в том числе:

- Интерактивность: клиентские компоненты могут использовать прослушиватели состояний, эффектов и событий, то есть они могут предоставлять немедленную обратную связь пользователю и обновлять пользовательский интерфейс.
- API-интерфейсы браузера. Клиентские компоненты имеют доступ к API-интерфейсам браузера, таким как [geolocation](https://developer.mozilla.org/docs/Web/API/Geolocation_API) or [localStorage](https://developer.mozilla.org/docs/Web/API/Window/localStorage).

## [Using Client Components in Next.js](https://nextjs.org/docs/app/building-your-application/rendering/client-components#using-client-components-in-nextjs)
Чтобы использовать клиентские компоненты, вы можете добавить директиву React [`"use client"` directive](https://react.dev/reference/react/use-client), над импортируемыми данными.

«использовать клиент» используется для объявления  [boundary](https://nextjs.org/docs/app/building-your-application/rendering#network-boundary) между модулями серверного и клиентского компонентов. Это означает, что при определении «использования клиента» в файле все остальные модули, импортированные в него, включая дочерние компоненты, считаются частью клиентского пакета.

На диаграмме ниже показано, что использование onClick и useState во вложенном компоненте (toggle.js) приведет к ошибке, если директива use client не определена. Это связано с тем, что по умолчанию все компоненты в App Router являются серверными компонентами, для которых эти API недоступны. > [!note] Определение точек входа для многократного использования клиента:
> Вы можете определить несколько точек входа «использовать клиент» в дереве компонентов React. Это позволяет разделить ваше приложение на несколько клиентских пакетов.
> Однако «использовать клиент» не обязательно определять в каждом компоненте, который необходимо отобразить на клиенте. После определения границы все дочерние компоненты и модули, импортированные в нее, считаются частью клиентского пакета. 
## [How are Client Components Rendered?](https://nextjs.org/docs/app/building-your-application/rendering/client-components#how-are-client-components-rendered)
В Next.js клиентские компоненты отображаются по-разному в зависимости от того, является ли запрос частью полной загрузки страницы (первоначальное посещение вашего приложения или перезагрузка страницы, вызванной обновлением браузера) или последующей навигации.
### [Full page load](https://nextjs.org/docs/app/building-your-application/rendering/client-components#full-page-load)
Чтобы оптимизировать начальную загрузку страницы, Next.js будет использовать API-интерфейсы React для рендеринга статического предварительного просмотра HTML на сервере как для клиентских, так и для серверных компонентов. Это означает, что когда пользователь впервые посещает ваше приложение, он сразу увидит содержимое страницы, не дожидаясь, пока клиент загрузит, проанализирует и выполнит пакет клиентского компонента JavaScript.

On the server:

1. React преобразует серверные компоненты в специальный формат данных, называемый полезной нагрузкой серверного компонента React (RSC Payload), который включает ссылки на клиентские компоненты.
2. Next.js использует инструкции JavaScript для полезной нагрузки RSC и клиентского компонента для визуализации HTML для маршрута на сервере.
Then, on the client:
1. HTML используется для немедленного отображения быстрого неинтерактивного первоначального предварительного просмотра маршрута.
2. Полезная нагрузка React Server Components используется для согласования деревьев клиентских и серверных компонентов и обновления DOM.
3. Инструкции JavaScript используются для гидратации клиентских компонентов и обеспечения интерактивности их пользовательского интерфейса.
> [!hint] **What is hydration?**
> Гидратация — это процесс подключения прослушивателей событий к DOM, чтобы сделать статический HTML интерактивным. За кулисами гидратация осуществляется с помощью API [`hydrateRoot`](https://react.dev/reference/react-dom/client/hydrateRoot) React.
>

### [Subsequent Navigations](https://nextjs.org/docs/app/building-your-application/rendering/client-components#subsequent-navigations)
При последующих переходах клиентские компоненты полностью отображаются на клиенте, без HTML-кода, отображаемого на сервере.

Это означает, что пакет клиентского компонента JavaScript загружается и анализируется. Как только пакет будет готов, React будет использовать полезную нагрузку RSC для согласования деревьев клиентских и серверных компонентов и обновления DOM.
## [Going back to the Server Environment](https://nextjs.org/docs/app/building-your-application/rendering/client-components#going-back-to-the-server-environment)
Иногда после объявления границы «использовать клиент» вам может потребоваться вернуться в серверную среду. Например, вы можете уменьшить размер клиентского пакета, получить данные на сервере или использовать API, доступный только на сервере.

Вы можете хранить код на сервере, даже если он теоретически вложен в клиентские компоненты, чередуя клиентские и серверные компоненты и [Server Actions](https://nextjs.org/docs/app/building-your-application/data-fetching/server-actions-and-mutations).. Дополнительную информацию см. на странице « [Composition Patterns](https://nextjs.org/docs/app/building-your-application/rendering/composition-patterns)».
