Next.js повышает производительность вашего приложения и снижает затраты за счет кэширования работы рендеринга и запросов данных. На этой странице подробно рассматриваются механизмы кэширования Next.js, API-интерфейсы, которые можно использовать для их настройки, и то, как они взаимодействуют друг с другом.

> [!note] 
> Полезно знать: эта страница поможет вам понять, как работает Next.js, но не является необходимым знанием для продуктивной работы с Next.js. Большая часть эвристики кэширования Next.js определяется использованием вашего API и имеет значения по умолчанию для наилучшей производительности при нулевой или минимальной настройке.
> 

## [Overview](https://nextjs.org/docs/app/building-your-application/caching#overview)
Вот общий обзор различных механизмов кэширования и их назначения:
![[Pasted image 20240711160157.png]]
По умолчанию Next.js кэширует как можно больше данных для повышения производительности и снижения затрат. Это означает, что маршруты отображаются статически, а запросы данных кэшируются, если вы не откажетесь. На диаграмме ниже показано поведение кэширования по умолчанию: когда маршрут статически отображается во время сборки и когда статический маршрут посещается впервые.

![[Pasted image 20240711160411.png]]Поведение кэширования меняется в зависимости от того, отображается ли маршрут статически или динамически, кэшируются или не кэшируются данные, а также является ли запрос частью первоначального посещения или последующей навигации. В зависимости от вашего варианта использования вы можете настроить поведение кэширования для отдельных маршрутов и запросов данных.
## [Request Memoization](https://nextjs.org/docs/app/building-your-application/caching#request-memoization)
React расширяет API выборки для автоматического запоминания запросов, имеющих одинаковый URL-адрес и параметры. Это означает, что вы можете вызвать функцию выборки для одних и тех же данных в нескольких местах дерева компонентов React, выполнив ее только один раз.

![[Pasted image 20240711162427.png]]
Например, если вам нужно использовать одни и те же данные по маршруту (например, в макете, странице и нескольких компонентах), вам не нужно извлекать данные в верхней части дерева и пересылать реквизиты между компонентами. Вместо этого вы можете получать данные в компонентах, которым они нужны, не беспокоясь о последствиях для производительности, связанных с выполнением нескольких запросов по сети для одних и тех же данных.

![[Pasted image 20240711162601.png]]
**How Request Memoization Works**
![[Pasted image 20240711162634.png]]
- При рендеринге маршрута при первом вызове определенного запроса его результат не будет находиться в памяти и будет MISS в кеше.
- Таким образом, функция будет выполнена, данные будут получены из внешнего источника, а результат сохранится в памяти.
- Последующие вызовы функции запроса в том же проходе рендеринга будут HIT кэша, и данные будут возвращены из памяти без выполнения функции.
- После визуализации маршрута и завершения этапа рендеринга память «сбрасывается», и все записи запоминания запросов очищаются.

> [!note] Good to know
>
>- Запоминание запроса — это функция React, а не функция Next.js. Он включен сюда, чтобы показать, как он взаимодействует с другими механизмами кэширования.
>-  Мемоизация применяется только к методу GET в запросах на выборку.
>-  Мемоизация применяется только к дереву компонентов React, это означает:
>	- Это применимо к запросам на выборку в GeneMetadata, GenerateStaticParams, Layouts, Pages и других компонентах сервера.
>	- Это не относится к запросам на выборку в обработчиках маршрутов, поскольку они не являются частью дерева компонентов React.
>- В случаях, когда fetch не подходит (например, некоторые клиенты баз данных, клиенты CMS или клиенты GraphQL), вы можете использовать [React `cache` function](https://nextjs.org/docs/app/building-your-application/caching#react-cache-function) для запоминания функций. 
### [Duration](https://nextjs.org/docs/app/building-your-application/caching#duration)
Кэш сохраняется в течение всего времени существования запроса к серверу, пока дерево компонентов React не завершит рендеринг.

### [Revalidating](https://nextjs.org/docs/app/building-your-application/caching#revalidating)
Поскольку мемоизация не передается по запросам сервера и применяется только во время рендеринга, нет необходимости проводить ее повторную проверку.

### [Opting out](https://nextjs.org/docs/app/building-your-application/caching#opting-out)
Мемоизация применяется только к методу GET в запросах на выборку, другие методы, такие как POST и DELETE, не запоминаются. Это поведение по умолчанию является оптимизацией React, и мы не рекомендуем отказываться от него.
Для управления отдельными запросами вы можете использовать свойство  [`signal`](https://developer.mozilla.org/en-US/docs/Web/API/AbortController/signal) из [`AbortController`](https://developer.mozilla.org/en-US/docs/Web/API/AbortController). Однако это не приведет к исключению запросов из запоминания, а скорее прервет текущие запросы.

## [Data Cache](https://nextjs.org/docs/app/building-your-application/caching#data-cache)
Next.js имеет встроенный кэш данных, который сохраняет результаты выборки данных во входящих запросах сервера и развертываниях. Это возможно, поскольку Next.js расширяет собственный API выборки, позволяя каждому запросу на сервере устанавливать собственную семантику постоянного кэширования.

> [!note] Good to know
> 
>  В браузере параметр кэша fetch указывает, как запрос будет взаимодействовать с кэшем HTTP браузера, в Next.js параметр кэша указывает, как запрос на стороне сервера будет взаимодействовать с кэшем данных сервера.

По умолчанию запросы данных, использующие выборку, кэшируются. Вы можете использовать параметры выборки [`cache`](https://nextjs.org/docs/app/building-your-application/caching#fetch-optionscache) и [`next.revalidate`](https://nextjs.org/docs/app/building-your-application/caching#fetch-optionsnextrevalidate) для настройки поведения кэширования.

**How the Data Cache Works**
![[Pasted image 20240711163845.png]]
- При первом вызове запроса на выборку во время рендеринга Next.js проверяет кэш данных на наличие кэшированного ответа.
- Если найден кэшированный ответ, он немедленно возвращается и запоминается.
- Если кэшированный ответ не найден, выполняется запрос к источнику данных, результат сохраняется в кэше данных и запоминается.
- Для некэшированных данных (например, {cache: 'no-store'}) результат всегда извлекается из источника данных и запоминается.
- Независимо от того, кэшируются ли данные или нет, запросы всегда запоминаются, чтобы избежать дублирования запросов на одни и те же данные во время прохода рендеринга React.

> [!important] **Differences between the Data Cache and Request Memoization**
> Хотя оба механизма кэширования помогают повысить производительность за счет повторного использования кэшированных данных, кэш данных сохраняется во всех входящих запросах и развертываниях, тогда как мемоизация сохраняется только на протяжении всего времени существования запроса.
> 
> С помощью мемоизации мы уменьшаем количество повторяющихся запросов в одном проходе рендеринга, которые должны пересекать границу сети от сервера рендеринга к серверу кэша данных (например, CDN или Edge Network) или источнику данных (например, базе данных или CMS). С помощью Data Cache мы уменьшаем количество запросов к нашему исходному источнику данных.

### [Duration](https://nextjs.org/docs/app/building-your-application/caching#duration-1)
Кэш данных сохраняется во всех входящих запросах и развертываниях, пока вы не выполните повторную проверку или не откажетесь от нее.
### [Revalidating](https://nextjs.org/docs/app/building-your-application/caching#revalidating-1)
Кэшированные данные можно повторно проверить двумя способами:

- Повторная проверка по времени. Повторная проверка данных по прошествии определенного времени и выполнении нового запроса. Это полезно для данных, которые изменяются нечасто, а свежесть не так важна.
- Повторная проверка по требованию. Повторная проверка данных на основе события (например, отправки формы). При повторной проверке по требованию можно использовать подход на основе тегов или путей для одновременной повторной проверки групп данных. Это полезно, если вы хотите, чтобы самые последние данные отображались как можно скорее (например, при обновлении контента из вашей headless CMS).

#### [Time-based Revalidation](https://nextjs.org/docs/app/building-your-application/caching#time-based-revalidation)
Чтобы повторно проверять данные через определенный интервал времени, вы можете использовать опцию fetch next.revalidate, чтобы установить время жизни кэша ресурса (в секундах).

![[Pasted image 20240711164400.png]]
Кроме того, вы можете использовать параметры конфигурации  [Route Segment Config options](https://nextjs.org/docs/app/building-your-application/caching#segment-config-options) для настройки всех запросов на выборку в сегменте или в тех случаях, когда вы не можете использовать выборку.

![[Pasted image 20240711164545.png]]
> [!note] 
> Кэш данных в настоящее время доступен только на страницах/маршрутах, а не на промежуточном программном обеспечении. Любые выборки, выполняемые внутри вашего промежуточного программного обеспечения, по умолчанию не кэшируются.

## [Full Route Cache](https://nextjs.org/docs/app/building-your-application/caching#full-route-cache)
**Related terms**:
Вы можете увидеть, что термины «Автоматическая статическая оптимизация», «Генерация статического сайта» или «Статический рендеринг» используются как взаимозаменяемые для обозначения процесса рендеринга и кэширования маршрутов вашего приложения во время сборки.

Next.js автоматически отображает и кэширует маршруты во время сборки. Это оптимизация, которая позволяет вам обслуживать кэшированный маршрут вместо рендеринга на сервере для каждого запроса, что приводит к более быстрой загрузке страниц.

Чтобы понять, как работает Full Route Cache, полезно посмотреть, как React обрабатывает рендеринг и как Next.js кэширует результат:

### [1. React Rendering on the Server](https://nextjs.org/docs/app/building-your-application/caching#1-react-rendering-on-the-server)
На сервере Next.js использует API React для управления рендерингом. Работа рендеринга разбивается на куски: по отдельным сегментам маршрутов и границам саспенса.

Каждый фрагмент рендерится в два этапа:
1) React преобразует серверные компоненты в специальный формат данных, оптимизированный для потоковой передачи, который называется полезной нагрузкой серверного компонента React.
2) Next.js использует инструкции JavaScript для React Server Component и Client Component для рендеринга HTML на сервере.
Это означает, что нам не нужно ждать, пока все отрисуется, прежде чем кэшировать работу или отправлять ответ. Вместо этого мы можем передавать ответ по мере завершения работы.
> [!tip] **What is the React Server Component Payload?**
> Полезная нагрузка компонента сервера React — это компактное двоичное представление визуализированного дерева компонентов сервера React. Он используется React на клиенте для обновления DOM браузера. Полезная нагрузка компонента сервера React содержит:
> - Визуализированный результат серверных компонентов Заполнители для отображения клиентских компонентов и ссылки на их файлы JavaScript.
> - Любые реквизиты, передаваемые из серверного компонента в клиентский компонент.
Дополнительную информацию см. в документации по [Server Components](https://nextjs.org/docs/app/building-your-application/rendering/server-components) .

### [2. Next.js Caching on the Server (Full Route Cache)](https://nextjs.org/docs/app/building-your-application/caching#2-nextjs-caching-on-the-server-full-route-cache)
![[Pasted image 20240711170720.png]]
По умолчанию Next.js кэширует визуализированный результат (полезную нагрузку компонента сервера React и HTML) маршрута на сервере. Это относится к статически отображаемым маршрутам во время сборки или во время повторной проверки.
### [3. React Hydration and Reconciliation on the Client](https://nextjs.org/docs/app/building-your-application/caching#3-react-hydration-and-reconciliation-on-the-client)
Во время запроса на клиенте:

 - HTML используется для немедленного отображения быстрого неинтерактивного первоначального предварительного просмотра клиентских и серверных компонентов.
- Полезная нагрузка React Server Components используется для согласования клиентских и визуализированных деревьев компонентов сервера, а также для обновления DOM.
- Инструкции JavaScript используются для гидратации клиентских компонентов и делают приложение интерактивным.

### [4. Next.js Caching on the Client (Router Cache)](https://nextjs.org/docs/app/building-your-application/caching#4-nextjs-caching-on-the-client-router-cache)
Полезная нагрузка компонента React Server хранится в кэше маршрутизатора на [Router Cache](https://nextjs.org/docs/app/building-your-application/caching#router-cache) — отдельном кэше в памяти, разделенном по отдельным сегментам маршрута. Этот кэш маршрутизатора используется для улучшения навигации за счет хранения ранее посещенных маршрутов и предварительной выборки будущих маршрутов.
### [5. Subsequent Navigations](https://nextjs.org/docs/app/building-your-application/caching#5-subsequent-navigations)
При последующих переходах или во время предварительной выборки Next.js проверит, хранится ли полезная нагрузка компонентов сервера React в кэше маршрутизатора. Если это так, он пропустит отправку нового запроса на сервер.

Если сегменты маршрута отсутствуют в кеше, Next.js получит полезную нагрузку компонентов сервера React с сервера и заполнит кеш маршрутизатора на клиенте.

### [Static and Dynamic Rendering](https://nextjs.org/docs/app/building-your-application/caching#static-and-dynamic-rendering)
Кэшируется ли маршрут или нет во время сборки, зависит от того, отображается ли он статически или динамически. Статические маршруты кэшируются по умолчанию, тогда как динамические маршруты отображаются во время запроса и не кэшируются.

### [Duration](https://nextjs.org/docs/app/building-your-application/caching#duration-2)
По умолчанию полный кэш маршрутов является постоянным. Это означает, что выходные данные рендеринга кэшируются по запросам пользователей.
### [Invalidation](https://nextjs.org/docs/app/building-your-application/caching#invalidation)
Вот два способа сделать недействительным полный кэш маршрутов:
- Повторная проверка данных. Повторная проверка кэша данных, в свою очередь, приведет к аннулированию кэша маршрутизатора путем повторного рендеринга компонентов на сервере и кэширования новых результатов рендеринга.
- Повторное развертывание. В отличие от кэша данных, который сохраняется во всех развертываниях, полный кэш маршрутов очищается при новых развертываниях.
