Server Actions are **asynchronous functions** that are executed on the server. They can be used in Server and Client Components to handle form submissions and data mutations in Next.js applications.
**Watch:** Learn more about forms and mutations with Server Actions → [YouTube (10 minutes)](https://youtu.be/dDpZfOQBMaU?si=cJZHlUu_jFhCzHUg).
## [Convention](https://nextjs.org/docs/app/building-your-application/data-fetching/server-actions-and-mutations#convention)
Действие сервера можно определить с помощью директивы React «use server». Вы можете разместить директиву в верхней части асинхронной функции, чтобы пометить функцию как действие сервера, или в верхней части отдельного файла, чтобы пометить все экспорты этого файла как действия сервера.

## [Behavior](https://nextjs.org/docs/app/building-your-application/data-fetching/server-actions-and-mutations#behavior)
- Server actions can be invoked using the `action` attribute in a [`<form>` element](https://nextjs.org/docs/app/building-your-application/data-fetching/server-actions-and-mutations#forms):
	- Серверные компоненты по умолчанию поддерживают прогрессивное улучшение, то есть форма будет отправлена, даже если JavaScript еще не загружен или отключен.
	- В клиентских компонентах формы, вызывающие действия сервера, будут ставить отправки в очередь, если JavaScript еще не загружен, отдавая приоритет гидратации клиента.
	- После гидратации браузер не обновляется при отправке формы.
- Server Actions are not limited to `<form>` and can be invoked from event handlers, `useEffect`, third-party libraries, and other form elements like `<button>`
- Действия сервера интегрируются с архитектурой Next.js [кэширование и повторная проверка](https://nextjs.org/docs/app/building-your-application/caching). При вызове действия Next.js может возвращать как обновленный пользовательский интерфейс, так и новые данные за один цикл сервера.
- Behind the scenes, actions use the `POST` method, and only this HTTP method can invoke them.
- The arguments and return value of Server Actions must be serializable by React. See the React docs for a list of [serializable arguments and values](https://react.dev/reference/react/use-server#serializable-parameters-and-return-values).
- Server Actions are functions. This means they can be reused anywhere in your application.
- Действия сервера наследуют среду выполнения от страницы или макета, на которых они используются.
- Действия сервера наследуют конфигурацию сегмента маршрута со страницы или макета, на котором они используются, включая такие поля, как maxDuration.

Определение действия сервера внутри компонента создает замыкание, при котором действие имеет доступ к области действия внешней функции. Например, действие публикации имеет доступ к переменной publicVersion:
![[Pasted image 20240525013427.png]]
Замыкания полезны, когда вам нужно сделать снимок данных (например, publicVersion) во время рендеринга, чтобы его можно было использовать позже при вызове действия.

Однако для этого захваченные переменные отправляются клиенту и обратно на сервер при вызове действия. Чтобы предотвратить раскрытие конфиденциальных данных клиенту, Next.js автоматически шифрует закрытые переменные. Новый закрытый ключ генерируется для каждого действия при каждой сборке приложения Next.js. Это означает, что действия можно вызывать только для конкретной сборки.
> **Good to know:**
> Мы не рекомендуем полагаться только на шифрование, чтобы предотвратить раскрытие конфиденциальных значений на клиенте. Вместо этого вам следует использовать API-интерфейсы React taint, чтобы заранее предотвратить отправку определенных данных клиенту.

### [Overwriting encryption keys (advanced)](https://nextjs.org/docs/app/building-your-application/data-fetching/server-actions-and-mutations#overwriting-encryption-keys-advanced)
При самостоятельном размещении приложения Next.js на нескольких серверах каждый экземпляр сервера может иметь свой ключ шифрования, что может привести к потенциальным несоответствиям.
Чтобы избежать этого, вы можете перезаписать ключ шифрования, используя переменную средыprocess.env.NEXT_SERVER_ACTIONS_ENCRYPTION_KEY. Указание этой переменной гарантирует, что ваши ключи шифрования будут постоянными во всех сборках, и все экземпляры сервера будут использовать один и тот же ключ.

Это расширенный вариант использования, в котором согласованное поведение шифрования в нескольких развертываниях имеет решающее значение для вашего приложения. Вам следует учитывать стандартные методы обеспечения безопасности, такие как ротация ключей и подписание.
> **Good to know:**
> Next.js applications deployed to Vercel automatically handle this.

### [Allowed origins (advanced)](https://nextjs.org/docs/app/building-your-application/data-fetching/server-actions-and-mutations#allowed-origins-advanced)
Since Server Actions can be invoked in a `<form>` element, this opens them up to [CSRF attacks](https://developer.mozilla.org/en-US/docs/Glossary/CSRF).
За кулисами действия сервера используют метод POST, и только этому методу HTTP разрешено их вызывать. Это предотвращает большинство уязвимостей CSRF в современных браузерах, особенно если файлы cookie SameSite используются по умолчанию.
В качестве дополнительной защиты действия сервера в Next.js также сравнивают заголовок Origin с заголовком Host (или X-Forwarded-Host). Если они не совпадают, запрос будет прерван. Другими словами, действия сервера можно вызывать только на том же хосте, что и страница, на которой он размещен.
Для крупных приложений, использующих обратные прокси-серверы или многоуровневые серверные архитектуры (где API сервера отличается от рабочего домена), рекомендуется использовать параметр конфигурации serverActions.allowedOrigins, чтобы указать список безопасных источников.
