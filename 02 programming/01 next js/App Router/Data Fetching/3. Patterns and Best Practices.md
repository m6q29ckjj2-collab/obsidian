There are a few recommended patterns and best practices for fetching data in React and Next.js. This page will go over some of the most common patterns and how to use them.
## [Fetching data on the server](https://nextjs.org/docs/app/building-your-application/data-fetching/patterns#fetching-data-on-the-server)
По возможности мы рекомендуем получать данные на сервере с помощью серверных компонентов. Это позволяет вам:
- Иметь прямой доступ к внутренним ресурсам данных (например, базам данных).
- Обеспечьте большую безопасность вашего приложения, предотвращая раскрытие клиенту конфиденциальной информации, такой как токены доступа и ключи API.
- Получите данные и выполните рендеринг в одной и той же среде. Это уменьшает как обратную связь между клиентом и сервером, так и работу в основном потоке клиента.
- Выполняйте несколько выборок данных за один цикл вместо нескольких отдельных запросов на клиенте.
- Уменьшите водопады клиент-сервер.
- В зависимости от вашего региона извлечение данных также может происходить ближе к источнику данных, что снижает задержку и повышает производительность.
Then, you can mutate or update data with [Server Actions](https://nextjs.org/docs/app/building-your-application/data-fetching/server-actions-and-mutations).
## [Fetching data where it's needed](https://nextjs.org/docs/app/building-your-application/data-fetching/patterns#fetching-data-where-its-needed)
Если вам нужно использовать одни и те же данные (например, текущего пользователя) в нескольких компонентах дерева, вам не нужно извлекать данные глобально или пересылать реквизиты между компонентами. Вместо этого вы можете использовать выборку или кеш React в компоненте, которому нужны данные, не беспокоясь о последствиях для производительности, связанных с выполнением нескольких запросов к одним и тем же данным.
This is possible because `fetch` requests are automatically memoized. Learn more about [request memoization](https://nextjs.org/docs/app/building-your-application/caching#request-memoization)
> **Good to know**:
> Это также относится к макетам, поскольку невозможно передавать данные между родительским макетом и его дочерними элементами.

## [Streaming](https://nextjs.org/docs/app/building-your-application/data-fetching/patterns#streaming)
Streaming and [Suspense](https://react.dev/reference/react/Suspense) are React features that allow you to progressively render and incrementally stream rendered units of the UI to the client.
Благодаря серверным компонентам и вложенным макетам вы можете мгновенно отображать части страницы, для которых не требуются данные, и показывать состояние загрузки для частей страницы, которые извлекают данные. Это означает, что пользователю не нужно ждать загрузки всей страницы, прежде чем он сможет начать с ней взаимодействовать.

To learn more about Streaming and Suspense, see the [Loading UI](https://nextjs.org/docs/app/building-your-application/routing/loading-ui-and-streaming) and [Streaming and Suspense](https://nextjs.org/docs/app/building-your-application/routing/loading-ui-and-streaming#streaming-with-suspense) pages.
## [Parallel and sequential data fetching](https://nextjs.org/docs/app/building-your-application/data-fetching/patterns#parallel-and-sequential-data-fetching)
When fetching data inside React components, you need to be aware of two data fetching patterns: Parallel and Sequential.
![[Pasted image 20240525020826.png]]
- При последовательной выборке данных запросы на маршруте зависят друг от друга и, следовательно, создают каскады. Могут быть случаи, когда вам нужен этот шаблон, потому что одна выборка зависит от результата другой, или вы хотите, чтобы условие было выполнено перед следующей выборкой для экономии ресурсов. Однако такое поведение также может быть непреднамеренным и привести к увеличению времени загрузки.
- При параллельной выборке данных запросы на маршруте инициируются быстро и одновременно загружают данные. Это уменьшает каскады клиент-сервер и общее время, необходимое для загрузки данных.
### [Sequential Data Fetching](https://nextjs.org/docs/app/building-your-application/data-fetching/patterns#sequential-data-fetching)
Если у вас есть вложенные компоненты, и каждый компонент извлекает свои собственные данные, то извлечение данных будет происходить последовательно, если эти запросы данных различаются (это не относится к запросам одних и тех же данных, поскольку они автоматически запоминаются).

Например, компонент «Списки воспроизведения» начнет получать данные только после того, как компонент «Артист» завершит получение данных, поскольку списки воспроизведения зависят от свойства ArtistID:
![[Pasted image 20240525021532.png]]В подобных случаях вы можете использовать loading.js (для сегментов маршрута) или React \<Suspense> (для вложенных компонентов), чтобы показать мгновенное состояние загрузки, в то время как React передает результат.

Это предотвратит блокировку всего маршрута при получении данных, и пользователь сможет взаимодействовать с незаблокированными частями страницы.
> **Blocking Data Requests:**
> Альтернативный подход к предотвращению водопадов — получение данных глобально, в корне вашего приложения, но это заблокирует рендеринг для всех сегментов маршрута, находящихся под ним, до тех пор, пока данные не завершат загрузку. Это можно описать как выборку данных по принципу «все или ничего». Либо у вас есть все данные для вашей страницы или приложения, либо нет.
> Любые запросы на выборку с await блокируют рендеринг и получение данных для всего дерева под ним, если только они не заключены в границу \<Suspense> или не используется loading.js. Другая альтернатива — использовать параллельную выборку данных или шаблон предварительной загрузки.

### [Parallel Data Fetching](https://nextjs.org/docs/app/building-your-application/data-fetching/patterns#parallel-data-fetching)
Чтобы получать данные параллельно, вы можете с готовностью инициировать запросы, определяя их вне компонентов, использующих данные, а затем вызывая их изнутри компонента. Это экономит время за счет параллельной инициации обоих запросов, однако пользователь не увидит визуализированный результат, пока оба обещания не будут выполнены.

## [Preloading Data](https://nextjs.org/docs/app/building-your-application/data-fetching/patterns#preloading-data)
Другой способ предотвратить водопады — использовать шаблон предварительной загрузки. При желании вы можете создать функцию предварительной загрузки для дальнейшей оптимизации параллельной выборки данных. При таком подходе вам не придется передавать обещания как реквизит. Функция предварительной загрузки также может иметь любое имя, поскольку это шаблон, а не API.

### [Using React `cache`, `server-only`, and the Preload Pattern](https://nextjs.org/docs/app/building-your-application/data-fetching/patterns#using-react-cache-server-only-and-the-preload-pattern)
You can combine the `cache` function, the `preload` pattern, and the `server-only`package to create a data fetching utility that can be used throughout your app.
![[Pasted image 20240525022402.png]]
При таком подходе вы можете быстро получать данные, кэшировать ответы и гарантировать, что эта выборка данных происходит только на сервере.
Экспорт utils/get-item может использоваться макетами, страницами или другими компонентами, чтобы дать им контроль над тем, когда извлекаются данные элемента.
> **Good to know:**
> - We recommend using the [`server-only` package](https://nextjs.org/docs/app/building-your-application/rendering/composition-patterns#keeping-server-only-code-out-of-the-client-environment) to make sure server data fetching functions are never used on the client.
