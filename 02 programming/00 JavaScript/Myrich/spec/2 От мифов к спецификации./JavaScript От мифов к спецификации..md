# Архитектурный обзор спецификации ECMAScript® 2025 (16-е издание)

## 1. Генезис и эволюция стандарта

История ECMAScript представляет собой процесс эволюции от узкоспециализированного инструмента для «оживления» веб-страниц (Netscape Navigator 2.0) до статуса одного из самых распространенных в мире языков программирования общего назначения. С момента выхода 1-го издания в 1997 году стандарт прошел через фундаментальные трансформации: 3-е издание (1999) заложило основы мощной обработки строк и регулярных выражений, 5-е издание (2009) внедрило строгий режим (Strict Mode) и поддержку JSON, а 6-е издание (2015) стало кульминацией пятнадцатилетней работы по адаптации языка для создания крупномасштабных приложений и библиотек.

Стратегический переход на ежегодный цикл выпуска (yearly release cadence), начатый в 2016 году, изменил саму природу спецификации, превратив ее в «живой документ». Разработка под эгидой комитета TC39 в открытом формате на GitHub позволяет интегрировать предложения (Stage 4) и исправления оперативно, не дожидаясь многолетних циклов. С точки зрения системной архитектуры, стандарт демонстрирует уникальную преемственность: он сохраняет совместимость через «устаревшие разделы» (Legacy clauses), при этом обеспечивая строгое определение фундаментальных основ. Такая структура позволяет языку оставаться современным, не разрушая экосистему существующего веб-контента, что подводит нас к необходимости разбора понятийного аппарата, на котором зиждется эта стабильность.

## 2. Концептуальная архитектура и терминология

Для обеспечения полной совместимости различных движков (implementations) спецификация выступает в роли «Абсолютной истины», где точность терминологии превалирует над деталями реализации. ECMAScript по своей сути является «вычислительно несамодостаточным» (computationally non-self-sufficient): он не содержит встроенных средств ввода-вывода, делегируя их хост-среде.

### Уровни определения фасилити

Спецификация разделяет ответственность за поведение системных функций на три уровня:

- **Implementation-defined:** Определение полностью делегируется внешней реализации без дополнительных рекомендаций.
- **Implementation-approximated:** Реализация определяет поведение самостоятельно, но спецификация рекомендует «идеальный» вариант (например, математические вычисления в `Math.exp`).
- **Host-defined:** Поведение определяется внешней хост-средой (браузером или сервером) через механизмы, описанные в Annex D.

### Типология объектов

В архитектуре ECMAScript объект — это коллекция свойств с атрибутами. Они классифицируются по источнику и поведению:

- **Обычные объекты (ordinary objects):** Обладают поведением по умолчанию для всех обязательных внутренних методов (Essential Internal Methods).
- **Экзотические объекты (exotic objects):** Объекты, чье поведение хотя бы одного внутреннего метода (например, `[[Get]]` или `[[Set]]`) отличается от стандартного. Это позволяет спецификации определять особые сущности, такие как `Proxy` или `Array`.
- **Стандартные объекты (standard objects):** Семантика которых определена данным стандартом.
- **Встроенные объекты (built-in objects):** Поставляются реализацией; могут быть как стандартными, так и специфичными для хоста.

### Типы данных ECMAScript (ECMAScript Language Types)

Спецификация жестко ограничивает систему типов восемью сущностями: **Undefined**, **Null**, **Boolean**, **String**, **Symbol**, **Number**, **BigInt** и **Object**. Первые семь являются примитивными значениями — данными, представленными на самом низком уровне реализации. Объектный тип является единственным не-примитивом и служит фундаментом для всех сложных структур данных.

## 3. Ключевые нововведения ECMAScript 2025 (16-е издание)

16-е издание фокусируется на снижении вариативности реализаций в экосистеме путем стандартизации часто используемых паттернов управления данными и расширения математических возможностей.

- **Глобальный объект Iterator:** Внедрение конструктора `Iterator` и методов прототипа (таких как `map`, `filter`, `take`, `drop`) переводит абстракцию итерации на уровень первого класса, унифицируя работу с последовательностями.
- **Расширение Set.prototype:** Добавление методов для операций над множествами (`union`, `intersection`, `difference`, `symmetricDifference`) и проверок (`isSubsetOf`, `isSupersetOf`, `isDisjointFrom`) устраняет необходимость в сторонних библиотеках для базовых задач теории множеств.
- **Модульная система и JSON:** Стандартизирован импорт JSON-модулей и синтаксис атрибутов импорта (Import Attributes), что усиливает безопасность и предсказуемость загрузки ресурсов.
- **Тип Float16Array и математика:** Появление `Float16Array`, методов `DataView.prototype.getFloat16`/`setFloat16` и `Math.f16round` отвечает запросам областей машинного обучения и графики, где важна экономия памяти при хранении чисел с плавающей запятой низкой точности.
- **Улучшения RegExp:** Метод `RegExp.escape` обеспечивает безопасное создание выражений из динамических строк, а модификаторы флагов внутри выражения позволяют менять параметры сопоставления (например, `i` для регистронезависимости) локально для отдельных фрагментов паттерна.
- **Promise.try:** Данный метод гарантирует, что результат вызова функции (даже если она синхронная или выбрасывает исключение) всегда будет обернут в `Promise`, обеспечивая единообразие в асинхронных цепочках.

Эти функциональные надстройки опираются на строгую методологию описания алгоритмов, которая гарантирует идентичность поведения во всех конформных средах.

## 4. Методология спецификации и нотационные соглашения

Вместо формальной математической логики спецификация использует специализированный псевдокод, который детерминированно описывает шаги выполнения.

### Грамматические уровни

Архитектура языка разделена на несколько уровней грамматики, каждый из которых использует свои соглашения:

|   |   |   |
|---|---|---|
|Уровень грамматики|Назначение|Разделитель|
|**Лексическая (Lexical)**|Группировка кодовых точек в токены (идентификаторы, литералы).|`::`|
|**Синтаксическая (Syntactic)**|Определение структуры скриптов и модулей из токенов.|`:`|
|**Числовых строк (Numeric String)**|Правила преобразования строк в числа (используется `StringToNumber`).|`:::`|
|**Регулярных выражений**|Парсинг паттернов внутри RegExp.|`::`|

### Алгоритмические сокращения

Для упрощения чтения абстрактных операций используются префиксы:

- `**?**` **(ReturnIfAbrupt):** Если операция возвращает «аномальное завершение» (например, исключение), оно немедленно пробрасывается выше. В противном случае значение извлекается из записи завершения.
- `**!**`**:** Указывает на то, что операция никогда не вернет аномальное завершение, и ее результат (Value) используется напрямую.

### Математические сущности

Спецификация строго разделяет контексты вычислений:

- **ℝ (Mathematical values):** Абстрактные вещественные числа.
- **𝔽 (Numbers):** Числа в формате IEEE 754-2019 (double-precision).
- **ℤ (BigInts):** Целые числа произвольной точности.

Разделение на **статическую семантику** (проверки Early Errors до выполнения) и **динамическую семантику** (Runtime Semantics) позволяет выявлять ошибки на этапе разбора, что критично для стабильности объектной модели.

## 5. Объектная модель и прототипное наследование

ECMAScript представляет собой систему коммуницирующих объектов, где структура и поведение наследуются через механизм прототипов.

### Реконструкция логики (на основе Figure 1)

Рассмотрим системную связь между конструктором (**CF**), его прототипом (**CFp**) и экземплярами (**cf1**–**cf5**):

1. **CF** — это объект-функция, имеющий собственное состояние (свойства `P1`, `P2`), которое **не наследуется** экземплярами.
2. **CFp** — объект-прототип, содержащий разделяемое свойство `CFP1`.
3. **cf1**–**cf5** — экземпляры, созданные через `new CF()`. Они обладают собственными свойствами (`q1`, `q2`).

**Архитектурный вывод:** При обращении к `cf1.CFP1` движок использует цепочку прототипов (prototype chain). Поскольку свойство отсутствует в `cf1`, поиск делегируется прототипу `CFp`. Это обеспечивает колоссальную эффективность памяти: тысячи экземпляров разделяют один набор методов и констант, хранящихся в прототипе, вместо их дублирования.

### Внутренние методы и атрибуты

Каждый объект обязан поддерживать обязательные внутренние методы (Essential Internal Methods), определяющие ядро его поведения:

- `[[Get]]` и `[[Set]]`: Доступ и запись.
- `[[Delete]]`: Удаление свойства.
- `[[GetPrototypeOf]]`: Получение ссылки на родительский объект.

Свойства объектов описываются атрибутами: `Value` (значение), `Writable` (запись), `Enumerable` (перечисляемость) и `Configurable` (возможность изменения самого определения свойства). Такая гранулярность позволяет создавать неизменяемые структуры данных на уровне движка.

## 6. Среда выполнения и режимы исполнения

Гибкость ECMAScript обеспечивается через «Host Hooks» — абстрактные операции, реализация которых оставлена на усмотрение хост-среды, что позволяет языку одинаково успешно работать и в браузерах, и в серверных окружениях (Node.js).

### Варианты исполнения

- **Строгий режим (Strict Variant):** Активирует усиленные проверки и запрещает небезопасные конструкции. Ключевая архитектурная особенность — выбор режима на уровне «единицы исходного текста» (source text unit). Это позволяет комбинировать строгие модули и нестрогие скрипты в рамках одной программы, обеспечивая плавную миграцию кода.
- **Агенты и параллелизм:** Модель параллельного выполнения базируется на «агентах». Взаимодействие между ними в кластерах осуществляется через разделяемую память (`SharedArrayBuffer`) и атомарные операции (`Atomics`), предотвращающие состояние гонки (data races).
- **Annex B (Legacy):** Нормативное дополнение, описывающее специфические функции для веб-браузеров. Это «слой совместимости», который формально находится за пределами ядра языка, но является обязательным для реализации в браузерах для поддержки исторического кода.

**Заключение:** ECMAScript 2025 (16-е издание) подтверждает статус высокоструктурированной системы. Четкое разделение на ядро, хост-зависимые расширения и легаси-слои позволяет языку масштабироваться, сохраняя академическую точность определений и предсказуемость исполнения, необходимую для современной программной инженерии.
