# Образовательные карточки: Архитектура исполнения JavaScript-кода (по материалам «As For JS»)

JavaScript — это встраиваемый скриптовый язык, чья архитектура продиктована его «беспомощностью» на уровне спецификации. У JS нет собственного ввода-вывода (I/O), он не умеет самостоятельно читать файлы или выводить текст в консоль. Это стратегически важный момент: понимание JS как гостевого языка, полностью зависящего от Host-среды, позволяет разработчику четко разделять ответственность движка и внешних API. Эта зависимость определяет, как код попадает на исполнение и как он взаимодействует с миром через интерфейсы, предоставленные браузером или Node.js.

Что такое скриптовый язык согласно спецификации ECMAScript?::Скриптовый язык — это язык, который не предназначен для автономного существования и лишен встроенных механизмов ввода-вывода. Согласно спецификации, JavaScript «бесполезен» сам по себе: он не может считать данные извне или отобразить результат работы, так как в нем отсутствуют соответствующие процедуры. Он всегда встраивается в Host-среду, которая обеспечивает его функционалом через внешние API, поэтому любой console.log или alert — это не часть языка, а инструмент окружения, в которое JS был интегрирован.

Какова роль Host-среды и как JS взаимодействует с внешними API для ввода-вывода?::Host-среда (браузер или Node.js) выступает внешней оболочкой, которая предоставляет JS-движку API и данные для работы. JS-код не считывает информацию самостоятельно; он получает ее на вход от среды и отправляет результаты обратно через специальные интерфейсы. Фактически, среда — это «кусты», в которых прячутся инструменты I/O, а JS лишь оперирует ими. Разработчик должен понимать этот контракт, чтобы осознавать, где заканчивается логика языка и начинаются специфические механизмы конкретного окружения.

В чем заключается «философия программиста» и почему важно «принять клизму ответственности»?::Профессиональная деформация программиста требует постоянного сомнения в абстракциях и поиска ответа на вопрос «как это работает на самом деле». Разработчик обязан регулярно «бить себя по лбу», признавая свое незнание, чтобы докопаться до истины уровня спецификации. Как говорит автор источника, наставник или старший инженер должен «принять клизму» (клизму ответственности) — взять на себя обязательство транслировать абсолютную истину спецификации, даже если она противоречит популярным, но поверхностным туториалам, которыми «кормят» новичков.

Понимание внешней зависимости языка позволяет нам перейти к изучению внутренних структур, которые управляют выполнением кода внутри самого движка.

**Execution Context (Контекст исполнения): Анатомия «Окружения»**

Execution Context (Контекст исполнения) — это стратегическая инфраструктурная единица, сопровождающая выполнение любого фрагмента кода. В JavaScript ни одна строка не выполняется в вакууме: ей требуется описание состояния системы, переменных и текущей позиции. EC — это базовая единица ответственности движка, которая позволяет сохранять, прерывать и восстанавливать работу кода, обеспечивая целостность окружения на каждом этапе.

Что такое Execution Context (EC) и как работает аналогия с «закрытой крышкой ноутбука»?::Execution Context — это служебная структура, которая инкапсулирует всё необходимое для выполнения конкретного блока кода (функции, модуля или глобального скрипта). Его работа идеально описывается аналогией с ноутбуком: когда вы закрываете крышку, система «замораживается» (suspend), сохраняя состояние всех окон и данных, а при открытии — мгновенно восстанавливает работу ровно с той же точки (resume). В JS каждый контекст позволяет точно так же фиксировать состояние выполнения, чтобы движок мог прерваться на другую задачу и вернуться без потери контекста данных.

Из каких составных частей состоит Execution Context?::Каждый контекст исполнения включает в себя ссылки на исходный код (Source Code), указатели на текущую исполняемую строку и специальные структуры — Environment Records. Записи окружения работают по принципу «ключ-значение», храня идентификаторы переменных и функций. Кроме того, контекст может содержать специфические поля для поддержки генераторов, асинхронности и связи с глобальными настройками (Realm). Это не просто «стек вызовов», а полноценная инкапсулированная среда обитания конкретного фрагмента кода.

Почему концепция «окружения» (Environment) является сквозной для всей спецификации?::Концепция Environment пронизывает спецификацию, потому что она отражает природу JS как встраиваемого языка: код всегда находится «внутри чего-то». Любая структура в JS рассматривается как оболочка (Global, Functional или Module Environment), регламентирующая поведение того, что она окружает. Спецификация не плодит уникальные механизмы, а использует единую логику окружений для управления видимостью данных и порядком исполнения, фиксируя связь «кода» и его «контейнера».

Подведение итога тому, как отдельные контексты организуются в управляемую систему.

**Execution Context Stack против Call Stack: Разрушение мифов**

Общепринятый термин «Call Stack» является существенным упрощением и, по сути, архитектурным мифом. В то время как Call Stack — это жесткое понятие из архитектуры x86 (адреса возврата), спецификация ECMAScript использует Execution Context Stack (ECS). Это различие фундаментально: ECS управляет не просто адресами команд, а целыми состояниями системы, что делает JS гибким и позволяет реализовывать сложные асинхронные конструкции.

В чем разница между x86 Call Stack и Execution Context Stack (ECS) в JavaScript?::Термин Call Stack относится к архитектуре процессоров (например, 8086), где в стек кладутся только адреса возврата — числа, указывающие, на какую строку прыгнуть после функции. В спецификации JS такого термина нет; там используется Execution Context Stack, который оперирует ссылками на полноценные структуры (контексты). В отличие от x86, ECS управляет «мирами» переменных и настроек, которые можно независимо замораживать, извлекать и вклинивать обратно, что физически невозможно в классическом аппаратном стеке.

Как механизм Suspend/Resume в ECS делает возможными генераторы и async/await?::Гибкость ECS позволяет движку в любой момент времени приостанавливать (suspend) текущий контекст и переходить к другому, не следуя строгому порядку LIFO. В случае генераторов или async/await контекст функции извлекается из стека при встрече с yield/await, сохраняя всё внутреннее состояние. Когда условие возобновления выполнено, этот же контекст просто снова помещается в стек, и выполнение продолжается ровно с той же точки, в то время как классический Call Stack заставил бы нас просто вернуться по адресу, потеряв локальное состояние.

Что означает «летаргическая» (ленивая) природа ссылок и почему DevTools «лжет»?::Архитектура JS «ленива»: данные и код остаются там, где были созданы, а система оперирует лишь указателями-стрелочками. Execution Context не копирует код функций, а лишь ссылается на них. При этом DevTools часто «лжет» разработчику, показывая в Call Stack функции, которые уже завершились и физически покинули ECS. Это делается для удобства отладки (создания видимости цепочки вызовов), но в реальности движок оперирует гораздо более динамичным и «пустым» стеком контекстов.

Объяснение того, как стек контекстов взаимодействует с более крупными структурами — агентами и реамами.

**Инфраструктурные уровни: Realm и Agent**

Иерархия исполнения JavaScript поднимается от конкретной строки кода до глобального управления ресурсами через Realm и Agent. Эти абстракции отвечают за изоляцию в разных вкладках браузера и организацию многопоточности. Понимание этих уровней необходимо, чтобы осознать, как движок разделяет память и почему глобальные объекты в разных частях приложения изолированы друг от друга.

Что такое Realm (Реалм) и почему у каждой вкладки свой глобальный объект?::Realm — это инфраструктурная запись, служащая «хранилищем настроек по умолчанию». Каждая вкладка или iframe имеет свой Realm, что гарантирует изоляцию: у каждой среды свой набор встроенных конструкторов (Array, Object) и свой глобальный объект. Это «базовое хранилище» создается хост-средой до начала выполнения кода, чтобы предоставить движку стартовую точку и необходимые API. Реалмы позволяют сосуществовать нескольким независимым мирам JS внутри одного процесса.

Что представляет собой сущность Agent (Агент) и как он связан с потоками?::Агент — это атомарная структура, которая непосредственно выполняет код и управляет стеком контекстов. Агент может быть связан с одним или несколькими потоками ОС, но по спецификации он выполняет ровно одну задачу в единицу времени. Хост-среда может объединять несколько вкладок под управлением одного агента (как это было в старых Chrome) или выделять по агенту на каждый воркер. Агент — это исполнитель, который «крутит педали» контекстов в рамках своего кластера.

Каково значение поля [[CanBlock]] в записи агента и его влияние на Atomics?::Поле [[CanBlock]] — это булево значение, которое определяет, имеет ли право агент приостанавливать выполнение в ожидании доступа к разделяемой памяти (Atomics.wait). В основном потоке браузера это поле всегда false, чтобы «индусский код» или тяжелые вычисления не заморозили UI. В веб-воркерах оно установлено в true, что позволяет им эффективно синхронизироваться через SharedArrayBuffer, буквально замирая до момента разблокировки ресурса другим агентом.

Переход от структур хранения к механизмам планирования задач.

**Альтернатива Event Loop: Очереди Jobs в спецификации**

Термин «Event Loop» принадлежит стандарту HTML, тогда как спецификация ECMAScript оперирует понятием «Jobs» (работы). Это разделение критически важно: именно механизмы Jobs определяют, как движок расставляет приоритеты между промисами и другими операциями внутри самого языка, прежде чем передать управление событийному циклу браузера.

Какие три типа очередей Jobs выделяются в спецификации?::Спецификация выделяет PromiseJobs, GenericJobs и TimeoutJobs. Каждая очередь предназначена для своего типа отложенных задач. Это внутренние механизмы движка, которые сигнализируют Host-среде о необходимости выполнения определенных действий. Разделение на эти очереди позволяет JS-движку поддерживать внутреннюю логику асинхронности (например, разрешение промисов) независимо от того, как устроен событийный цикл конкретного браузера.

В чем специфика TimeoutJobs и пример «индусского кода» в DevTools?::TimeoutJobs не имеет отношения к setTimeout; это механизм для работы с Atomics и лимитами времени ожидания в многопоточной среде. Что касается привычных таймеров, в источнике приводится пример «типичного индусского кода» с вложенными setTimeout и промисами: в таких случаях DevTools рисует «фейковый» асинхронный стек трейс, склеивая задачи, которые в реальности выполнялись в абсолютно разных итерациях и контекстах. Это делается для отладки, но физически в ECS этих связей нет — там задачи появляются и исчезают атомарно.

Синтез разницы в гарантиях порядка между PromiseJobs и GenericJobs?::Очередь PromiseJobs (микрозадачи) дает строгую гарантию порядка: задачи выполняются ровно в той последовательности, в которой были добавлены. GenericJobs — это «свободная» очередь, где спецификация не гарантирует порядок исполнения, оставляя его на усмотрение Host-среды. Понимание этой разницы позволяет проектировать надежную асинхронную логику, зная, где порядок детерминирован языком, а где он зависит от прихотей браузера.

Как эти очереди позволяют Host-среде гибко вклинивать задачи в стек исполнения.

**Лексические структуры и поиск идентификаторов**

Поиск переменных в JavaScript — это детерминированный процесс, основанный на иерархии лексических структур. Движок использует механизмы VariableEnvironment и LexicalEnvironment для управления областями видимости, что объясняет разное поведение var, let и const, особенно в глобальном контексте, где сохраняется историческая «магия» связи с глобальным объектом.

Как работает механизм рекурсивного поиска идентификаторов через поле [[OuterEnv]]?::Когда коду нужна переменная, он заглядывает в текущую запись Environment Record. Если ее там нет, движок идет по ссылке в поле [[OuterEnv]] (внешнее окружение) к родительской структуре. Этот поиск идет рекурсивно вверх по «цепочке стрелочек», пока не найдет идентификатор или не упрется в null. Если цепочка закончилась на null — вылетает ReferenceError. Это и есть физическое воплощение механизма замыканий и областей видимости.

В чем причина разделения на VariableEnvironment и LexicalEnvironment?::Это разделение — следствие эволюции языка. VariableEnvironment отвечает за старые добрые var, а LexicalEnvironment — за современные let и const. В функциональных контекстах они часто указывают на одну структуру, но в глобальном коде они разделены, чтобы обеспечить обратную совместимость. Это позволяет новым типам объявлений (let/const) не захламлять глобальный объект, в то время как var продолжает вести себя «по-старому».

Каковы особенности Global Environment Record и роль структуры [[VarNames]]?::Global Environment Record — уникальная сущность, объединяющая обычное окружение и глобальный объект хоста (window). В ней есть поле [[VarNames]], хранящее список всех имен, объявленных через var и function. Это нужно для поддержания «магической» связи: когда вы объявляете var в глобале, движок через [[VarNames]] понимает, что нужно создать свойство в глобальном объекте. При этом let и const туда не попадают, существуя в параллельной лексической структуре того же глобального рекорда.

Понимание этих принципов делает разработчика независимым от особенностей конкретных движков.
